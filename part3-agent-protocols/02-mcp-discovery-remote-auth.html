1) What MCP is (and what it isn’t)

MCP (Model Context Protocol) is a standard interface for connecting an AI host (the app that runs the model) to external capabilities offered by servers. Those capabilities usually fall into three buckets: ResourcesPromptsTools.

User ↔ Host App (LLM) ↔ MCP Server(s)

The host app is in control: it decides what context to provide to the LLM and which tools the LLM is allowed to call. The MCP server is “just” a capability provider.

Important: MCP is not “the LLM calling the Internet by itself.” In well-designed systems, the LLM proposes actions, but the host executes calls and enforces policy. That is exactly how you keep things safe and predictable.

1.1 Beginner mental model (the 3 roles)

(1) User asks a question ↓ (2) Host app (your code) decides what to do ↓ (3) Host talks to MCP server(s) for data/actions ↓ (4) Host feeds results back to the LLM and the user

Beginners often think “the model calls the tool”. In reality, the host is the component that makes network calls and enforces policy (allow-lists, rate limits, auth headers).

1.2 Key MCP terms (quick glossary)

Term Beginner explanation Example
Host Your app that runs the LLM and enforces policy. A Python service using an LLM SDK.
MCP Server A capability provider that exposes resources/prompts/tools. A “shop” server exposing inventory_check.
Transport How messages move between host and server (stdio or HTTP). Streamable HTTP at /mcp.
Discovery Listing what exists (tools/resources/prompts) before using them. list_tools() then call_tool().
Stateless vs stateful Whether the server expects a durable session per client. Stateless HTTP is simplest to scale.

2) Core MCP components: resources, prompts, tools

Here’s a conceptual mapping that makes MCP intuitive:

Component What it is What the host typically does with it Example in a “shop” domain
Resources Named/URI-addressed context objects (data you want the model to read). Fetch and inject into the LLM’s context (or summarize first). shop://catalog, shop://schema
Prompts Reusable prompt templates or workflows published by the server. Present to user or internally use as standard instructions. “Write a purchase summary for SKU + quantity”
Tools Callable functions with structured inputs/outputs. Can have side effects. Allow the model to call them (with restrictions), then return results back to the model. inventory_check(sku, qty), create_order(sku, qty)
Quick rule: If something is read-only or “context,” make it a Resource. If it’s a reusable instruction, make it a Prompt. If it changes state (writes, creates, sends), make it a Tool.

3) Discovery: how a client finds resources/prompts/tools

Discovery is the step that confused you earlier (“where is mcp_call? how does discovery happen?”). The answer is: the host connects to the server and asks for the catalogs.

Typical host flow (high-level):
  1. Connect to MCP server transport (stdio, local HTTP, remote HTTP)
  2. Initialize
  3. List tools (names + input schema)
  4. List prompts (names + args)
  5. List resources (URIs + metadata)

After discovery, the host can do two important policy steps:

  • Tool allow-list: choose which tools the model may call (e.g., only read-only tools in production).
  • Context selection: decide which resources to inject (to avoid over-sharing or prompt bloat).
Discovery doesn’t mean the model automatically sees everything. The host chooses what to show and what to enable.

3.1 Transports (how MCP messages travel)

MCP is a protocol. It can be carried over different transports. Beginners usually start with Streamable HTTP (easy to deploy) or stdio (great for local tools).

Common transports:
  • stdio: host spawns the server process and communicates via stdin/stdout (local only).
  • HTTP: server runs as a web app; host calls it over the network (local or remote).

Remote servers usually use HTTP because it works across machines and supports standard web controls (auth, rate limits, logs).

3.2 What “initialize” means

initialize() is the moment the host and server agree on protocol expectations. After initialization, the host can safely call listing endpoints and tools.

3.3 Tool schemas: why structured inputs matter

Tools are not “free-form strings”. Each tool has a schema (think: a mini API contract). That lets the host validate inputs before calling and gives the LLM a clear structure to follow.

Beginner rule: If you can’t describe a tool’s inputs/outputs as JSON, the tool is probably too vague. Split it into smaller tools.

4) Server tutorial: a simple MCP “shop” server in Python

This server has: 2 resources (catalog and schema), 1 prompt (purchase summary prompt), and 2 tools (inventory check and create order).

This is a toy example but it demonstrates the full pattern: read context (resources), reuse instructions (prompt), perform actions (tools). 

# server.py
from typing import TypedDict, Optional
from mcp.server.fastmcp import FastMCP

# Create the MCP server
mcp = FastMCP("MiniShop MCP", stateless_http=True)

# In-memory demo data
CATALOG = {
  "SKU-APPLE": {"name": "Apple", "price": 1.50, "stock": 12},
  "SKU-CHAIR": {"name": "Chair", "price": 49.00, "stock": 0},
}

# ----------------------
# RESOURCES
# ----------------------
@mcp.resource("shop://catalog")
def catalog_snapshot() -> str:
    import json
    return json.dumps(CATALOG, indent=2)

@mcp.resource("shop://schema")
def schema_doc() -> str:
    return """Data shapes:
- Catalog item: { name: string, price: number, stock: number }
- Order request: { sku: string, qty: number }
- Order result: { order_id: string, status: 'created'|'rejected', reason?: string }
"""

# ----------------------
# PROMPTS
# ----------------------
@mcp.prompt(title="Purchase Summary")
def purchase_summary_prompt(sku: str, qty: int) -> str:
    item = CATALOG.get(sku)
    if not item:
        return f"Write a polite message: SKU {sku} does not exist."
    return f"""Write a short purchase summary for the user.
Item: {item['name']} ({sku})
Quantity: {qty}
Unit price: ${item['price']}
Mention stock status and alternatives if stock is low/zero.
"""

# ----------------------
# TOOLS
# ----------------------
class InventoryResult(TypedDict):
    sku: str
    requested: int
    available: int
    can_fulfill: bool

@mcp.tool()
def inventory_check(sku: str, qty: int) -> InventoryResult:
    item = CATALOG.get(sku, {"stock": 0})
    available = int(item["stock"])
    return {
        "sku": sku,
        "requested": qty,
        "available": available,
        "can_fulfill": available >= qty,
    }

class OrderResult(TypedDict, total=False):
    order_id: str
    status: str     # "created" | "rejected"
    reason: Optional[str]

@mcp.tool()
def create_order(sku: str, qty: int) -> OrderResult:
    check = inventory_check(sku, qty)
    if not check["can_fulfill"]:
        return {"status": "rejected", "reason": "Insufficient stock"}
    # Side effect: reserve stock
    CATALOG[sku]["stock"] -= qty
    import uuid
    return {"status": "created", "order_id": f"ord_{uuid.uuid4().hex[:8]}"}

# ----------------------
# Expose as a Streamable HTTP MCP app
# ----------------------
app = mcp.streamable_http_app()

# Run with: uvicorn server:app --host 127.0.0.1 --port 8000
What you just built
  • Resources are readable by URI (catalog/schema)
  • Prompt provides a reusable instruction template
  • Tools enable actions (inventory check, create order)

In a real deployment, your resources might be “database schema”, “latest sales report”, “customer profile”. Tools might be “run SQL”, “create ticket”, “approve order”.

5) Client tutorial: list resources/prompts/tools and call them

The client is the “host brain” that: (1) discovers what the server provides, and then (2) reads resources or calls tools.

This client uses MCP’s streamable HTTP client transport. You can adapt the same pattern for stdio or other supported transports. 

# client.py
import asyncio
from mcp.client.session import ClientSession
from mcp.client.streamable_http import streamable_http_client

MCP_URL = "http://127.0.0.1:8000/mcp"

async def main():
    async with streamable_http_client(MCP_URL) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # 1) Discover tools
            tools = await session.list_tools()
            print("\nTOOLS:")
            for t in tools:
                print("-", t.name, ":", t.description)

            # 2) Discover resources
            resources = await session.list_resources()
            print("\nRESOURCES:")
            for r in resources:
                print("-", r.uri)

            # 3) Read a resource
            catalog = await session.read_resource("shop://catalog")
            print("\nCATALOG RESOURCE CONTENT:\n", catalog.contents[0].text)

            # 4) Discover prompts
            prompts = await session.list_prompts()
            print("\nPROMPTS:")
            for p in prompts:
                print("-", p.name)

            # 5) Call tools
            inv = await session.call_tool("inventory_check", {"sku": "SKU-APPLE", "qty": 2})
            print("\nINVENTORY CHECK RESULT:\n", inv.content[0].text)

            order = await session.call_tool("create_order", {"sku": "SKU-APPLE", "qty": 2})
            print("\nCREATE ORDER RESULT:\n", order.content[0].text)

      if __name__ == "__main__":
        asyncio.run(main())
How to run the client 
# In another terminal (server already running):
python client.py

You should see the discovered TOOLS/RESOURCES/PROMPTS, the catalog resource content, and the JSON-ish tool results.

What you should notice in this client

  • Discovery is explicit: you call list_tools, list_resources, list_prompts.
  • Resources are read with read_resource(uri) and returned as content blocks.
  • Tools are invoked with call_tool(name, args_dict).
  • Prompts appear in discovery; many hosts surface them to users as “recipes”.
Connection to what we discussed earlier:
This is the MCP equivalent of “passing dictionaries” in LangChain: tool calls are structured dict inputs, and tool results become structured outputs that the host can feed back to the LLM.

6) Remote MCP servers and the “mcp.shop” analogy

Everything you did above works the same when the server is remote. The only differences are:

  • The MCP URL becomes something like https://mcp.shop.example/mcp
  • You must add auth (almost always)
  • You should add operational controls: rate limits, logs, monitoring, timeouts
Example “mcp.shop” capabilities
  • Resources: shop://catalog, shop://shipping_zones, shop://return_policy
  • Tools: search_products, inventory_check, create_order, track_order
  • Prompts: “Recommend products”, “Write an invoice summary”, “Draft a customer confirmation message”
Remote MCP should be treated like a production API: authentication, authorization, and least-privilege tool exposure become the main concerns.

7) Using a remote MCP server from an LLM (OpenAI analogy)

You said: “I use OpenAI and my own client with an LLM — the LLM is actually invoking the server.” The accurate picture is:

LLM proposes a tool call ↓ Your host/client executes the tool call (HTTP request to MCP server) ↓ MCP server returns tool result ↓ Host/client sends tool result back to the LLM as context

So the LLM isn’t holding sockets or managing auth. Your host is. The LLM “decides” which tool to use (within the boundaries you allow).

Conceptual OpenAI-style configuration

The exact API fields can differ by SDK/version, but the concept is always: configure an MCP tool with a URL, restrict tools, attach headers. 

# Conceptual OpenAI host wiring for a remote MCP server
tools = [
  {
    "type": "mcp",
    "server_label": "mcp.shop",
    "server_url": "https://mcp.shop.example/mcp",
    "allowed_tools": ["inventory_check", "create_order"],
    "headers": {"Authorization": "Bearer <token>"}
  }
]
Safety levers you should always use:
  • allowed_tools: start minimal (read-only), expand carefully.
  • server-side authorization: don’t rely on “the model will behave”.

8) Authentication & Identity Providers in MCP

MCP runs over HTTP. So authentication looks exactly like a typical API:

  • Client obtains an access token from an Identity Provider (IdP)
  • Client sends Authorization: Bearer <token> to the MCP server
  • Server middleware validates token and attaches identity/scopes to request context
  • Tools enforce authorization (scopes/roles) before doing side effects
“Authentication” answers who are you?
“Authorization” answers what are you allowed to do?
Middleware usually handles authentication; tool handlers should handle authorization.

9) Self-hosted IdP + Bearer token + middleware validation

This is the pattern you asked about: “client gets bearer token from auth server; server validates it.” We’ll show two validation approaches:

  • Opaque token validated by calling IdP introspection endpoint
  • JWT validated locally using public keys fetched from IdP (JWKS)

9.1 Opaque token path (introspection)

Client → IdP: POST /token (username/password or auth-key) → access_token (opaque)
Client → MCP: Authorization: Bearer access_token
MCP → IdP: POST /introspect (token) → active? claims? scopes?
MCP → allow/deny

9.2 Server middleware example (introspection)

This middleware sits in front of /mcp and rejects requests without a valid token. 

# middleware_snippet.py (conceptual)
auth = request.headers.get("authorization", "")
if not auth.startswith("Bearer "):
    return JSONResponse({"detail": "Missing Bearer token"}, status_code=401)

token = auth.split(" ", 1)[1].strip()

r = await client.post(IDP_INTROSPECT_URL, json={"token": token})
data = r.json()
if not data.get("active"):
    return JSONResponse({"detail": "Invalid token"}, status_code=401)

request.state.sub = data.get("sub")
request.state.scope = data.get("scope", "")
Once identity is attached to request.state, your tools can enforce permissions: e.g., order:write required to call create_order.

10) How clients get tokens: password/auth-key (local) vs PKCE (public)

You asked two key questions:

  • “In this case we don’t use an auth key just username/password — can we use an auth key and why?”
  • “Client has to access Google server to get bearer token still? Is this PKCE?”

10.1 Local services (can keep secrets): password OR auth-key exchange

If your client is a backend service, CLI, or trusted machine environment, it can keep secrets safely (vault/env vars). You can obtain bearer tokens by:

  • Username/password (user identity) → token (scoped to user)
  • Auth key / client credential (service identity) → token (scoped to service)
Why use auth keys? They help when you don’t want to store user passwords or you need machine-to-machine access. They also simplify rotation and auditing at the service level.

10.2 Public clients (cannot keep secrets): PKCE

In browsers/mobile apps, secrets can be extracted. PKCE exists to secure the authorization code flow for these clients. The client must talk to the IdP (Google or your IdP) to obtain the token—yes, that’s required. The server keeping a public key does not remove the need for the client to obtain the token; it only makes server-side validation faster.

Public client (SPA/mobile) gets token from IdP via PKCE
→ then calls MCP server with Authorization: Bearer token

11) Server validation: introspection vs JWKS (public key retrieval at start)

11.1 Introspection (opaque tokens)

Opaque tokens are validated by calling the IdP (or cached results). This is simple and supports immediate revocation.

11.2 JWKS (JWT tokens)

If your IdP issues JWTs, your MCP server can validate locally. It needs public keys. The usual pattern is:

  1. On startup, fetch IdP discovery document (or configure JWKS URL directly)
  2. Fetch JWKS and cache keys keyed by kid
  3. For each request, read the JWT header.kid and choose key to verify signature
  4. Validate claims: issuer (iss), audience (aud), expiry (exp)
  5. Refresh keys on cache expiration or unknown kid
Answering your earlier question: “How server gets the public key at start?” — it fetches the JWKS from the IdP and caches it. It may refresh periodically or when it sees an unknown key id (kid).

12) Best practices & troubleshooting

12.1 Always separate “tool selection” from “tool execution”

Let the model suggest which tool to call, but enforce policy in the host: allowed tool list, argument validation, rate limits, and safe defaults.

12.2 Validate inputs server-side

Treat MCP tool calls like API requests. Validate types, lengths, SKU formats, and scope permissions. Don’t trust that the model will always supply correct values.

12.3 Limit data exposure via resources

Just because a resource exists doesn’t mean it should be offered to all callers. Consider per-user resource filtering (e.g., only their own catalog segment or their own orders).

12.4 CORS (browser hosts)

If a browser-based host calls your MCP server cross-origin, you must configure CORS properly. Expect OPTIONS preflight if you use Authorization headers.

One-sentence summary:
MCP defines how capabilities are exposed; your host decides what the model can use; authentication and middleware make remote access safe.

Final end-to-end picture

1) Host connects to MCP server and discovers tools/resources/prompts
2) Host obtains bearer token from IdP (password/auth-key for local services; PKCE for public clients)
3) Host calls MCP server with Authorization: Bearer token
4) MCP middleware validates token (introspection or JWT+JWKS public key)
5) Host lets model call allow-listed tools; executes calls; returns results to model
6) Tools enforce scopes and perform actions (DB, APIs, etc.)
