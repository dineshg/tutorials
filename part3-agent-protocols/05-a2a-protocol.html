1) What A2A is (the mental model)

A2A is a standard way for agents (not just tools) to communicate with each other over a network. The important word is “agents”: each participant is allowed to be autonomous, stateful, long-running, and potentially opaque internally. A2A does not require agents to share their chain-of-thought, private memory, or internal tools. Instead, they collaborate via a shared external protocol.

An agent wants help → it finds another agent → it delegates work as a Task → it receives Messages and Artifacts → it continues / completes the Task.
One-liner: A2A is “HTTP + structured messages + task lifecycle” for agent-to-agent collaboration across vendors, frameworks, and deployments.

2) Why A2A exists (problems it solves)

Problem 1: Agents live in silos

One team builds an agent with Framework A, another team builds a specialist agent with Framework B, a third vendor provides a hosted agent. Without a common “language,” collaboration becomes bespoke glue code.

Problem 2: Work is long-running and multi-turn

Real workflows have approvals, waiting on humans, retries, partial progress, streaming updates, and artifacts (files, forms, structured outputs).

Problem 3: You need discovery

If an agent is going to delegate, it must discover capabilities: “Who can do inventory checks?” “Who can approve spending?” “What modalities do they accept?”

Problem 4: Enterprises need governance

Identity, authentication, authorization, audit logs, versioning, and safe handling of files/URLs must be aligned with typical enterprise security practices.

A2A is not “a prompt format.” It’s a collaboration protocol with discovery, structured data, and lifecycle.

3) Core concepts and vocabulary

These are the A2A building blocks you should hold in your head:

Concept Meaning Why it matters
Agent A network service that can accept tasks and return results. Agents are peers: each can delegate to another agent.
AgentCard A machine-readable “business card” that describes an agent’s endpoint, skills, modalities, and auth. Enables discovery and safe routing.
Skill A declared capability like “check_inventory” or “approve_purchase”. Used to choose the right agent for a subtask.
Task The core unit of delegated work with an id, status, history, and outputs. Supports long-running work and human-in-the-loop.
Message A communication unit inside a task (request, update, clarification, final answer). Multi-turn interactions without leaking internal state.
Part A piece of message content (text, file reference, structured data, etc.). Enables multi-modality and structured payloads.
Artifact An output object produced by the task (report, file, decision record, structured JSON). Separates “conversation” from “deliverables.”
Think “project management”: a Task has a status, messages are the discussion, artifacts are the deliverables.

4) Agent discovery via AgentCard

Discovery starts with the AgentCard: a JSON document published by an A2A server at a well-known path. The purpose is to make it possible for other agents (or an orchestrator) to learn:

  • Who this agent is (name/version/vendor)
  • Where to send requests (service endpoint)
  • What the agent can do (skills)
  • How to authenticate (schemes, requirements)
  • What interaction styles it supports (sync, streaming, async)
Concierge agent wants “inventory check” ↓ GET well-known AgentCard for candidate agents ↓ Choose agent whose Skills match the needed work ↓ Create Task + exchange Messages/Artifacts until complete
Security note: even though the AgentCard is a discovery document, you should treat it as a potentially sensitive surface and protect it appropriately (don’t embed secrets).

5) Skills: how agents advertise capability

A2A “Skills” are intentionally high-level: they describe what an agent does in a way another system can reason about. Skills are not the internal tool list, not chain-of-thought, and not implementation details.

A skill typically answers:
  • What the agent can do (“approve_purchase”, “inventory_availability”)
  • Inputs expected (fields, formats, constraints)
  • Outputs returned (artifacts, structured payload)
  • Constraints (needs human approval, requires certain auth scope)

Practically, your orchestrator (concierge) uses skills like a routing index: “Which agent should I hire for this sub-problem?”

6) Tasks, status, and long-running work

A2A is “async-first.” That means it is designed for tasks that may take time: waiting on a human, waiting on another system, or doing multi-step reasoning.

Typical task status states (conceptual):
  • created / accepted: task was received and scheduled
  • running: work is in progress
  • needs_input: agent is asking for clarification / missing fields
  • blocked: waiting for external system or human approval
  • succeeded: completed with artifacts
  • failed: unrecoverable error; may include error artifact

Exact enums can vary by version, but the principle is consistent: tasks have lifecycle, tasks can be polled or streamed, and tasks can produce artifacts.

Why tasks exist: If agents only did one-shot request/response, they’d fail on most real workflows. Tasks let agents coordinate across time, state, and dependencies.

7) Messages, parts, and artifacts

7.1 Messages

A Message is “what we said” within a task: requests, clarifications, updates, and final responses. Messages typically contain one or more “Parts.”

7.2 Parts

A Part is “what kind of content this is.” A2A is modality-agnostic: it can carry plain text, references to files, or structured data parts.

7.3 Artifacts

Artifacts are the outputs: a report, a decision, a JSON payload, an attached file reference, a filled form. The key design win is: artifacts represent deliverables; messages represent conversation.

If you mix “final deliverables” and “chatty conversation” into one blob of text, you make downstream automation fragile. Artifacts fix that by making outputs explicit.

8) Interaction models: sync vs streaming vs async

A2A supports multiple interaction modes so agents can choose what’s appropriate:

Mode When to use How it feels
Synchronous Fast tasks that can complete quickly. Request → immediate response message/artifact.
Streaming You want progress updates or partial results. Server-sent events / incremental updates to task state.
Asynchronous Long-running tasks, human-in-loop approvals, external dependencies. Request → “task accepted” → later poll/notify for completion.
A2A is built to treat “waiting” as normal. That’s why the task object and its status/history are central.

9) Orchestration patterns

In multi-agent systems, there are common architectures:

Concierge (Orchestrator) Pattern

One “front” agent speaks to the user and hires specialist agents for subtasks. This keeps the user experience consistent and concentrates policy + governance in one place.

User ↔ Concierge ↙ ↘ Inventory Approval
Hub-and-Spoke (Platform) Pattern

A central platform routes tasks and enforces org-wide authentication, authorization, quotas, and auditing. Individual agents remain independent services.

Agents ↔ Hub ↔ Agents
Mesh (Peer-to-Peer) Pattern

Agents directly discover and delegate to each other. Powerful, but harder to govern unless you add strong identity and allow-lists.

Agent A ↔ Agent B ↔ Agent C ↖──────────↙
Human-in-the-Loop Variant

One or more tasks become “blocked” until a human approves or provides missing information. A2A’s async-first model makes this a first-class concept.

If you’re building your first system, start with the concierge pattern. It’s easiest to debug, easiest to secure, and easiest to explain.

10) Concrete scenario: Purchasing Concierge + Inventory + Approval

Let’s implement the exact “why do we need this and what does it do?” story using A2A. Your user says: “Buy 20 keyboards for the team.” The concierge agent must:

  1. Find an inventory agent to check stock and pricing
  2. Find an approval agent to approve budget (human-in-loop)
  3. Return a final “approved purchase plan” to the user
Agents (roles)
  • Concierge Agent: user-facing, orchestrates subtasks
  • Inventory Agent: checks availability, lead time, estimated cost
  • Approval Agent: determines if purchase is allowed and records approval

10.1 Step-by-step flow (exactly what happens)

  1. Discovery: Concierge fetches AgentCards for Inventory and Approval agents.
  2. Routing: Concierge selects agents by matching required Skills.
  3. Delegate inventory task: Concierge creates a Task on Inventory agent: “Check 20 keyboards.”
  4. Inventory responds: Inventory produces an Artifact: availability + cost breakdown.
  5. Delegate approval task: Concierge creates a Task on Approval agent with the cost breakdown.
  6. Approval blocks: Approval agent returns status “blocked/needs_human,” providing a form-like Part.
  7. Human approves: Concierge (or UI) submits approval decision back to Approval agent.
  8. Approval completes: Approval agent emits an approval Artifact (decision + approver + timestamp).
  9. Concierge finalizes: Concierge returns a final plan to the user.
This is what A2A “gives you”: the structured way to discover agents, start tasks, wait safely, and collect artifacts. Without A2A, you end up hardcoding endpoints, inventing your own payload formats, and hacking long-running state.

11) Practical JSON examples (AgentCard + task lifecycle)

11.1 AgentCard (Inventory Agent) — example shape

This is an illustrative example to show what discovery documents tend to look like: identity + endpoint + skills + auth requirements. Your real card will include the fields your chosen A2A binding expects. 

// GET https://inventory.example/.well-known/agent-card.json
{
  "name": "inventory-agent",
  "version": "1.0.0",
  "description": "Checks stock, price, lead time for purchase requests.",
  "endpoint": "https://inventory.example/a2a",
  "auth": {
    "type": "oauth2",
    "audience": "inventory-api",
    "requiredScopes": ["inventory:read"]
  },
  "skills": [
    {
      "id": "inventory.check",
      "title": "Inventory Check",
      "inputSchema": {
        "type": "object",
        "properties": {
          "item": {"type": "string"},
          "quantity": {"type": "integer", "minimum": 1}
        },
        "required": ["item", "quantity"]
      }
    }
  ]
}

11.2 Create a task (Concierge → Inventory Agent)

// POST https://inventory.example/a2a/tasks
{
  "skillId": "inventory.check",
  "message": {
    "role": "requester",
    "parts": [
      { "type": "text", "text": "Check availability and price for 20 keyboards." },
      { "type": "data", "data": { "item": "keyboard", "quantity": 20 } }
    ]
  }
}

11.3 Inventory responds with status + artifact

// 200 OK (or 202 Accepted then later poll/stream)
{
  "taskId": "task_123",
  "status": { "state": "succeeded" },
  "artifacts": [
    {
      "type": "data",
      "name": "inventory_quote",
      "data": {
        "item": "keyboard",
        "quantity": 20,
        "available": 18,
        "backorderEtaDays": 7,
        "unitPrice": 35.0,
        "currency": "USD",
        "estimatedTotal": 700.0
      }
    }
  ]
}

11.4 Approval becomes blocked (human-in-the-loop)

{
  "taskId": "task_456",
  "status": {
    "state": "blocked",
    "message": "Needs manager approval: total exceeds $500."
  },
  "messages": [
    {
      "role": "agent",
      "parts": [
        { "type": "text", "text": "Please confirm manager approval to proceed." },
        { "type": "data", "data": { "approvalForm": { "approver": "", "decision": "approve|reject" } } }
      ]
    }
  ]
}
Notice the “shape” advantages: the approval agent can request structured inputs (a form-like data part), and the output can be an explicit approval artifact that downstream systems can trust.

12) Security and authentication (what you must implement)

A2A intentionally relies on standard web security practices. That means:

Transport security
  • Use HTTPS for integrity and authenticity
  • Consider mTLS in enterprise/internal networks
Authentication
  • Bearer tokens (OAuth2/OIDC) are common
  • Service-to-service credentials for private deployments
Authorization
  • Scopes/roles per skill (e.g., approval:write)
  • Don’t let “any agent” call “any skill”
File/URL safety
  • Validate file references to avoid SSRF
  • Sanitize and validate user-provided data parts
Practical governance recommendation: Use allow-lists for which agents your concierge is permitted to hire, and limit which skills can be invoked based on policy.

13) Minimal Python implementation (server + concierge client)

Below is a minimal educational implementation that captures the A2A ideas: publish an AgentCard, accept task creation, track status, and return artifacts. This is not a complete spec implementation, but it’s the fastest way to internalize how A2A “feels.”

13.1 A2A-ish Specialist Agent Server (FastAPI)

# inventory_agent.py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import uuid, time

app = FastAPI(title="Inventory Agent (A2A demo)")

# In-memory task store (demo only)
TASKS = {}

def agent_card():
    return {
        "name": "inventory-agent",
        "version": "1.0.0",
        "description": "Checks stock/price/ETA for purchase requests.",
        "endpoint": "http://127.0.0.1:9101/a2a",
        "auth": {"type": "none"},
        "skills": [
            {
                "id": "inventory.check",
                "title": "Inventory Check",
                "inputSchema": {
                    "type": "object",
                    "properties": {
                        "item": {"type": "string"},
                        "quantity": {"type": "integer", "minimum": 1}
                    },
                    "required": ["item", "quantity"]
                }
            }
        ]
    }

# Well-known discovery document (demo path)
@app.get("/.well-known/agent-card.json")
def get_agent_card():
    return agent_card()

class CreateTask(BaseModel):
    skillId: str
    message: dict

@app.post("/a2a/tasks")
def create_task(body: CreateTask):
    if body.skillId != "inventory.check":
        raise HTTPException(status_code=400, detail="Unsupported skillId")

    task_id = f"task_{uuid.uuid4().hex[:8]}"

    # Parse the structured data part if present
    item = "unknown"
    qty = 1
    for part in body.message.get("parts", []):
        if part.get("type") == "data":
            data = part.get("data", {})
            item = data.get("item", item)
            qty = data.get("quantity", qty)

    # Demo logic
    available = 18 if item.lower() == "keyboard" else 5
    unit_price = 35.0 if item.lower() == "keyboard" else 10.0
    eta_days = 7 if qty > available else 0
    total = float(qty) * unit_price

    TASKS[task_id] = {
        "taskId": task_id,
        "createdAt": time.time(),
        "status": {"state": "succeeded"},
        "artifacts": [{
            "type": "data",
            "name": "inventory_quote",
            "data": {
                "item": item,
                "quantity": qty,
                "available": available,
                "backorderEtaDays": eta_days,
                "unitPrice": unit_price,
                "currency": "USD",
                "estimatedTotal": total
            }
        }]
    }

    return TASKS[task_id]

@app.get("/a2a/tasks/{task_id}")
def get_task(task_id: str):
    t = TASKS.get(task_id)
    if not t:
        raise HTTPException(status_code=404, detail="Unknown task")
    return t

# Run:
#   pip install fastapi uvicorn
#   uvicorn inventory_agent:app --port 9101

13.2 Concierge client: discover skills, create a task, read artifact

# concierge.py
import requests

INVENTORY_BASE = "http://127.0.0.1:9101"

def discover_agent(base: str) -> dict:
    r = requests.get(base + "/.well-known/agent-card.json", timeout=5)
    r.raise_for_status()
    return r.json()

def create_inventory_task(base: str, item: str, qty: int) -> dict:
    payload = {
      "skillId": "inventory.check",
      "message": {
        "role": "requester",
        "parts": [
          {"type": "text", "text": f"Check availability and pricing for {qty} {item}s."},
          {"type": "data", "data": {"item": item, "quantity": qty}}
        ]
      }
    }
    r = requests.post(base + "/a2a/tasks", json=payload, timeout=10)
    r.raise_for_status()
    return r.json()

if __name__ == "__main__":
    card = discover_agent(INVENTORY_BASE)
    print("Discovered agent:", card["name"], "skills=", [s["id"] for s in card["skills"]])

    task = create_inventory_task(INVENTORY_BASE, "keyboard", 20)
    print("Task status:", task["status"]["state"])
    artifact = task["artifacts"][0]["data"]
    print("Quote artifact:", artifact)
Even in this simplified demo, you can see the A2A “shape”: discover an agent (AgentCard), pick a skill, create a task, receive an artifact. Expanding this to Approval + blocked/needs_input is a straightforward extension.

14) Operational checklist (retries, idempotency, observability)

Idempotency
  • Task creation should be safe to retry (idempotency keys)
  • Artifact writes should be append-only or versioned
Retries + timeouts
  • Use bounded retries with jitter
  • Different timeouts for: card fetch, create task, poll task
Observability
  • Trace IDs per task and propagate them across agents
  • Log skill invoked, caller identity, status transitions
Safety
  • Validate file references and URLs (SSRF defense)
  • Keep AgentCards free of secrets
  • Use allow-lists for which agents can be contacted
The easiest multi-agent failures are “silent failure” and “infinite waiting.” Always implement: (1) clear status transitions, (2) timeouts, (3) explicit failure artifacts.

Summary (the 30-second version)

  • AgentCard enables discovery: who the agent is, what it can do, how to call it.
  • Skills enable routing: choose the right specialist agent for a subtask.
  • Tasks enable long-running collaboration: statuses, multi-turn messages, and deliverable artifacts.
  • Messages/Parts enable structured, multi-modality communication.
  • Artifacts are explicit outputs that downstream automation can reliably consume.
