MCP Tutorial with a Simple "Database" Example

What MCP is

MCP (Model Context Protocol) is a standard way for an "AI host" (an app that uses an LLM) to connect to external capabilities through MCP servers.

Instead of hard-coding integrations (DB, CRM, ticketing, etc.) inside every app, you run a server that publishes capabilities in a consistent shape. The host connects as an MCP client, discovers what the server offers, and then uses those capabilities to answer user questions (often with an LLM making decisions).

The Key Insight: MCP provides a unified framework so the agent doesn't need to learn every system's custom API—it just talks MCP.

Why MCP Exists: The Problem It Solves

Without MCP

Every connector looks different:

  • Different endpoints, auth mechanisms, and payload shapes
  • Agents must hardcode provider-specific logic
  • Hard to secure and audit consistently
  • Hard to swap systems (e.g., DB vendor changes) without changing agent logic

Example of the Problem:

Without MCP, if you want user data, you might need to know:

  • Is it GET https://api.vendor.com/users?email=...?
  • Or SELECT * FROM users WHERE email = ...?
  • Or sdk.users.get(email)?

Each requires different code, different error handling, different security patterns.

With MCP: Unified Abstraction

MCP standardizes the interface. Whether the tool internally:

  • calls an HTTP API,
  • runs a database query,
  • invokes a SaaS SDK,
  • reads from a file system,
  • publishes to a message queue,

The agent always does:

call_tool("tool_name", {"arg": "value"})
One-Sentence Summary: MCP standardizes how agents discover safe context (resources), learn correct workflows (prompts), and execute controlled operations (tools), so underlying HTTP/DB/SaaS operations become consistent and auditable.

MCP Architecture: How the Pieces Fit Together

Three Main Components:

1. MCP Server

  • Exposes Tools / Resources / Prompts
  • Enforces its own security policies
  • Hides implementation details (DB, HTTP, SDK)

2. MCP Client (your app)

  • Connects to the server
  • Fetches prompts and resources
  • Calls tools
  • Feeds content to the LLM

3. LLM / Agent

  • Reads the prompt + policy + schema
  • Decides which tool to call and with what arguments
  • Never directly touches the DB/API—only through tools
Important: MCP is not "replacing HTTP." MCP is a standard protocol on top of whatever the server uses internally. The MCP server might call HTTP APIs, databases, SDKs, etc., but the MCP client always speaks MCP.

What MCP is used for

MCP is used when you want an LLM-powered system to:

  • Discover what it can do (what tools exist, what resources exist, what prompts exist)
  • Invoke actions safely (call a tool to fetch data or perform operations)
  • Fetch reference context reliably (read policies/schemas/config as resources)
  • Reuse standard workflows (prompts that encode "how to do this safely")
In short: MCP helps you build tool-using LLM apps without reinventing a connector interface for every integration.

Why This Pattern Matters: Key Benefits

1. Separation of Concerns: Read vs. Write

Resources are safe, cacheable "reference documents" for the agent.

Tools are the only things that touch live data (DB/API), so you can lock them down and audit them.

This prevents the "LLM invents SQL" problem and makes governance easier.

2. Reduced Coupling and Increased Portability

You can replace the server implementation (DB → HTTP API → different vendor) without changing:

  • the agent logic
  • the client tool-calling logic

...as long as the MCP tool contract remains stable.

3. Improved Security Posture

In enterprise setups, you want:

  • Strict tool allowlists
  • Input schemas and validation
  • Auditing of tool calls
  • Minimal exfiltration surface

MCP naturally fits this because the LLM can't "reach into the DB" directly—only through defined tools.

4. Standardized Workflows

Prompts standardize agent behavior across many clients. They teach the agent the safe/recommended sequence of operations.

MCP components (the 3 primitives)

1) Tools = "do something"

A Tool is a callable function exposed by the server.

  • Input: arguments (e.g., user_id)
  • Output: structured data (e.g., dict/JSON)
  • Use for: database reads, API calls, computations, actions

In this tutorial:

  • db_get_user_profile(user_id)
  • db_get_subscription(user_id)

2) Resources = "read something"

A Resource is read-only content, addressed by a URI.

  • Input: a URI like db://tooling/guide
  • Output: text or JSON content
  • Use for: tool guides, schemas, policy text, reference docs

In this tutorial:

  • db://tooling/guide (instructions for when to call which tool)

3) Prompts = "how to think / workflow templates"

A Prompt is a server-provided template that returns structured messages (system/user) to feed to an LLM.

  • Input: parameters (e.g., question, user_id)
  • Output: "messages" that guide the LLM reliably

In this tutorial:

  • account_assistant(question, user_id) returns a workflow telling the LLM:
    1. read the guide resource,
    2. choose the correct tool(s),
    3. summarize results.

Example 1: Single User Lookup (Simple)

User asks:

"Confirm my name and what plan I'm on."

The LLM should:

  • call db_get_user_profile(user_id) for name/email
  • call db_get_subscription(user_id) for plan/renewal
  • combine them into a single answer

We'll expose:

  • 2 tools (two "DB reads")
  • 1 resource (a read-only tool guide)
  • 1 prompt (a standard workflow template)

Part A — MCP Server (Tools + Resource + Prompt)

Save as: mcp_db_server.py

mcp_db_server.py
# pip install "mcp[cli]"

from mcp.server.fastmcp import FastMCP
from mcp.server.fastmcp.prompts import base

# This creates an MCP server instance.
# "json_response=True" means tool outputs can be structured JSON.
mcp = FastMCP("DBMCP", json_response=True)


# ============================================================
# RESOURCE: read-only context
# ============================================================
# URI: db://tooling/guide
# The client reads this via session.read_resource("db://tooling/guide")
#
# Think of this as "documentation the LLM can read":
# - what tools exist
# - when to use which tool
# - rules (like "user_id must come from host identity")
@mcp.resource("db://tooling/guide")
def tooling_guide() -> str:
    # Returned value is just text (could also be JSON).
    # The MCP client will receive it as resource contents.
    return (
        "Tool guide:\n"
        "- Use db_get_user_profile(user_id) for: name, email, profile.\n"
        "- Use db_get_subscription(user_id) for: plan, billing, renew, payment.\n\n"
        "Rules:\n"
        "- user_id must come from the host identity, not from user text.\n"
        "- If question asks for both profile + plan, call BOTH tools.\n"
    )


# ============================================================
# PROMPT: reusable workflow template
# ============================================================
# Name: account_assistant (the function name is the prompt name)
#
# The client calls this via session.get_prompt("account_assistant", arguments={...})
# The server returns structured messages (system/user) suitable for an LLM.
#
# This is NOT a tool call. It does not read DB. It provides instructions.
@mcp.prompt(title="Account assistant")
def account_assistant(question: str, user_id: str) -> list[base.Message]:
    return [
        base.SystemMessage(
            "You are an account assistant.\n"
            "Workflow:\n"
            "1) Read resource db://tooling/guide\n"
            "2) Decide which tool(s) to call\n"
            "3) Call the tool(s)\n"
            "4) Summarize results briefly\n"
        ),
        base.UserMessage(
            # This is the actual user task formatted in a consistent way.
            f"user_id={user_id}\nquestion={question}"
        ),
    ]


# ============================================================
# TOOL 1: action (DB read) - profile
# ============================================================
# Tools are invoked by the client as: tools/call with name + arguments.
@mcp.tool()
async def db_get_user_profile(user_id: str) -> dict:
    # In real life: query your database.
    # Here: mocked data.
    return {
        "user_id": user_id,
        "full_name": "Alice Example",
        "email": "alice@bank.com",
    }


# ============================================================
# TOOL 2: action (DB read) - subscription
# ============================================================
@mcp.tool()
async def db_get_subscription(user_id: str) -> dict:
    # In real life: query your database.
    # Here: mocked data.
    return {
        "user_id": user_id,
        "plan": "Pro",
        "renews_at": "2026-01-10",
    }


if __name__ == "__main__":
    # Run the MCP server over Streamable HTTP (network server).
    # Your MCP client will connect to http://127.0.0.1:8765/mcp (default).
    mcp.run(transport="streamable-http")

Run it:

python mcp_db_server.py

Part B — MCP Client + LangChain LLM (uses Prompt + Resource + Tools)

Save as: mcp_client_langchain.py

mcp_client_langchain.py
# pip install langchain langchain-openai langchain-mcp-adapters mcp
# export OPENAI_API_KEY="..."

import asyncio
from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage

from langchain_mcp_adapters.client import MultiServerMCPClient
from langchain_mcp_adapters.resources import load_mcp_resources
from langchain_mcp_adapters.prompts import load_mcp_prompt

MCP_URL = "http://127.0.0.1:8765/mcp"

# IMPORTANT: this is host-derived identity (from your auth system in real life).
# The user/LLM should not be allowed to choose it.
USER_ID = "u_742"

async def main():
    # ------------------------------------------------------------
    # 1) Connect to MCP server
    # ------------------------------------------------------------
    mcp_client = MultiServerMCPClient({
        "db": {"transport": "http", "url": MCP_URL}
    })

    # ------------------------------------------------------------
    # 2) Load server tools -> LangChain tools
    # ------------------------------------------------------------
    # tools is a list of LangChain Tool wrappers.
    # Example:
    #   tools[0].name == "db_get_user_profile"
    #   tools[1].name == "db_get_subscription"
    tools = await mcp_client.get_tools()

    # Create a tool-capable LLM and bind these tools so it can request tool calls.
    llm = ChatOpenAI(model="gpt-4.1-mini", temperature=0).bind_tools(tools)

    question = "Confirm my name and what plan I'm on."

    # ------------------------------------------------------------
    # 3) Pull Resource + Prompt from MCP server
    # ------------------------------------------------------------
    async with mcp_client.session("db") as session:
        # 3A) Read the Resource (guide/policy text)
        # blobs is a list of Blob objects returned by the adapter.
        # Since we ask for 1 URI, blobs typically contains exactly 1 element.
        blobs = await load_mcp_resources(session, uris=["db://tooling/guide"])

        # guide_text is a plain string extracted from the Blob:
        # example:
        # "Tool guide:\n- Use db_get_user_profile(user_id) for: ...\n..."
        guide_text = blobs[0].as_string() if blobs else ""

        # 3B) Get the Prompt (server-defined messages for the LLM)
        # prompt_msgs is a list of LangChain message objects.
        # typically something like:
        #   [SystemMessage("...workflow..."), HumanMessage("user_id=... question=...")]
        prompt_msgs = await load_mcp_prompt(
            session,
            "account_assistant",
            arguments={"question": question, "user_id": USER_ID}
        )

    # ------------------------------------------------------------
    # 4) Build the messages we send to the LLM
    # ------------------------------------------------------------
    messages = []

    # Add the resource guide as system context.
    # This lets the LLM learn "which tool to use for which question".
    if guide_text.strip():
        messages.append(SystemMessage(content="(MCP Resource) Tool guide:\n" + guide_text))

    # Add the server-defined prompt messages (workflow template).
    messages.extend(prompt_msgs)

    # ------------------------------------------------------------
    # 5) Agent loop: ask LLM, run tool calls, feed results back
    # ------------------------------------------------------------
    while True:
        # Ask the LLM for the next step
        ai_msg = await llm.ainvoke(messages)
        messages.append(ai_msg)

        # If the LLM requests tool calls, they appear here.
        # Example tool_calls:
        # [
        #   {"name":"db_get_user_profile","args":{"user_id":"u_742"}},
        #   {"name":"db_get_subscription","args":{"user_id":"u_742"}}
        # ]
        tool_calls = getattr(ai_msg, "tool_calls", None)

        # If no tool calls, the model produced a final natural-language answer.
        if not tool_calls:
            print("\nFINAL ANSWER:")
            print(ai_msg.content)
            break

        # Execute each tool call the model requested
        for tc in tool_calls:
            tool_name = tc["name"]   # string: which tool to run
            tool_args = tc["args"]   # dict: arguments for that tool

            # SAFETY: force user_id from host, not from the model.
            # Even if the model tried to pass a different user_id, we override it.
            if "user_id" in tool_args:
                tool_args["user_id"] = USER_ID

            # Find the tool object by name
            tool = next(t for t in tools if t.name == tool_name)

            # Run the tool. tool_result is usually a dict returned by the MCP tool.
            # Example:
            # {"user_id":"u_742","full_name":"Alice Example","email":"alice@bank.com"}
            tool_result = await tool.ainvoke(tool_args)

            # Feed tool output back to the LLM as system context.
            messages.append(SystemMessage(content=f"(Tool result) {tool_name}: {tool_result}"))

if __name__ == "__main__":
    asyncio.run(main())

Run it:

python mcp_client_langchain.py

Example 2: Multi-Tool Router (Advanced)

This example demonstrates the power of MCP's unified framework with:

  • 2 tools: profile lookup and orders lookup
  • 2 prompts: workflow for each tool
  • 3 resources: shared policy and separate schemas
  • LLM routing: agent chooses which tool to call based on the question

Server Code (Multiple Tools + Resources + Prompts)

mcp_server_advanced.py

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("Bank-DB-MCP", json_response=True)

# -------------------------
# RESOURCES (read-only)
# -------------------------

@mcp.resource("db://policy/access")
def policy_access() -> str:
    return (
        "POLICY:\n"
        "- Only allow lookups by email (no bulk export).\n"
        "- Return only permitted fields.\n"
        "- If request is ambiguous, ask a clarifying question.\n"
    )

@mcp.resource("db://schema/user_profile")
def schema_user_profile() -> str:
    return (
        "SCHEMA user_profile:\n"
        "- email: string (unique)\n"
        "- full_name: string\n"
        "- tier: enum(silver|gold|platinum)\n"
        "- status: enum(active|suspended)\n"
        "Permitted output fields: email, full_name, tier, status\n"
    )

@mcp.resource("db://schema/user_orders")
def schema_user_orders() -> str:
    return (
        "SCHEMA user_orders:\n"
        "- email: string\n"
        "- orders: list of { order_id: string, total: number, state: string }\n"
        "Permitted output fields: email, orders\n"
    )

# -------------------------
# PROMPTS (server workflows)
# -------------------------

@mcp.prompt()
def workflow_user_profile() -> str:
    return (
        "WORKFLOW: User Profile Lookup\n"
        "1) Read db://policy/access\n"
        "2) Read db://schema/user_profile\n"
        "3) Call tool db_get_user_profile with {\"email\": \"...\"}\n"
        "4) Return ONLY permitted fields.\n"
    )

@mcp.prompt()
def workflow_user_orders() -> str:
    return (
        "WORKFLOW: User Orders Lookup\n"
        "1) Read db://policy/access\n"
        "2) Read db://schema/user_orders\n"
        "3) Call tool db_get_user_orders with {\"email\": \"...\"}\n"
        "4) Return ONLY permitted fields.\n"
    )

# -------------------------
# TOOLS (actions)
# -------------------------

@mcp.tool()
def db_get_user_profile(email: str) -> dict:
    # Unified interface: whether this internally calls HTTP, SQL, or SDK,
    # the client always uses: call_tool("db_get_user_profile", {"email": "..."})
    fake_profiles = {
        "alice@bank.com": {
            "email": "alice@bank.com",
            "full_name": "Alice Chen",
            "tier": "gold",
            "status": "active"
        },
        "bob@bank.com": {
            "email": "bob@bank.com",
            "full_name": "Bob Singh",
            "tier": "silver",
            "status": "suspended"
        },
    }
    return fake_profiles.get(email) or {"error": "not_found"}

@mcp.tool()
def db_get_user_orders(email: str) -> dict:
    fake_orders = {
        "alice@bank.com": {
            "email": "alice@bank.com",
            "orders": [
                {"order_id": "A1001", "total": 129.50, "state": "PAID"},
                {"order_id": "A1002", "total": 58.00, "state": "REFUNDED"},
            ],
        }
    }
    return fake_orders.get(email) or {"error": "not_found"}

if __name__ == "__main__":
    mcp.run(transport="streamable-http")

Client Code (LLM Router + Multiple Resources)

mcp_client_advanced.py
# filepath: /Users/dineshgamage/Developer/pytorch/END/mcp_client_advanced.py
# pip install mcp

import asyncio
import json
import re
from typing import Any, Dict, Literal, TypedDict

from mcp import ClientSession
from mcp.client.streamable_http import streamable_http_client

MCP_URL = "http://localhost:8000/mcp"


class ToolDecision(TypedDict):
    tool_name: Literal["db_get_user_profile", "db_get_user_orders"]
    arguments: Dict[str, Any]
    prompt_to_use: Literal["workflow_user_profile", "workflow_user_orders"]


def extract_email(text: str) -> str | None:
    m = re.search(r"[\w\.-]+@[\w\.-]+\.\w+", text)
    return m.group(0) if m else None


def llm_choose_tool(
    user_question: str,
    workflow_profile: str,
    workflow_orders: str,
    policy_text: str,
    schema_profile: str,
    schema_orders: str,
) -> ToolDecision:
    """
    This simulates what GPT would do:
    - reads workflows + policy + schemas
    - picks the right tool
    - returns a structured decision

    In production, replace this with a real LLM call that outputs JSON.
    """
    email = extract_email(user_question) or "alice@bank.com"

    # Minimal routing logic (standing in for LLM reasoning)
    if "order" in user_question.lower() or "purchase" in user_question.lower():
        return {
            "tool_name": "db_get_user_orders",
            "arguments": {"email": email},
            "prompt_to_use": "workflow_user_orders"
        }
    else:
        return {
            "tool_name": "db_get_user_profile",
            "arguments": {"email": email},
            "prompt_to_use": "workflow_user_profile"
        }


async def main():
    user_question = "Show me the orders for alice@bank.com"

    async with streamable_http_client(MCP_URL) as (read, write, _):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # 1) Discover server-provided workflows (prompts)
            p_profile = await session.get_prompt("workflow_user_profile")
            p_orders = await session.get_prompt("workflow_user_orders")

            workflow_profile = json.dumps(p_profile, indent=2, default=str)
            workflow_orders = json.dumps(p_orders, indent=2, default=str)

            # 2) Read shared policy + schemas (resources)
            r_policy = await session.read_resource("db://policy/access")
            r_schema_profile = await session.read_resource("db://schema/user_profile")
            r_schema_orders = await session.read_resource("db://schema/user_orders")

            policy_text = json.dumps(r_policy, indent=2, default=str)
            schema_profile = json.dumps(r_schema_profile, indent=2, default=str)
            schema_orders = json.dumps(r_schema_orders, indent=2, default=str)

            # 3) LLM chooses which tool to call
            decision = llm_choose_tool(
                user_question=user_question,
                workflow_profile=workflow_profile,
                workflow_orders=workflow_orders,
                policy_text=policy_text,
                schema_profile=schema_profile,
                schema_orders=schema_orders,
            )

            print("\n--- TOOL DECISION (what GPT would output) ---")
            print(json.dumps(decision, indent=2))

            # 4) Client executes the chosen tool via MCP
            tool_name = decision["tool_name"]
            args = decision["arguments"]

            result = await session.call_tool(tool_name, arguments=args)

            print("\n--- TOOL RESULT (from MCP server) ---")
            print(result.structuredContent or {"content": result.content})


if __name__ == "__main__":
    asyncio.run(main())

Summary: Why MCP Matters

  1. MCP gives a unified "capabilities interface"

    To the agent/client, tools look like callable functions, resources look like URIs you can read, and prompts look like workflow templates—regardless of the underlying implementation.

  2. MCP separates "read-only context" from "actions"

    This is a big safety and design win. Resources are safe, audit-friendly, and cacheable. Tools are gated, validated, and permissioned.

  3. MCP reduces coupling and increases portability

    You can replace the server implementation without changing the agent logic or client tool-calling logic, as long as the MCP tool contract remains stable.

  4. MCP improves security posture

    The LLM can't "reach into the DB" directly—only through defined, auditable tools with strict allowlists and input schemas.
