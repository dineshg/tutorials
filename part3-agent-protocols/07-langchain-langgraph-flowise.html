Bridge from HTTP → “dict pipelines”

You already learned HTTP methods (GET/POST/PUT/PATCH/DELETE). When you build LLM apps, you’ll usually expose them as HTTP endpoints too. The key connection:

HTTP request body is typically JSON → in Python it becomes a dict.
LangChain/LangGraph pipelines are easiest to understand as “functions that transform dictionaries”.
Client JSON
└─> Python dict
└─> LangChain runnable pipeline (dict -> dict/string)
└─> HTTP JSON response
Visual: request → dict → runnable pipeline

This is the “dict pipeline” mental model: you’re always passing one structured object through transforms.

LangChain: everything is a runnable that transforms inputs → outputs

1) The “passing dictionaries” mental model

Think of a chain as a pipeline that receives a dictionary, adds/changes keys, and eventually returns a final output.

Input dict 
{
  "user_id": "u_123",
  "question": "What is my total spend last month?"
}
Output 
"Your total spend last month was $1,245.40"

Internally, intermediate steps might create keys like sql, rows, summary.

2) Runnables: one interface + composition

A Runnable is “anything” you can call in a consistent way: one input → one output.

ConceptWhat it meansWhy you care
invoke(input) Run once, return output Simple request/response
batch([inputs]) Run many inputs efficiently Lower cost + faster throughput
stream(input) Yield partial outputs as they’re produced Great UX for chat/long responses
Composition rule (LCEL): pipe output of one runnable into the next
prompt_template | llm | output_parser

Mentally: output = parser(llm(prompt(input)))

Why this “uniform runnable” idea is powerful:
Once everything speaks the same interface, you can mix-and-match prompts, models, retrievers, parsers, and custom logic like LEGO.
Visual: one interface, different runtimes

3) RunnablePassthrough (and “keeping keys alive”)

In real pipelines you often need to compute something new without losing the original input. That’s where “passthrough” patterns help.

Typical RAG shape (conceptual) 
{
  "question": "...",
  "context": "<retrieved docs text>"
}  --> prompt --> llm --> answer

RunnablePassthrough behaves like identity, and can also be used with an “assign” pattern to add keys. 

# Conceptual example (LangChain LCEL style)
from operator import itemgetter
from langchain_core.runnables import RunnablePassthrough

# Suppose you already have: retriever, format_docs, prompt, llm, parser
setup = RunnablePassthrough.assign(
    context=itemgetter("question") | retriever | format_docs
)

chain = setup | prompt | llm | parser

result = chain.invoke({"question": "What are the top 3 products by revenue?"})
print(result)
Key mental model: Passthrough keeps your dict intact, while letting you attach extra computed fields.

4) Custom node creation in LangChain (custom runnable)

“Node” isn’t an official LangChain term the way it is in graph frameworks, but developers often call each pipeline step a “node”. The simplest way to create a custom node is to wrap a Python function as a runnable. 

from langchain_core.runnables import RunnableLambda

def normalize_question(inputs: dict) -> dict:
    q = inputs["question"].strip()
    return {**inputs, "question": q}

normalize_node = RunnableLambda(normalize_question)

# Now it composes like any other runnable:
# normalize_node | prompt | llm | parser
Rule of thumb
  • Input and output can be strings, lists, or dicts… but dicts give you the most flexibility.
  • Prefer “pure transforms” (no side effects) for easy debugging.
  • If you need side effects (DB writes, HTTP calls), isolate them in a clearly-named node.

5) Mini end-to-end LangChain example (SQL helper, dict-first)

Here’s a tiny pipeline that: 1) builds SQL from a question, 2) executes it, 3) summarizes rows. This is intentionally small so you can “see the dictionaries move”. 

from langchain_core.runnables import RunnableLambda, RunnablePassthrough
from operator import itemgetter

# --- pretend implementations ---
def llm_write_sql(inputs: dict) -> dict:
    # In practice: prompt | llm | parser
    q = inputs["question"]
    sql = f"SELECT SUM(amount) AS total FROM orders WHERE month = 'last' -- for: {q}"
    return {**inputs, "sql": sql}

def run_sql(inputs: dict) -> dict:
    # In practice: call your DB driver here
    rows = [{"total": 1245.40}]
    return {**inputs, "rows": rows}

def summarize(inputs: dict) -> str:
    total = inputs["rows"][0]["total"]
    return f"Your total spend last month was ${total:,.2f}"

# --- wrap as nodes ---
write_sql = RunnableLambda(llm_write_sql)
execute = RunnableLambda(run_sql)
summarize_node = RunnableLambda(summarize)

# --- keep dict, add keys, finish with a string ---
chain = (
    RunnablePassthrough()   # starts as identity (keeps original dict)
    | write_sql             # adds sql
    | execute               # adds rows
    | summarize_node        # returns final string
)

print(chain.invoke({"user_id": "u_123", "question": "What is my total spend last month?"}))
What you should notice:
Each step either (a) returns a new dict with extra keys, or (b) returns the final answer. That’s the “passing dictionaries” idea in one page.

LangGraph: stateful graphs (State → partial State)

1) State passing

LangGraph is what you use when a simple linear chain becomes a workflow: branching, loops, retries, tool-calling cycles, or multi-step “agent” behavior.

Core rule:
Each node reads State and returns a partial update to State
Visual: State → partial State updates

You usually define a typed state (e.g., TypedDict) and each node returns only the keys it changed.

Reducers (why they exist):
If multiple nodes update the same key (e.g., message history), reducers define how those updates are merged (append, overwrite, etc.).

2) Example: “purchase request” graph (inventory + approval)

This mirrors the A2A story you built earlier (concierge → inventory + approval), but inside a single LangGraph workflow with shared state.

Graph shape
START | parse_request | check_inventory ----(out_of_stock)----> finalize | request_approval ----(rejected)-------> finalize | finalize | END
Visual: branching workflow 
from typing import TypedDict, Literal, Optional
from langgraph.graph import StateGraph, END

class PurchaseState(TypedDict, total=False):
    # input
    sku: str
    qty: int
    amount: float
    po_id: str

    # derived/intermediate
    in_stock: bool
    reserved: bool
    approval: Literal["approved", "rejected", "skipped"]
    reason: Optional[str]

    # final
    result: str

# --- nodes ---
def parse_request(state: PurchaseState) -> PurchaseState:
    # In real life, parse from user text. Here we assume state already has fields.
    return {}

def check_inventory(state: PurchaseState) -> PurchaseState:
    # pretend inventory lookup
    stock_map = {"SKU-APPLE": 12, "SKU-CHAIR": 0}
    available = stock_map.get(state["sku"], 0)
    if available < state["qty"]:
        return {"in_stock": False, "reserved": False, "reason": f"Only {available} left"}
    return {"in_stock": True, "reserved": True}

def request_approval(state: PurchaseState) -> PurchaseState:
    # pretend policy: auto-approve up to 1000
    if state["amount"] <= 1000:
        return {"approval": "approved"}
    return {"approval": "rejected", "reason": "Over auto-approval limit"}

def finalize(state: PurchaseState) -> PurchaseState:
    if not state.get("in_stock", True):
        return {"result": f"FAILED: out of stock ({state.get('reason')})"}
    if state.get("approval") == "rejected":
        return {"result": f"FAILED: approval rejected ({state.get('reason')})"}
    return {"result": "SUCCESS: reserved and approved"}

# --- build graph ---
g = StateGraph(PurchaseState)
g.add_node("parse_request", parse_request)
g.add_node("check_inventory", check_inventory)
g.add_node("request_approval", request_approval)
g.add_node("finalize", finalize)

g.set_entry_point("parse_request")
g.add_edge("parse_request", "check_inventory")

# conditional branching after inventory
def inventory_router(state: PurchaseState):
    return "finalize" if not state.get("in_stock", True) else "request_approval"

g.add_conditional_edges("check_inventory", inventory_router, {
    "finalize": "finalize",
    "request_approval": "request_approval",
})

# conditional branching after approval
def approval_router(state: PurchaseState):
    return "finalize"

g.add_conditional_edges("request_approval", approval_router, {
    "finalize": "finalize",
})

g.add_edge("finalize", END)

app = g.compile()

print(app.invoke({"sku": "SKU-APPLE", "qty": 2, "amount": 4.50, "po_id": "PO-1001"})["result"])
print(app.invoke({"sku": "SKU-CHAIR", "qty": 1, "amount": 150.0, "po_id": "PO-2002"})["result"])
print(app.invoke({"sku": "SKU-APPLE", "qty": 1, "amount": 2500.0, "po_id": "PO-3003"})["result"])
Key difference vs LangChain chain:
LangChain pipelines are typically linear transforms; LangGraph makes the “workflow control” (branching/loops) explicit and stateful.

Flowise: LangChain concepts as a UI

Flowise takes the same building blocks you used in LangChain (LLM, prompt, retriever, memory, tools, custom code) and represents them as drag-and-drop nodes.

In code (LangChain)
input dict
└─> prompt | llm | parser
└─> output
In UI (Flowise)
[Prompt Node] → [LLM Node] → [Output Parser Node]
(+ optional Retriever/Memory/Tools nodes)
Visual: code pipeline ↔ UI nodes

Same building blocks — Flowise just makes the wiring visual.

How Flowise maps to the “passing dictionaries” idea

  • Each UI node corresponds to a component (often a LangChain runnable under the hood).
  • Edges are “data flow”: output of one node becomes input to the next.
  • Node settings in the UI become the parameters you’d pass in code (model name, temperature, prompt template, etc.).
Practical workflow:
Prototype in Flowise (fast), then replicate the same structure in LangChain/LangGraph code when you need version control, tests, and CI/CD.

Where LangGraph shows up in UI tools

Flowise can represent more complex workflows too, but graph-style systems (branching, loops, “agent cycles”, state management) are where LangGraph shines in code. In UIs, you’ll often see those features as conditional nodes, routers, or “agent” templates.

Rule of thumb
  • LangChain: compose steps; treat it like a pipeline of transforms.
  • LangGraph: when you need workflow logic + durable shared state.
  • Flowise: visual builder that mirrors these components as draggable nodes.
