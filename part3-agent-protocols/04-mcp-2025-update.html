MCP 2025 Update — Streamable HTTP, OAuth 2.1, Elicitation, Structured Output

Why this chapter exists Chapters 1–3 cover MCP from first principles using the original mcp[cli] server and either stdio or HTTP+SSE transport. The MCP spec versions 2025-03-26 and 2025-06-18 made several important changes. This chapter is a focused diff so the rest of Part III remains conceptually correct but you know what to use in production.

1) Transport: HTTP+SSE → Streamable HTTP

In the older spec, an MCP server over HTTP exposed two endpoints — a POST /messages for client‑to‑server requests and a parallel GET /sse Server‑Sent‑Events stream for server‑to‑client messages. This was hard to deploy behind enterprise load balancers (sticky sessions, two endpoints, idle‑timeout edge cases).

The new Streamable HTTP transport collapses this into a single endpoint, typically /mcp. The semantics are:

  • POST /mcp with a JSON‑RPC request: the server may respond with a single JSON response or upgrade the response to an SSE stream that delivers the response plus any progress notifications.
  • GET /mcp: opens an SSE stream the server can use to push messages independently of any in‑flight request (e.g. resource changes, sampling, elicitation).
  • A Mcp-Session-Id header issued on initialize keeps the two sides correlated through any proxy.

What this means for the code in Chapter 1

Practically nothing changes for the user — FastMCP already calls mcp.run(transport="streamable-http") in the example. The new transport is the default for HTTP servers in current MCP SDKs, and the old HTTP+SSE transport is deprecated.

2) Authorization: pinned to OAuth 2.1 + Protected Resource Metadata

Chapter 12.1 in the original tutorials sketched a “bearer + middleware” pattern. The 2025 spec formalizes this and is now an OAuth 2.1 profile:

  • The MCP server is a protected resource. It publishes /.well-known/oauth-protected-resource (RFC 9728), pointing the client at one or more authorization servers it trusts.
  • Clients use OAuth 2.1 — Authorization Code + PKCE only, no implicit, no password grant. (Same flow as Chapter 7, just the newer consolidated spec.)
  • Tokens are validated as in Chapter 9: JWKS for JWTs, introspection for opaque tokens.
  • The resource indicator (resource= parameter, RFC 8707) is required, so a token issued for one MCP server cannot be silently replayed at another.
  • For MCP servers that lack a built‑in IdP, the spec recommends Dynamic Client Registration (RFC 7591) so clients can register on the fly.
Tie‑in to Part II Combine this with Chapter 15 — Modern auth additions to get sender‑constrained tokens at MCP boundaries: DPoP for browser/host clients and mTLS for workload‑to‑MCP calls. Token exchange (RFC 8693) is the cleanest way for a host service to forward end‑user identity to a downstream MCP server.

3) Elicitation — server asks the user for input mid‑task

Previously, an MCP tool that needed extra info (a missing parameter, a confirmation, a clarification) had to either fail the call or guess. The new elicitation/create method lets the server ask the user for input via the host:

# Server side (hypothetical FastMCP API; check your SDK's exact name)
@mcp.tool()
async def book_flight(ctx, origin: str, destination: str) -> dict:
    if not origin:
        result = await ctx.elicit(
            message="Which city should I book the flight from?",
            schema={"type": "object",
                    "properties": {"origin": {"type": "string"}},
                    "required": ["origin"]},
        )
        if result.action != "accept":
            return {"error": "user_declined"}
        origin = result.content["origin"]
    return await reserve(origin, destination)

The host displays the question, collects a structured response that matches the schema, and returns either accept, decline, or cancel. This is the cleanest pattern for human‑in‑the‑loop confirmations on destructive tools.

4) Structured tool output

In the original chapters, a tool returned text content and the LLM had to re‑parse it. The new spec lets a tool declare a JSON Schema for its return value and emit both a human‑readable rendering and a structured result:

@mcp.tool(
    output_schema={"type":"object",
                   "properties":{"price":{"type":"number"},
                                 "currency":{"type":"string"}}},
)
def quote(symbol: str) -> dict:
    return {"price": 142.55, "currency": "USD"}

Clients receive structuredContent on the tool result, which the host application can render natively (charts, badges, JSON viewers) instead of round‑tripping through the LLM.

5) Resource links — tools can return references, not blobs

Long‑form data (PDFs, search results, table dumps) bloats the model's context window. Tools can now return resource links — typed URIs the host can fetch on demand or pass back to the model as needed:

{
  "content": [
    {"type": "text", "text": "I found 3 matching contracts."},
    {"type": "resource_link",
     "uri": "doc://contracts/2024-Q4/acme-msa.pdf",
     "mimeType": "application/pdf",
     "name": "Acme MSA (2024-Q4)"}
  ]
}

The model sees a small handle; only when the host (or the LLM via resources/read) fetches the link does the actual bytes enter the loop. This is the right primitive for RAG‑style answers.

6) Other notable changes you should know about

  • Tool annotations — read‑only / destructive / idempotent / open‑world flags so hosts can decide which tools to require user confirmation for.
  • Progress notifications on long‑running tools, surfaced via the SSE side of Streamable HTTP.
  • Sampling — server‑initiated requests for the host to run LLM completions on its behalf. Lets a server compose a multi‑step reasoning chain without embedding a model.
  • JSON‑RPC batching was removed in 2025-06-18. If your old client sent batched requests, switch to one request per call.
