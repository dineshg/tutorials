HTTP methods (the “verbs” of the web) — beginner friendly

HTTP methods tell the server (and intermediaries like browsers, proxies, CDNs) what you intend to do with a URL: read, create, replace, patch, delete. That intent unlocks important behavior like caching, retries, idempotency, and clearer security controls.

Quick takeaway

  • GET read (safe)
  • POST create or run an action
  • PUT replace full resource
  • PATCH partial update
  • DELETE remove

You’ll use the ⭐ methods 99% of the time.

Core ideas you should know

These concepts explain why method choice matters (caching, retries, correctness).

Safe vs unsafe

Safe means: “should not change server state” (read-only).

Safe: GET, HEAD, OPTIONS Unsafe: POST, PUT, PATCH, DELETE

Idempotent vs non-idempotent

Idempotent means: repeating the same request leads to the same final state as doing it once.

Idempotent: GET, PUT, DELETE, HEAD, OPTIONS Often non-idempotent: POST (and sometimes PATCH)

Cacheable (common in practice)

GET is commonly cacheable Others are usually not cached unless explicitly configured

Beginner rule: Pick the method that matches intent first. Then think about safety, idempotency, caching, and security. Method + URL = contract

Widely used methods (you’ll use these 99% of the time)

⭐ GET — read

Use when: fetching data (read-only).

Example: GET /users/42 → return user 42

Safe Idempotent Commonly cacheable

Full request message

GET /users/42?include=orders HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Accept: application/json

FastAPI

from fastapi import FastAPI

app = FastAPI()

@app.get("/users/{user_id}")
def get_user(user_id: int, include: str | None = None):
    return {"user_id": user_id, "include": include}
Beginner tip: If you can bookmark it, refresh it, or share the URL and it still makes sense, it’s probably a GET.

⭐ POST — create or “do an action”

Use when:

  • Creating a new resource under a collection (POST /orders)
  • Triggering an operation that doesn’t map cleanly to CRUD (POST /reports/run)

Example: POST /orders with body {"items": [...]} → creates a new order

Unsafe Not idempotent by default Body usually contains the “create/action” input

Full request message

POST /orders HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Content-Type: application/json
Accept: application/json

{ "items": [{"sku": "ABC", "qty": 2}] }

FastAPI

from pydantic import BaseModel

class CreateOrder(BaseModel):
    items: list[dict]

@app.post("/orders")
def create_order(payload: CreateOrder):
    return {"created": True, "order_id": 123, "items": payload.items}

⭐ PUT — replace (or create at a known URL)

Use when: you want the resource at a URL to become exactly what you send.

Example: PUT /users/42 with a full user object → replaces user 42

Unsafe Idempotent Body usually contains the full resource state

Full request message

PUT /users/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Content-Type: application/json
Accept: application/json

{ "id": 42, "name": "Asha", "email": "asha@example.com" }
Common beginner bug: Using PUT with a partial body.
By intent, PUT is “replace the full resource”, so missing fields might get wiped unless you implement merge behavior.

⭐ PATCH — partial update

Use when: updating only some fields.

Example: PATCH /users/42 with {"email": "new@x.com"}

Unsafe Idempotency depends on design Body contains only changed fields (usually)

Example PATCH request message

PATCH /users/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Content-Type: application/json
Accept: application/json

{ "email": "new@x.com" }
Idempotency note: A patch like “set email to X” is usually idempotent. A patch like “increment login_count by 1” is not.

⭐ DELETE — remove

Use when: deleting a resource.

Example: DELETE /users/42

Unsafe Idempotent in intent
Beginner intuition: Deleting twice should still end “deleted”. The second delete often returns 404 or 204 depending on API design.

Common but less used (still important)

HEAD — like GET, but only headers

Use when: you want metadata without downloading the body.

Example: check if a file changed using ETag/Last-Modified.

OPTIONS — “what’s allowed here?”

Use when: discovering supported methods/headers.

Most common real-world use: CORS preflight (browser checks if it can call your API cross-origin).

If you ever see a browser sending an OPTIONS request before your real request, that’s usually CORS, not “your frontend being weird”.

Rare / niche methods (you’ll see them sometimes)

TRACE — diagnostic (often disabled)

Often disabled for security reasons.

CONNECT — tunneling (proxies + HTTPS)

Used to establish a tunnel through an HTTP proxy (common in corporate networks).

Practical mapping to REST-style endpoints

Collections vs single resources

Type Method Example Meaning
Collection GET GET /items List/search items
Collection POST POST /items Create a new item (server assigns ID)
Single resource GET GET /items/{id} Read one item
Single resource PUT PUT /items/{id} Replace item (or create if not exists)
Single resource PATCH PATCH /items/{id} Partial update
Single resource DELETE DELETE /items/{id} Delete

The “PUT vs POST feels same” confusion (quick intuition)

POST = “You decide what happens”

Think: “Here’s something; you decide what happens (often create a new thing).” The server usually chooses the new ID.

PUT = “Make this URL have this exact content”

Think: “Make /users/42 look exactly like this payload.” You’re targeting a specific URL.

If you repeat the same request:
  • Repeating PUT should converge to the same final state (idempotent).
  • Repeating POST can create duplicates unless you add idempotency keys.

What an HTTP request looks like

A client talks to your FastAPI backend by sending an HTTP request message:

METHOD /path?query=string HTTP/1.1
Header-Name: value
Authorization: Bearer <token>
Content-Type: application/json

{ "optional": "body" }
  • METHOD: intent (GET/POST/PUT/PATCH/DELETE)
  • /path: which resource (e.g. /documents/123)
  • ?query=…: options/filters (e.g. ?limit=20)
  • Headers: metadata (auth token, content type)
  • Body: payload (often JSON; usually only for POST/PUT/PATCH)

How FastAPI “decodes” the message

FastAPI turns parts of the HTTP message into Python values using type hints:

  • Path params → function args (e.g. user_id: int)
  • Query params → function args (e.g. limit: int = 20)
  • Headers → via Header(...) or security dependencies
  • JSON body → Pydantic models (request schemas)

If types don’t match, FastAPI returns a 422 validation error automatically.
