Pydantic Data Models: From Python Classes to API Contracts

FastAPI uses Pydantic because web applications live at the boundary between untyped external data and typed Python code. This chapter starts from ordinary Python classes, moves through dataclasses, then shows why Pydantic became the standard way to validate JSON, serialize objects, manage settings, and enforce structured LLM outputs.

Where this fits in the book Chapter 3 showed how FastAPI maps HTTP requests into function arguments. This chapter explains the data model layer behind that mapping. After this, the concurrency, authentication, tenancy, and integration chapters can rely on one consistent idea: external data must be validated at the boundary and converted into explicit Python objects before business logic touches it.

1) Why Pydantic Exists

Most backend bugs at API boundaries are not caused by complicated algorithms. They are caused by ordinary shape mismatches:

  • The client sends "age": "30" but Python code expects age: int.
  • The frontend uses personalEmail but Python code should use personal_email.
  • A required field is missing, but the function assumes it exists.
  • A nested payload has the correct outer shape but invalid inner objects.
  • An LLM returns text that looks like JSON but is not the contract your code promised to handle.

Pydantic was introduced to solve this boundary problem using Python type annotations. It gives you a class that still feels like normal Python, but it adds runtime validation, conversion, error reporting, JSON schema generation, serialization, and integration hooks used by frameworks such as FastAPI.

Core mental model: Pydantic turns unreliable outside data into reliable inside objects. Your API should accept dictionaries/JSON at the boundary, validate them once, then pass typed objects through the rest of the system.

2) The Data Problem Before Pydantic

JSON has fewer types than Python

JSON is the common language of HTTP APIs, browser clients, queues, config files, and many LLM tool calls. It has only a small set of data types:

JSON typeExamplePython usually wants
string"Alice"str, sometimes datetime, EmailStr, UUID, Enum
number30, 19.99int, float, Decimal
booleantrue, falsebool
nullnullNone, only if the field is nullable
array["Python", "FastAPI"]list[str], tuple, set
object{"city": "Sydney"}nested model, typed dict, dictionary

That mismatch explains why validation belongs at the API boundary. JSON does not know what a Python datetime, UUID, enum, or domain object is. Your code has to decide what to accept, what to convert, what to reject, and what error to return.

Plain Python classes hold data but do not validate input

class User:
    def __init__(self, id: int, name: str, active: bool = True):
        self.id = id
        self.name = name
        self.active = active

user = User(id="123", name=42, active="yes")
print(user.id, user.name, user.active)
# "123" 42 "yes"   # type hints did not validate anything at runtime

Python type hints help editors, static type checkers, and humans. They do not automatically validate runtime values. Without extra code, the constructor accepts whatever the caller passes.

Class variables are not instance fields

A common beginner mistake is trying to define a data shape with class attributes:

class BadUser:
    id: int
    name: str
    roles: list[str] = []

a = BadUser()
b = BadUser()

a.roles.append("admin")
print(b.roles)
# ["admin"]  # the list is shared because it is a class-level object

An instance variable belongs to one object. A class variable belongs to the class and is shared unless an instance shadows it. For data containers, you usually want instance fields.

3) Dataclasses: The First Improvement

Python dataclasses reduce boilerplate. They generate __init__, __repr__, equality methods, and field handling from annotations.

from dataclasses import dataclass, field

@dataclass
class User:
    id: int
    name: str
    roles: list[str] = field(default_factory=list)

user = User(id="123", name=42)
print(user)
# User(id='123', name=42, roles=[])

Dataclasses solve the constructor and mutable-default problems, but they still do not validate types by default. id="123" and name=42 are accepted because dataclasses are data containers, not runtime validators.

Industry distinction: use a dataclass when data is already trusted and you want a lightweight internal container. Use Pydantic when data crosses a boundary: HTTP request, queue event, file, environment variable, third-party API response, database document, or LLM output.

4) Pydantic Models: Dataclass Ergonomics + Runtime Validation

A Pydantic model is a class that inherits from BaseModel. It uses annotations like a dataclass, generates a constructor like a dataclass, but also validates and converts data at runtime.

from pydantic import BaseModel

class User(BaseModel):
    id: int
    name: str
    active: bool = True
    roles: list[str] = []

user = User(id="123", name="Alice", active="true")
print(user)
# id=123 name='Alice' active=True roles=[]

Several important things happened:

  • "123" became 123 because the field is int.
  • "true" became True because the field is bool.
  • The model has a clear set of fields: id, name, active, roles.
  • Pydantic handles mutable defaults safely by copying them per instance.

That default conversion behavior is useful for APIs because JSON, query parameters, forms, environment variables, and CSV values often arrive as strings. If you want no conversion, use strict mode, covered below.

5) Validation Errors Are Part of the Contract

Good validation is not just "reject bad data." It should tell the caller exactly what failed, where it failed, and why.

from pydantic import BaseModel, ValidationError

class User(BaseModel):
    id: int
    name: str

try:
    User(id="not-a-number", name=123)
except ValidationError as exc:
    print(exc.errors())

The error payload contains locations such as ("id",), an error message, an error type, and the bad input. FastAPI turns this into an HTTP 422 response for request-body validation failures.

[
  {
    "type": "int_parsing",
    "loc": ["id"],
    "msg": "Input should be a valid integer, unable to parse string as an integer",
    "input": "not-a-number"
  }
]

6) Required, Optional, Nullable, and Defaults

This is one of the most important Pydantic topics because Python developers often mix up "optional" and "nullable."

from pydantic import BaseModel

class Customer(BaseModel):
    id: int                    # required, cannot be missing
    name: str                  # required, cannot be missing
    age: int | None            # required, but may be null/None
    nickname: str | None = None # optional and nullable
    country: str = "AU"         # optional, default used if missing
DeclarationMay be missing?May be None?Meaning
name: strNoNoRequired non-null string
age: int | NoneNoYesRequired key, value may be null
nickname: str | None = NoneYesYesOptional key, null/default allowed
country: str = "AU"YesNoOptional key with non-null default
API design rule: do not write age: int = None. If None is valid, say so in the type: age: int | None = None. If None is not valid, use a real default or make the field required.

7) Field Constraints with Field

Type annotations describe the broad shape. Field describes constraints, aliases, examples, and metadata for JSON schema/OpenAPI.

from decimal import Decimal
from pydantic import BaseModel, Field

class Product(BaseModel):
    sku: str = Field(min_length=3, max_length=32, pattern=r"^[A-Z0-9-]+$")
    name: str = Field(min_length=1)
    price: Decimal = Field(gt=0, max_digits=10, decimal_places=2)
    tags: list[str] = Field(default_factory=list, max_length=10)

product = Product(sku="SKU-100", name="Keyboard", price="129.95")
print(product.price)
# Decimal('129.95')

Use constraints for domain rules that can be stated locally on one field: minimum length, maximum length, numeric range, regex/pattern, decimal precision, default factories, examples, and descriptions.

8) Coercion Mode vs Strict Mode

Pydantic's default behavior is intentionally practical: it tries to convert compatible values. Your notes call this "lax" mode. That is the normal API mode because external data often arrives as strings.

from pydantic import BaseModel

class SearchParams(BaseModel):
    top_k: int
    include_archived: bool = False

params = SearchParams(top_k="10", include_archived="false")
print(params)
# top_k=10 include_archived=False

For money movement, security policy, compliance checks, or any place where conversion could hide a bug, use strict validation.

from pydantic import BaseModel, ConfigDict, ValidationError

class StrictSearchParams(BaseModel):
    model_config = ConfigDict(strict=True)

    top_k: int
    include_archived: bool = False

try:
    StrictSearchParams(top_k="10")
except ValidationError as exc:
    print(exc.errors()[0]["msg"])
    # Input should be a valid integer

You can also make selected fields strict instead of making the whole model strict.

from pydantic import BaseModel, Field

class Payment(BaseModel):
    amount_cents: int = Field(strict=True, gt=0)
    currency: str

9) Aliases: JSON camelCase and Python snake_case

External APIs often use camelCase. Python code should use snake_case. Pydantic aliases let you support both without leaking JSON naming conventions into internal code.

from pydantic import BaseModel, ConfigDict, Field

class UserProfile(BaseModel):
    model_config = ConfigDict(validate_by_name=True, validate_by_alias=True)

    user_id: str = Field(alias="userId")
    personal_email: str = Field(alias="personalEmail")

payload = {"userId": "u_123", "personalEmail": "alice@example.com"}
profile = UserProfile.model_validate(payload)

print(profile.personal_email)
# alice@example.com

print(profile.model_dump(by_alias=True))
# {'userId': 'u_123', 'personalEmail': 'alice@example.com'}

In real systems, aliases are a contract boundary. Internal code should speak Python. External JSON should speak the API contract.

Alias generator for large APIs

from pydantic import BaseModel, ConfigDict

def to_camel(name: str) -> str:
    first, *rest = name.split("_")
    return first + "".join(word.capitalize() for word in rest)

class ApiModel(BaseModel):
    model_config = ConfigDict(
        alias_generator=to_camel,
        validate_by_name=True,
        validate_by_alias=True,
    )

class Account(ApiModel):
    account_id: str
    billing_email: str

account = Account.model_validate({"accountId": "a_1", "billingEmail": "ops@example.com"})
print(account.model_dump(by_alias=True))

10) Nested Models

Real JSON payloads are rarely flat. A model can contain other models, lists of models, dictionaries of models, enums, and constrained fields.

from enum import Enum
from pydantic import BaseModel, Field

class Plan(str, Enum):
    FREE = "free"
    TEAM = "team"
    ENTERPRISE = "enterprise"

class Address(BaseModel):
    line1: str
    city: str
    country: str = "AU"

class Organisation(BaseModel):
    org_id: str
    plan: Plan
    billing_address: Address
    admins: list[str] = Field(default_factory=list)

org = Organisation.model_validate({
    "org_id": "org_123",
    "plan": "enterprise",
    "billing_address": {"line1": "1 Market St", "city": "Sydney"},
    "admins": ["alice@example.com", "bob@example.com"],
})

print(org.billing_address.city)
# Sydney

If a nested field fails, the error path points to the nested location, such as billing_address.city or admins.1. That makes validation errors actionable for both frontend engineers and API clients.

11) Validators: Field-Level and Model-Level Rules

Use built-in types and Field constraints first. Add validators when the rule is domain-specific or depends on multiple fields.

Field validators

from pydantic import BaseModel, field_validator

class Signup(BaseModel):
    email: str
    display_name: str

    @field_validator("email")
    @classmethod
    def email_must_look_valid(cls, value: str) -> str:
        value = value.strip().lower()
        if "@" not in value:
            raise ValueError("email must contain @")
        return value

signup = Signup(email=" ALICE@EXAMPLE.COM ", display_name="Alice")
print(signup.email)
# alice@example.com

Model validators

Use a model validator when the rule needs more than one field.

from pydantic import BaseModel, model_validator

class DateRange(BaseModel):
    start_day: int
    end_day: int

    @model_validator(mode="after")
    def end_must_not_precede_start(self):
        if self.end_day < self.start_day:
            raise ValueError("end_day must be greater than or equal to start_day")
        return self
Pydantic v1 to v2 naming: older notes and examples often use @validator, @root_validator, .dict(), .json(), and parse_raw(). In modern Pydantic v2, prefer @field_validator, @model_validator, model_dump(), model_dump_json(), model_validate(), and model_validate_json().

12) Model Configuration

Configuration controls model-wide behavior. In Pydantic v2, use model_config = ConfigDict(...).

from pydantic import BaseModel, ConfigDict, Field

class SecurePayload(BaseModel):
    model_config = ConfigDict(
        extra="forbid",
        str_strip_whitespace=True,
        validate_assignment=True,
        frozen=False,
    )

    tenant_id: str = Field(min_length=1)
    user_id: str = Field(min_length=1)

payload = SecurePayload(tenant_id=" acme ", user_id=" u_123 ")
print(payload.tenant_id)
# acme
ConfigWhy it matters
extra="forbid"Reject unknown fields instead of silently ignoring them. Good for security-sensitive APIs.
extra="ignore"Ignore unknown fields. Useful when consuming third-party APIs that add fields over time.
extra="allow"Keep unknown fields. Useful for flexible metadata, but avoid it for core contracts.
strict=TrueDisable broad coercion. Good when conversion could hide defects.
validate_assignment=TrueValidate when fields are changed after model creation.
frozen=TrueMake models immutable after creation. Useful for config and value objects.
str_strip_whitespace=TrueNormalize incoming strings at the boundary.

13) Deserialization and Serialization

Deserialization means turning external data into Python objects. Serialization means turning Python objects back into dictionaries or JSON.

Validate a Python dictionary

from pydantic import BaseModel

class User(BaseModel):
    id: int
    name: str

data = {"id": "123", "name": "Alice"}
user = User.model_validate(data)

Validate JSON text

json_data = '{"id": "123", "name": "Alice"}'
user = User.model_validate_json(json_data)

Serialize to a Python dictionary or JSON string

print(user.model_dump())
# {'id': 123, 'name': 'Alice'}

print(user.model_dump_json())
# {"id":123,"name":"Alice"}

model_dump(mode="python") may include Python-native objects such as datetime and Decimal. Use mode="json" or model_dump_json() when preparing data for JSON responses, logs, queues, or cache entries.

14) JSON Schema and OpenAPI

Pydantic can produce JSON Schema from your model. FastAPI uses that schema to generate OpenAPI documentation and request/response examples.

from pydantic import BaseModel, Field

class CreateTicket(BaseModel):
    title: str = Field(min_length=3, examples=["Reset my VPN access"])
    priority: int = Field(ge=1, le=5, description="1 is highest priority")

print(CreateTicket.model_json_schema())

This is why FastAPI can show a form-like documentation page for your request body: the Pydantic model is both runtime validation and machine-readable API contract.

15) TypeAdapter: Validate a Type Without Creating a Model

Sometimes you do not need a named model. You just need to validate list[int], dict[str, Decimal], or list[SomeModel]. Use TypeAdapter.

from pydantic import BaseModel, TypeAdapter

class Item(BaseModel):
    id: int
    name: str

adapter = TypeAdapter(list[Item])

items = adapter.validate_python([
    {"id": "1", "name": "Keyboard"},
    {"id": 2, "name": "Mouse"},
])

print(items)
# [Item(id=1, name='Keyboard'), Item(id=2, name='Mouse')]

TypeAdapter is useful for validating lists from files, queue batches, third-party responses, and LLM output arrays without inventing a wrapper model.

16) Settings Management with pydantic-settings

In Pydantic v2, settings moved to the separate pydantic-settings package. Use it to read environment variables, .env files, and secrets into a typed object.

pip install pydantic-settings
from pydantic import Field
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=".env",
        env_prefix="APP_",
        extra="ignore",
    )

    app_name: str = "tutorial-api"
    debug: bool = False
    database_url: str
    jwt_audience: str = Field(default="tutorial-api")

settings = Settings()
print(settings.database_url)

With env_prefix="APP_", Pydantic reads variables such as APP_DATABASE_URL, APP_DEBUG, and APP_JWT_AUDIENCE. This follows twelve-factor application practice: configuration belongs in the environment, not hard-coded in source files.

Do not log settings objects blindly. If a settings model contains secrets, mark them as SecretStr or keep logging focused on non-sensitive fields.

17) FastAPI Integration: Request and Response Contracts

FastAPI uses Pydantic models in two directions:

  • Request models validate data coming into your endpoint.
  • Response models define and filter data leaving your endpoint.
from decimal import Decimal
from fastapi import FastAPI
from pydantic import BaseModel, ConfigDict, Field

app = FastAPI()

class CreateProductRequest(BaseModel):
    sku: str = Field(min_length=3)
    name: str
    price: Decimal = Field(gt=0)

class ProductResponse(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    sku: str
    name: str
    price: Decimal

@app.post("/products", response_model=ProductResponse)
def create_product(req: CreateProductRequest):
    # req is already validated here.
    product = save_product(req)
    # response_model controls what leaves the API.
    return product

response_model is not just documentation. It prevents accidental data leaks by shaping the returned object. If your internal object has cost_price, supplier_token, or other private fields, they are not included unless the response model declares them.

18) External API and Queue Message Validation

Pydantic is not only for incoming HTTP requests. Use it whenever your system receives data it did not create.

from datetime import datetime
from pydantic import BaseModel, ConfigDict, Field

class PaymentEvent(BaseModel):
    model_config = ConfigDict(extra="forbid")

    event_id: str
    tenant_id: str
    amount_cents: int = Field(gt=0)
    occurred_at: datetime

def handle_message(raw_json: str) -> None:
    event = PaymentEvent.model_validate_json(raw_json)
    process_payment_event(event)

This pattern is useful for Pub/Sub messages, Kafka events, webhook payloads, cron job inputs, uploaded JSON files, and service-to-service responses.

19) LLM Output Validation

Pydantic gained renewed importance with LLM systems because model outputs are probabilistic. Even when you ask for JSON, you still need to validate the shape before using it in application code.

from pydantic import BaseModel, Field, ValidationError

class SupportTriage(BaseModel):
    category: str
    confidence: float = Field(ge=0, le=1)
    summary: str = Field(min_length=1)
    next_action: str

raw_llm_json = '''
{
  "category": "account_access",
  "confidence": 0.87,
  "summary": "User cannot access VPN after password reset.",
  "next_action": "route_to_identity_team"
}
'''

try:
    triage = SupportTriage.model_validate_json(raw_llm_json)
except ValidationError as exc:
    # Retry, ask the model to repair, or send to human review.
    raise

print(triage.next_action)

In production LLM workflows, Pydantic is a guardrail, not a guarantee. It checks structure and local constraints. You still need business-rule validation, authorization checks, rate limits, audit logs, and human review for high-risk actions.

20) Pydantic Dataclasses

Pydantic can also validate dataclasses. This is useful when you prefer dataclass style but want Pydantic validation on construction.

from pydantic import ConfigDict, ValidationError
from pydantic.dataclasses import dataclass

@dataclass(config=ConfigDict(validate_assignment=True))
class Job:
    job_id: int
    queue: str

job = Job(job_id="42", queue="email")
print(job.job_id)
# 42

try:
    job.job_id = "not-an-int"
except ValidationError as exc:
    print(exc.errors()[0]["msg"])

Use this when integrating with dataclass-heavy code. For API contracts, settings, and OpenAPI integration, BaseModel is still the usual choice.

21) Practical Design Patterns

Separate input, domain, and output models

Do not use one giant model for every layer. API inputs, internal domain objects, database records, and API responses have different responsibilities.

from pydantic import BaseModel, Field

class CreateUserRequest(BaseModel):
    email: str
    display_name: str = Field(min_length=1)

class UserRecord(BaseModel):
    user_id: str
    tenant_id: str
    email: str
    display_name: str
    password_hash: str

class UserResponse(BaseModel):
    user_id: str
    email: str
    display_name: str

This avoids two common mistakes: accepting fields the client should not control, and returning fields the client should never see.

Validate at boundaries, not everywhere

Validate when data enters the system. Inside the system, pass typed objects. Repeated validation in every function makes code noisy and hides the real trust boundary.

Prefer explicit models over unstructured dictionaries

# Weak boundary
def create_ticket(payload: dict) -> dict:
    return {"title": payload["title"], "priority": int(payload["priority"])}

# Strong boundary
class CreateTicket(BaseModel):
    title: str
    priority: int = Field(ge=1, le=5)

def create_ticket(payload: CreateTicket) -> dict:
    return {"title": payload.title, "priority": payload.priority}

23) End-to-End Example: API Payload, Alias, Validation, and Response

This example combines the pieces: JSON aliases, strict API shape, field constraints, model-level validation, and response shaping.

from decimal import Decimal
from fastapi import FastAPI
from pydantic import BaseModel, ConfigDict, Field, model_validator

app = FastAPI()

class PurchaseRequest(BaseModel):
    model_config = ConfigDict(
        extra="forbid",
        validate_by_name=True,
        validate_by_alias=True,
    )

    tenant_id: str = Field(alias="tenantId", min_length=1)
    user_id: str = Field(alias="userId", min_length=1)
    sku: str = Field(min_length=3)
    quantity: int = Field(alias="qty", gt=0, le=100)
    unit_price: Decimal = Field(alias="unitPrice", gt=0)

    @model_validator(mode="after")
    def total_must_be_reasonable(self):
        total = self.quantity * self.unit_price
        if total > Decimal("50000"):
            raise ValueError("purchase requires manual approval")
        return self

class PurchaseResponse(BaseModel):
    purchase_id: str = Field(alias="purchaseId")
    status: str
    total: Decimal

@app.post("/purchase", response_model=PurchaseResponse)
def create_purchase(req: PurchaseRequest):
    total = req.quantity * req.unit_price
    return {
        "purchaseId": "po_123",
        "status": "accepted",
        "total": total,
    }

The endpoint can now rely on req.tenant_id, req.user_id, req.quantity, and req.unit_price having already passed the declared contract. The external JSON can still use tenantId, userId, qty, and unitPrice.

25) What to Remember

  • Plain classes and dataclasses organize data but do not validate runtime input by default.
  • Pydantic models are boundary contracts: validate, convert, serialize, and document data.
  • Default Pydantic mode is practical and coercive; use strict mode where conversion is risky.
  • Optional means "may be missing" only when there is a default. Nullable means the type includes None.
  • Aliases let external JSON use camelCase while Python remains snake_case.
  • Use v2 APIs in new code: model_validate, model_dump, field_validator, model_validator, and ConfigDict.
  • Pydantic is also useful beyond FastAPI: settings, webhooks, queues, external APIs, files, and LLM output validation.
