FastAPI Concurrency & User Isolation

How FastAPI handles 1000s of users + how it keeps requests separate

1. What happens when 1000 users hit your API

When thousands of users call your FastAPI endpoint simultaneously, Uvicorn (the ASGI server) accepts the connections and creates a separate “in-flight request task” for each one.

For every single request, FastAPI:

  • Matches the method + path (e.g., POST /query) to the correct function.
  • Parses headers, query parameters, and the body specific to that request.
  • Calls your function with fresh arguments.
Key Concept: Each user request triggers an independent function call.

2. Why async lets one server handle lots of traffic

Most API operations are I/O-bound: waiting for database queries, Elasticsearch results, or external APIs (like Azure OpenAI).

If your endpoint is defined with async def and uses await for I/O, the event loop can pause a waiting request (Request A) and process others (Request B, C) in the meantime. This allows a single process to handle thousands of concurrent connections efficiently.

@app.post("/query")
async def query(...):
    chunks = await es_search(...)       # waiting -> event loop runs other requests
    answer = await call_openai(...)     # waiting -> event loop runs other requests
    return {"answer": answer}
Important: If you use blocking code (like time.sleep() or a synchronous HTTP client) inside async def, you freeze the event loop and kill concurrency.

3. Concurrency vs. Parallelism

It is important to distinguish between these two scaling mechanisms:

A) Async Concurrency

(Recommended for I/O)

  • One process + one event loop.
  • Thousands of “tasks” can be in flight.
  • Best when you await I/O operations.

B) Parallelism (Workers)

(Multi-process)

  • Run multiple Uvicorn worker processes.
  • Each process has its own event loop.
  • Utilizes multiple CPU cores.

4. How FastAPI keeps users separate (Per-Request Identity)

FastAPI does not “assign a user to a thread.” Instead, every HTTP request is isolated. User identity is derived from the request itself (usually via a JWT in the Authorization header), and your code uses this identity to scope data access.

Concrete Example

Two users hit the same endpoint at the same time, but get different results.

from fastapi import FastAPI, Depends, Header, HTTPException
from pydantic import BaseModel

app = FastAPI()

class Identity(BaseModel):
    tenant_id: str
    user_id: str

# Dependency to extract identity from header
def get_identity(authorization: str | None = Header(None)) -> Identity:
    if not authorization or not authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="Missing bearer token")

    token = authorization.removeprefix("Bearer ").strip()

    # demo-only logic
    if token == "tokenA":
        return Identity(tenant_id="tenant-1", user_id="user-A")
    if token == "tokenB":
        return Identity(tenant_id="tenant-2", user_id="user-B")
    
    raise HTTPException(status_code=401, detail="Invalid token")

@app.post("/query")
async def query(req: QueryReq, ident: Identity = Depends(get_identity)):
    # ident differs per request based on the token
    return {
        "tenant_id": ident.tenant_id,
        "user_id": ident.user_id,
        "question": req.question,
    }

User A Request

Authorization: Bearer tokenA

FastAPI calls:

ident=Identity(tenant_id="tenant-1", ...)

User B Request

Authorization: Bearer tokenB

FastAPI calls:

ident=Identity(tenant_id="tenant-2", ...)

Same code, same route — but different identity → tenants stay isolated.

5. Critical Rule: No Global State

Never use global variables for per-user state. Globals are shared across all requests in a process.

CURRENT_USER = None # ❌ UNSAFE: shared across users

Instead, keep per-user/per-request data in:

  • Function local variables (ident, req)
  • Dependencies (Depends(get_identity))
  • request.state (middleware context)
