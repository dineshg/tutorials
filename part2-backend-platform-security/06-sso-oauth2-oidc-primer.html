SSO, OAuth2, and OIDC Explained

  • SSO is the experience: one login lets you access multiple apps.
  • OAuth2 is the permission system: it gives a client an access token that says “this app is allowed to call this API with these scopes.” It doesn’t, by itself, standardize “who is the user.”
  • OIDC is SSO-style login for modern apps, built on OAuth2: it keeps OAuth2’s token flow, but adds a standard “identity layer” so the client can reliably learn who logged in (via an ID token + standard claims like sub, email).

Analogy

Imagine a company building with a receptionist.

  • OAuth2 is like getting a visitor pass that grants access to certain rooms (“you may enter the API room”). It’s about authorization (access rights).
  • OIDC is that same visitor pass plus a verified badge with your name/photo issued by HR. Now different departments can trust the badge for “who you are” and you get SSO-like login across apps.
