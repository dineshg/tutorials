1) The admin mental model (the “plumbing”) most important

Think of GitHub governance like a gated building: workflows produce evidence (passes/fails) and rulesets are the bouncer. The bouncer does not do the inspection themselves; they only check the evidence before letting someone in.

Four building blocks you should memorize:

  • Workflow: the script that defines what to do (run tests, scan code, etc.) and when to do it.
  • Runner: the machine that executes workflow jobs.
  • Status check: the recorded outcome from the workflow job (pass/fail) attached to a commit/PR.
  • Ruleset: the policy gate that blocks merge/push if conditions aren’t met.

Key clarification: If you type aqua-trivy-scan into a ruleset, you are not “turning on Aqua”.

You are saying: “I expect a check with this name to exist and be passing.” Someone/something must create that check—usually a workflow file in the repository or a GitHub App integration.

Admin perspective: Your job is to ensure (1) the right workflows exist and run on the right events, and (2) the ruleset requires the correct checks and restricts bypass.

2) Git vs GitHub governance (what each can/can’t enforce)

This matters because people often say “Git should block X”. In reality: Git is the version control system, while GitHub is the policy and automation platform.

Git (alone)

Git controls commits/branches and lets you push/fetch. It can run optional local hooks, but those are not reliable for enforcement because users can bypass them. Git itself does not understand “tests passed” or “scan passed” across a company.

  • Good for: history, branching, collaboration primitives.
  • Not good for: enterprise policy enforcement of scans/tests.

GitHub (platform)

GitHub adds PRs, reviews, checks, and enforcement. This is where administrators set rules and apply them across repositories.

  • PR process: reviews, approvals, merge strategies.
  • Automation: GitHub Actions workflows.
  • Governance: rulesets, required checks/workflows, branch restrictions.

3) Workflows and actions (how GitHub knows what to run)

GitHub Actions is configured by YAML files stored in the repository at .github/workflows/*.yml. GitHub watches for specific events (like PR updates or pushes). When an event occurs, GitHub checks: “Do I have any workflows whose on: section matches this event?” If yes, those workflows are scheduled to run.

Workflow anatomy

TermPlain EnglishWhy admins care
Workflow A set of jobs triggered by events. Defines what will run (or not run) for a PR/push.
Job Runs on one runner; contains steps. Each job produces a check result; required checks often match job names.
Step One action call (uses) or one shell command (run). Steps are where tools run; failures here fail the job/check.
Action Reusable unit referenced via uses: owner/repo@ref. This is how you “bring in” scanners like Trivy/Aqua into workflows.
Runner The compute that executes the job. Admins control runner availability, security, and who can use them.

“How is that container/tool found?” (your earlier question)

In a workflow, the line uses: aquasecurity/trivy-action@master is an explicit pointer to an Action repository. GitHub fetches that action and runs it. If the action is Docker-based, its own action.yml defines the container image. If it is composite-based, it runs commands directly on the runner (often installing Trivy and executing it).

As an admin, you care about pinning versions (avoid floating refs like @master for stricter supply-chain control), and restricting which actions are allowed via org/enterprise policies.

4) Status checks (what the ruleset actually reads)

A status check is a named pass/fail result attached to a commit and shown on the PR. When a job finishes, GitHub records the result. Rulesets don’t inspect your code; they simply check whether the required check names are present and successful.

How check names are typically formed

The UI usually shows checks as <workflow name> / <job name>. Example: workflow name aqua-trivy-scan with job name scan → check appears as aqua-trivy-scan / scan.

Why admins often get “stuck”

  • They require aqua-trivy-scan but the real check is aqua-trivy-scan / scan.
  • The workflow doesn’t run on pull_request, so PRs never get that check.
  • Actions are disabled by org policy, so checks never run.
  • The check exists, but the job is “skipped” due to workflow conditions.

Admin practice: always read the PR “Checks” list and copy the exact check name to your ruleset.

5) Rulesets (what they enforce and why they don’t “run tools”)

A ruleset is a server-side enforcement policy attached to branch targets (e.g., main or release/*). It can block pushes, block merges, require approvals, require checks, and more. But it does not run scanners or execute workflows; it only evaluates the recorded states of checks and PR requirements.

Rulesets can enforce

  • Require PRs (no direct merge)
  • Require approvals / code owners
  • Require status checks to pass
  • Restrict direct pushes / force pushes
  • Restrict branch creation/updates (naming patterns)

Rulesets cannot do by themselves

  • Install tools like Aqua/Trivy
  • Decide what to run without workflows or required-workflow feature
  • Produce a check result without something executing

What this means for your “Aqua enforcement” requirement

You must have a workflow (or required workflow) that performs the Aqua scan and fails if policy is violated. Then the ruleset requires the check to be green before merge.

6) Required checks vs Required workflows (enterprise scale)

There are two ways to enforce scanning outcomes with rulesets. They look similar but behave differently:

Required checks (most common)

You list a check name (e.g., aqua-trivy-scan / scan) and GitHub blocks merges until that check is successful. This relies on each repo (or a shared reusable workflow) producing the same check name.

  • Pros: simple, widely used, clear in PR UI.
  • Cons: requires standardization; check name drift causes broken merges.

Required workflows (better for enterprise governance)

Instead of requiring a “name that must appear”, you require a specific workflow to run. This is useful when you want centralized enforcement and fewer per-repo variations.

  • Pros: less drift; admins can enforce a standard baseline.
  • Cons: still requires Actions to be enabled and configured for the targeted repos.

Practical admin advice: use required workflows for the baseline (security + compliance), and required checks for additional repo-specific checks.

7) Aqua/Trivy example (scan workflow + merge enforcement)

Below is a concrete “end-to-end” example: the workflow produces a scan check, and the ruleset blocks merges if it fails. The workflow must fail when policy is violated; that failure becomes the red status check.

7.1 Workflow: run Trivy (Aqua) and fail on HIGH/CRITICAL

Create: .github/workflows/aqua-trivy-scan.yml

name: aqua-trivy-scan

on:
  pull_request:
  push:
    branches: [ "main" ]

permissions:
  contents: read

jobs:
  scan:
    name: scan
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Trivy filesystem scan (fail on HIGH/CRITICAL)
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: fs
          scan-ref: .
          severity: HIGH,CRITICAL
          ignore-unfixed: true
          format: table
          exit-code: 1

What check appears in the PR?

Typically: aqua-trivy-scan / scan. That is the string you require in the ruleset.

7.2 Ruleset: block merge to main unless the scan check is green

  • Create/edit a ruleset targeting branch: main.
  • Enable “Require a pull request”.
  • Enable “Require status checks to pass”.
  • Select aqua-trivy-scan / scan as a required check.
  • Disable direct pushes to main (recommended).

Why must the workflow exist? Because the ruleset only checks for the recorded status check. If there is no workflow producing aqua-trivy-scan / scan, the ruleset has no green check to verify—merge stays blocked.

8) Push controls (naming, direct pushes, secrets)

“Push-time enforcement” means blocking a push before it lands in the repository history. GitHub can do this strongly for a few categories; for heavier scans, the common pattern is merge gating.

Strong push-time controls

  • Block direct pushes to protected branches (via ruleset).
  • Branch naming restrictions (allow only certain patterns).
  • Secret push protection (block pushes containing detected secrets).

Why not push-gate everything?

Many scans take time (minutes). Push rejection must be near-instant and synchronous. Therefore, organizations usually enforce “heavy scanning” at merge time (required checks) and keep push-time controls for fast, deterministic rules (naming, protected branches, secret push protection).

Branch naming enforcement example

As an admin, you typically allow only patterns like:

feature/*
bugfix/*
hotfix/*
chore/*
release/*

Note: In GitHub, rulesets often work by targeting patterns and restricting create/update. You configure allowed patterns by restricting broadly and explicitly permitting specific patterns (depending on your chosen ruleset approach).

9) Runners (hosted vs self-hosted) and runner security

A runner is the execution environment for jobs. If someone can run code on a runner, they effectively execute arbitrary commands in that environment. That’s why runner governance (which repos can use which runners) is a core admin responsibility.

GitHub-hosted runners

GitHub manages them; they are generally ephemeral and reset between jobs. This reduces persistence risk. Great default for most repos and for consistent environments.

  • Select with: runs-on: ubuntu-latest
  • Minimal operational work for admins

Self-hosted runners

You manage the machines (VMs/on-prem/Kubernetes). Choose these when you need strict network access, specialized tools, or compliance controls. However, you must treat them as sensitive and isolate them from untrusted workloads.

  • Select with labels: runs-on: [self-hosted, linux, x64, secure]
  • Admins should use runner groups to restrict usage

Self-hosted runner security (short, practical)

  • Use runner groups and allow only selected repos.
  • Keep a “secure” runner group for sensitive repos; do not allow broad access.
  • Prefer ephemeral runners (one job → destroy) for high-security posture.
  • Do not allow fork PRs or untrusted code to run on sensitive runners.
  • Minimize secrets exposure; assume workflow-exposed secrets can be exfiltrated.

10) Enterprise-wide patterns (recommended setup)

Enterprise governance works best when you standardize: workflow names, job names, required checks, and who can bypass rulesets. Your goal is to make compliance “boring” and predictable.

Recommended enterprise baseline

  • Enterprise ruleset applied to protected branches (e.g., main, release/*).
  • Require PR + require approvals (optional based on policy).
  • Require checks: tests + scan checks (aqua-trivy-scan / scan).
  • Block direct pushes to protected branches.
  • Branch naming restriction (limit branch creation/update patterns).
  • Secret push protection enabled org/enterprise-wide.

Scaling tip: required workflows

If you can use required workflows, centralize the baseline scanning workflow so every repo gets the same scanning behavior and check naming. This reduces drift and makes rulesets more stable.

11) Troubleshooting as an admin

Problem: “Ruleset requires Aqua scan but no check appears”

  • Does the repo contain .github/workflows/aqua-trivy-scan.yml?
  • Does it trigger on pull_request (not only on push)?
  • Are GitHub Actions enabled at org/enterprise level for that repo?
  • Does the workflow have conditions that skip it for some branches?
  • Is the job failing before it reports a check (syntax errors, permission errors, runner issues)?

Problem: “I required aqua-trivy-scan but merges still blocked”

  • Confirm the exact check name shown on PR (often aqua-trivy-scan / scan).
  • Require the exact name in the ruleset.
  • Ensure the check is not “skipped” or “neutral” for PRs.

Problem: “Self-hosted runner never picks up jobs”

  • Is the runner online and registered?
  • Does the workflow use the runner’s labels in runs-on?
  • Is the runner in a runner group that allows this repository?
  • Do org policies restrict runner usage?

Final admin summary:

  • Workflows define what runs (.github/workflows).
  • Runners execute jobs (runs-on).
  • Jobs produce named checks (workflow / job).
  • Rulesets enforce policy by requiring those checks and restricting push/merge behavior.
