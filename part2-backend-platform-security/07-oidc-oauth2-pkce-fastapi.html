OIDC + OAuth2 + PKCE → Bearer Token → FastAPI Identity

This is the same story end-to-end: how a React SPA securely obtains an access_token (PKCE), and how FastAPI validates that token and maps it to tenant_id + user_id per request.

Acronyms (quick reference)

  • IdP: Identity Provider (e.g., Google Identity, Auth0, Okta, Entra ID)
  • OAuth2: OAuth 2.0 (authorization framework; how clients get tokens to call APIs)
  • OIDC: OpenID Connect (authentication layer on top of OAuth2; adds login + identity claims)
  • PKCE: Proof Key for Code Exchange (protects authorization-code flow for public clients like SPAs)
  • JWT: JSON Web Token (self-contained, signed token format commonly used for access tokens)
  • JWKS: JSON Web Key Set (IdP endpoint that publishes public signing keys for JWT verification)
  • SPA: Single Page Application (React running in the browser)

JWT & JWKS (the missing piece)

The confusion usually comes from not separating two steps: (1) token issuance (frontend ↔ IdP) vs (2) token validation (backend verifying the token).

JWT (JSON Web Token)

  • A JWT is a compact token format commonly used for access_token. It is typically signed so APIs can verify it hasn’t been tampered with.
  • It usually contains claims (e.g., sub, iss, aud, exp) inside the token payload. 
JWT structure (conceptual):
header.payload.signature

- header: metadata (often includes "kid" = key id)
- payload: claims (sub, iss, aud, exp, ...)
- signature: proves the IdP signed it with its private key

JWKS (JSON Web Key Set)

  • A JWKS is the IdP’s published set of public keys. Your backend uses these public keys to verify the JWT signature.
  • The backend typically fetches JWKS from a well-known IdP URL, caches it, and refreshes periodically (key rotation).
  • The JWT header often includes kid so the backend can pick the correct public key from the JWKS.
Where they come into the picture:
The IdP issues an access_token (often a JWT). Then the backend validates the token signature using the IdP’s JWKS. This is why the backend usually does not need to call the IdP “to ask who the user is” on every request.

Bearer token per request (how FastAPI identifies callers)

Your backend identifies the user (and tenant) because every request carries:

Authorization: Bearer <token>

The key questions are:

  • How does the frontend generate <token> securely?
  • How does the backend trust it?

OAuth2 vs OIDC (in this story)

OAuth2
How the client obtains an access token to call an API.
OIDC
Adds authentication (“login”) and identity info (ID token + claims) on top of OAuth2.

In practice:

  • The React app uses OIDC to sign in with an IdP (e.g., Google Workspace).
  • The React app uses the resulting access token as the Bearer token when calling FastAPI.

What PKCE is (and why you need it)

The problem PKCE solves

A React SPA is a public client: it cannot safely store a permanent client secret in the browser. Without protection, an attacker who steals the authorization code could exchange it for tokens.

PKCE in one sentence

PKCE (“Proof Key for Code Exchange”) makes the authorization code usable only by the same client instance that started the login.

How PKCE works (simple)

  1. React generates a random secret string: code_verifier
  2. React hashes it into code_challenge
  3. React sends code_challenge during the login redirect
  4. Later, when exchanging the authorization code for tokens, React must present the original code_verifier
  5. The IdP checks the verifier matches the earlier challenge
Result: If someone stole the authorization code but doesn’t have the verifier, they can’t get tokens.
Important boundary: PKCE is between the frontend and the IdP. Your FastAPI backend does not perform PKCE.

What you must set up first (one-time)

On the Identity Provider (e.g., Google Identity / Workspace)

  • Create an OAuth client for your SPA
  • You get a client_id
  • Set redirect URI (e.g., https://app.example.com/callback)
  • Choose scopes (typical: openid email profile)

On FastAPI (what the backend “keeps”)

The backend remains stateless and does not store per-user sessions. It keeps:

  • Trusted issuer(s) (who is allowed to mint tokens)
  • Expected audience (token must be intended for your API/client)
  • Access to the IdP’s public signing keys (JWKS), usually cached
That’s enough to validate tokens for any user on any request.

Concrete message sequence (OIDC Authorization Code + PKCE)

1) React redirects user to the IdP /authorize (includes PKCE challenge)

GET https://accounts.google.com/o/oauth2/v2/auth?
  response_type=code&
  client_id=YOUR_CLIENT_ID&
  redirect_uri=https%3A%2F%2Fapp.example.com%2Fcallback&
  scope=openid%20email%20profile&
  state=RANDOM_STATE&
  code_challenge=PKCE_CHALLENGE&
  code_challenge_method=S256
  • response_type=code → request an authorization code
  • scope=openid ... → OIDC login + identity
  • code_challenge → PKCE protection

2) IdP redirects back with the authorization code

GET https://app.example.com/callback?code=AUTH_CODE&state=RANDOM_STATE

3) React exchanges code for tokens at /token (includes PKCE verifier)

POST https://oauth2.googleapis.com/token HTTP/1.1
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=AUTH_CODE&
redirect_uri=https%3A%2F%2Fapp.example.com%2Fcallback&
client_id=YOUR_CLIENT_ID&
code_verifier=PKCE_VERIFIER

4) IdP returns tokens

{
  "token_type": "Bearer",
  "access_token": "....",
  "id_token": "....",
  "expires_in": 3600,
  "scope": "openid email profile"
}

Now the frontend can call your API.

The API call to FastAPI (per user, per request)

POST /query HTTP/1.1
Host: api.example.com
Authorization: Bearer <access_token_for_user_742>
Content-Type: application/json

{ "question": "..." }

How the backend knows the token is “correct”

FastAPI doesn’t “trust the browser”. It trusts the IdP’s signature + standard token checks.

Does the backend call the IdP on every request?
Usually no. With JWT access tokens, the backend verifies the signature locally using the IdP’s public keys (from JWKS), plus it checks standard claims (iss, aud, exp). The backend may fetch JWKS occasionally (and cache it), not per request.

What “signature verification using JWKS” means

  1. The IdP signs the JWT with its private key when issuing the token.
  2. The backend reads the token header (often containing kid).
  3. The backend selects the matching public key from the cached JWKS.
  4. The backend verifies the JWT signature. If it matches, the token was minted by the IdP and wasn’t altered.

How the backend gets JWKS, where it caches it, and when it refreshes

Where does JWKS come from? In OIDC, the IdP exposes a discovery document at /.well-known/openid-configuration. That JSON includes a jwks_uri URL.

Where is it cached? Most APIs cache JWKS in memory inside each running process (each Uvicorn worker / each GKE pod). That means:

  • Pod A caches keys in Pod A memory (Pod B has its own cache).
  • On restart/rollout, the cache is empty and re-fetched on first request (or warm-up).

When does it refresh? Common triggers:

  • TTL expiry (e.g., refresh every 5–60 minutes).
  • Unknown kid (IdP rotated keys; refetch JWKS and retry once).
  • Startup warm-up (optional: fetch once at boot so first request is fast).
Practical note: caching is an optimization; correctness comes from signature verification + claim checks. Key rotation is why “refetch on unknown kid” is a standard pattern.

How the Backend Maps a Dynamic Token to a Stable User ID

The backend does not identify a user by the literal Bearer token string (because tokens change over time). Instead, it uses a 3-step mapping for every single request:

1) Validate the Bearer token

The token is usually a JWT signed by your Identity Provider (e.g., Google). FastAPI verifies the signature using the IdP’s public keys (JWKS) and checks basic claims like:

  • exp (not expired)
  • iss (issued by the expected IdP)
  • aud (intended for your app/API)

If any check fails, the request is rejected (401).

2) Extract a stable user identifier → user_id

After validation, FastAPI reads the token’s claims (the payload). The sub (subject) claim is typically the stable unique identifier for the user at that IdP.

user_id = claims["sub"]

3) Derive or look up the tenant → tenant_id

The backend then determines which tenant that user belongs to, using one of these common strategies:

  • Claim-based: If the token includes a tenant/org claim (e.g. tenant_id, tid, org_id), use it directly.
  • Database mapping (most common): Store a user table keyed by (iss, sub). Look up (claims["iss"], claims["sub"]) to find the tenant_id. If not found, you can optionally provision the user and assign a tenant.
  • Domain mapping (common for Google Workspace): Use the email claim and map the domain to a tenant (e.g., alice@acme.com → tenant “acme”).
Key idea: Tokens rotate, but sub (plus iss) is stable. The mapping is always: token → validated claims → stable user key → tenant.
Anti-pattern: don’t store “current user” in a global variable and don’t treat the token value as a permanent user identifier. Tokens are short-lived and will naturally differ across time and devices.
