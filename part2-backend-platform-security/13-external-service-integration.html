Connecting an External Service

To call an external service like an Azure OpenAI (GPT) endpoint from your backend, you use the same “workload identity” idea, but the target is not GCP—so you need some credential that Azure accepts, and you must store and use it safely.

There are two industrial patterns (pick one):

Option A (simpler): API key stored in GCP Secret Manager

Flow

  1. Your backend (running on GKE) authenticates to GCP Secret Manager using its GCP service account (Workload Identity).
  2. It reads the Azure OpenAI API key from Secret Manager.
  3. It calls Azure OpenAI with:
    • api-key: <key> header (typical for Azure OpenAI)

Why choose this

  • Easiest to implement
  • No Azure identity federation complexity
  • Common and reliable

What you “associate”

1) Store the secret

  • Secret: azure-openai-api-key

2) Grant your backend service account access to read it

  • IAM: give backend-sa@... role roles/secretmanager.secretAccessor on that secret.

3) In code: read secret + call Azure

import os
import httpx
from google.cloud import secretmanager

def read_secret(secret_id: str) -> str:
    client = secretmanager.SecretManagerServiceClient()
    name = f"projects/{os.environ['GCP_PROJECT']}/secrets/{secret_id}/versions/latest"
    resp = client.access_secret_version(request={"name": name})
    return resp.payload.data.decode("utf-8")

async def call_azure_openai(prompt: str) -> dict:
    api_key = read_secret("azure-openai-api-key")

    endpoint = os.environ["AZURE_OPENAI_ENDPOINT"]  # e.g. https://xxx.openai.azure.com
    deployment = os.environ["AZURE_OPENAI_DEPLOYMENT"]
    api_version = os.environ.get("AZURE_OPENAI_API_VERSION", "2024-xx-xx")

    url = f"{endpoint}/openai/deployments/{deployment}/chat/completions?api-version={api_version}"

    async with httpx.AsyncClient(timeout=30) as client:
        r = await client.post(
            url,
            headers={"api-key": api_key, "Content-Type": "application/json"},
            json={
                "messages": [{"role": "user", "content": prompt}],
                "temperature": 0.2,
            },
        )
        r.raise_for_status()
        return r.json()

Option B (more “enterprise”): Azure Entra workload identity federation (no long-lived keys)

Flow

  1. Your backend still runs on GKE with a GCP service account.
  2. Instead of storing an API key, the backend obtains a short-lived Azure token from Entra ID using federation.
  3. It calls Azure OpenAI using that token.

Why choose this

  • No long-lived API keys
  • Short-lived tokens, easier to rotate
  • Stronger enterprise security posture

Tradeoffs

  • More setup (Azure app registration, federated identity config, token exchange logic)
  • Slightly more moving parts than API keys

How this relates to “service accounts”

Important:

  • Your GCP service account gives you permission to read GCP secrets/logs/etc.
  • Azure doesn’t care about GCP service accounts. Azure cares about Azure identity (or API keys).
  • So you either:
    • use GCP SA → read API key → call Azure (Option A), or
    • configure federation so your workload can mint Azure tokens (Option B)

Best practice summary

  • If you want the fastest correct setup: Option A (Secret Manager + API key)
  • If you want “industrial enterprise” with least secrets: Option B (Entra federation)
