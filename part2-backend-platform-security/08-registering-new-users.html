Registering a New User

Assume the organisation's Google Workspace policy already allows this user to authenticate with Google.

New User Sign-In Flow (React + Google OIDC + FastAPI)

Key idea: The login page doesn't "register" the user with Google—Google already knows them. Your app "registers" the user by provisioning a record in your database the first time your backend sees a valid identity.
  1. User clicks "Sign in with Google"

    React starts OIDC Authorization Code + PKCE by redirecting the browser to Google's authorization endpoint.

    Outcome: user authenticates with Google (SSO).

  2. React receives tokens

    React exchanges the authorization code for tokens and gets an access token (this is what becomes the Bearer token for your API). React stores the token (commonly in memory) and uses it for API calls.

  3. React calls a bootstrap endpoint on your API

    Right after login, the frontend calls something like:

    • GET /me (common), or
    • POST /auth/bootstrap

    with:

    Authorization: Bearer <access_token>

    Purpose: let the backend validate the token and decide what to do with this identity.

  4. FastAPI validates token and extracts stable identity

    Backend verifies the token (signature + issuer/audience/expiry), then reads claims like:

    • sub → stable user identifier at this IdP
    • email → commonly used for tenant mapping (e.g., domain) or allowlists
  5. Backend checks if the user already exists in your DB

    Backend looks up by a stable key like (issuer, sub) (best practice), optionally storing email too.

    Case A: user exists

    • Return user profile + tenant info.
    • Frontend proceeds normally.

    Case B: user is new

    • This is where "registration" happens (provisioning).

Example FastAPI Endpoint for User Provisioning

This endpoint is the bootstrap call that validates the Bearer token, looks up the user, creates the DB row if missing, and returns user details.

from fastapi import FastAPI, Depends, HTTPException
from fastapi.security import HTTPAuthorizationCredentials, HTTPBearer
from pydantic import BaseModel
from datetime import datetime

app = FastAPI()
bearer = HTTPBearer(auto_error=True)

class BootstrapResponse(BaseModel):
    user_id: str
    tenant_id: str
    is_new_user: bool

# --- Pretend DB (replace with Postgres later) ---
USERS_BY_ISS_SUB: dict[tuple[str, str], dict] = {}  # key = (issuer, sub)

# --- Token verification (placeholder) ---
def verify_and_decode_access_token(token: str) -> dict:
    """
    Real implementation would:
      - verify JWT signature using Google JWKS
      - check issuer (iss), audience (aud), expiry (exp)
    Here we return mock claims so you see where they come from.
    """
    if token == "tokenA":
        return {
            "iss": "https://accounts.google.com",
            "sub": "user-A-sub",
            "email": "alice@acme.com",
        }
    raise HTTPException(status_code=401, detail="Invalid/expired token")

def map_email_to_tenant_id(email: str) -> str:
    """
    Example tenant mapping policy: email domain -> tenant.
    (Alternative policies: invite-only lookup, explicit claim, etc.)
    """
    domain = email.split("@")[-1].lower()
    if domain == "acme.com":
        return "tenant-acme"
    raise HTTPException(status_code=403, detail="No tenant mapping for this user")

@app.post("/auth/bootstrap", response_model=BootstrapResponse)
def auth_bootstrap(creds: HTTPAuthorizationCredentials = Depends(bearer)):
    # 1) Extract raw token from "Authorization: Bearer <token>"
    token = creds.credentials

    # 2) Validate + decode -> claims (iss/sub/email)
    claims = verify_and_decode_access_token(token)
    issuer = claims["iss"]
    sub = claims["sub"]
    email = claims.get("email", "")

    # 3) Look up user by stable key (issuer, sub)
    key = (issuer, sub)
    existing = USERS_BY_ISS_SUB.get(key)

    if existing:
        # User already provisioned
        return BootstrapResponse(
            user_id=existing["user_id"],
            tenant_id=existing["tenant_id"],
            is_new_user=False,
        )

    # 4) New user -> determine tenant_id (domain/invite/etc.)
    tenant_id = map_email_to_tenant_id(email)

    # 5) "Register" user in DB (provision)
    user_row = {
        "user_id": f"db-user-{sub}",  # in real DB: UUID
        "tenant_id": tenant_id,
        "issuer": issuer,
        "sub": sub,
        "email": email,
        "created_at": datetime.utcnow().isoformat(),
    }
    USERS_BY_ISS_SUB[key] = user_row

    return BootstrapResponse(
        user_id=user_row["user_id"],
        tenant_id=user_row["tenant_id"],
        is_new_user=True,
    )
  1. Provision the user (your app's "registration")

    The backend creates the required records:

    1. Determine tenant_id
      • Domain mapping: alice@acme.com → tenant "acme"
      • or invite-only: user must already be linked to a tenant
      • or self-serve tenant creation (first user creates tenant)
    2. Create user row
      • Store: issuer, sub, email, tenant_id, timestamps
    3. Optional defaults
      • Preferences, empty workspace, default roles, etc.
  2. Backend returns a bootstrap response

    Example:

    {
  "user_id": "…",
  "tenant_id": "…",
  "is_new_user": true
}

    React can now show an onboarding / "upload documents" screen.

Where "Registration" Actually Lives

  • Google authenticates the user.
  • Your backend decides whether the identity is allowed and attaches it to a tenant.
  • "Registration" = your DB provisioning, not creating credentials.
