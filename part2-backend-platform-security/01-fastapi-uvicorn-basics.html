FastAPI & Uvicorn Basics

Definitions

FastAPI

A modern Python web framework for building APIs. You write endpoint functions with Python type hints, and FastAPI uses them to validate requests, serialize responses, and generate interactive API docs automatically.

Uvicorn

A fast ASGI server. It’s the program that actually runs your FastAPI app and handles incoming network connections (HTTP/WebSockets), passing requests into your app and sending responses back out.

ASGI

Asynchronous Server Gateway Interface: a standard way for an async Python web app (FastAPI/Starlette) to communicate with an async web server (Uvicorn).

Why FastAPI + Uvicorn

  • FastAPI gives you the API framework (routing, validation, docs, dependency injection).
  • Uvicorn gives you the server runtime to serve it efficiently.
  • ASGI is the glue that enables async concurrency and things like WebSockets (useful for real-time apps).

Mini Tutorial: HTTP Methods

(GET / POST / PUT / PATCH / DELETE) + “safe” + “idempotent” (FastAPI context)

Anatomy of an HTTP Request

  • Method (GET/POST/PUT/…): the intent
  • URL (e.g. /documents/123): the target resource
  • Headers (e.g. Authorization)
  • Body (optional JSON): the data you send

Why there are different HTTP methods

HTTP has different methods because they communicate the intent of a request in a standardized way:

  • Safety: Some requests are meant to be read-only (GET) vs change data (POST/PUT/PATCH/DELETE).
  • Retries: Some operations are designed to be safe to repeat (PUT/DELETE are typically idempotent), which matters when networks fail.
  • Caching: Reads (GET) can be cached by browsers/CDNs/proxies; writes usually aren’t.
  • Tooling & clarity: The method + URL together form a clear contract (GET /documents/123 vs DELETE /documents/123), which helps docs, clients, gateways, and debugging.

Concepts

What “safe” means

A safe method should not change server state.

  • Safe: GET, HEAD
  • Not safe: POST, PUT, PATCH, DELETE

Why it matters: systems may prefetch, cache, or retry safe requests more freely because they shouldn’t cause side effects.

What “idempotent” means

An idempotent method means repeating the same request leads to the same final server state.

  • Idempotent: GET, HEAD, PUT, DELETE (PATCH is often idempotent if it “sets fields”, but not always)
  • Usually not idempotent: POST (repeating can create duplicates)

Why it matters: when a client times out, it might retry. Retrying idempotent operations is safer.

The Common HTTP Methods

(What they mean + when to use)

GET “Give me data” (read-only)
  • Meaning: Retrieve a resource (or list/search results)
  • Safe: ✅
  • Idempotent: ✅
  • Cacheable: often ✅
  • Use for: fetching documents, listing, searching with simple query params

Example: GET /documents/123

Client call

curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:8000/documents/123

FastAPI

@app.get("/documents/{doc_id}")
def get_document(doc_id: int):
    return {"id": doc_id, "title": "Example"}
POST “Create or perform an action”
  • Meaning: Submit data to create something or run a server action
  • Safe: ❌
  • Idempotent: ❌ (usually)
  • Cacheable: usually ❌
  • Use for: creating resources (POST /documents) or “commands” like Q&A query (POST /query)

Example: POST /query with a question in JSON

Client call

curl -X POST http://localhost:8000/query \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"question":"What does my policy say about refunds?"}'

FastAPI

from pydantic import BaseModel

class QueryRequest(BaseModel):
    question: str

@app.post("/query")
def query(req: QueryRequest):
    return {"answer": "Mock answer", "question": req.question}
PUT “Replace this resource completely”
  • Meaning: Set the resource at this URL to exactly this representation
  • Safe: ❌
  • Idempotent: ✅ (repeat = same final state)
  • Use for: full replacement updates

Example: PUT /documents/123 with full document fields

Client call

curl -X PUT http://localhost:8000/documents/123 \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"title":"New title","content":"Full content"}'
PATCH “Update part of this resource”
  • Meaning: Modify only certain fields
  • Safe: ❌
  • Idempotent: depends (often yes if it “sets” fields)
  • Use for: small updates like renaming a document

Example: PATCH /documents/123 with {"title":"Renamed"}

Client call

curl -X PATCH http://localhost:8000/documents/123 \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"title":"Renamed"}'
DELETE “Remove this resource”
  • Meaning: Delete the resource at this URL
  • Safe: ❌
  • Idempotent: ✅ (deleting twice still ends “deleted”)
  • Use for: deleting documents, records, jobs

Example: DELETE /documents/123

Client call

curl -X DELETE http://localhost:8000/documents/123 \
  -H "Authorization: Bearer $TOKEN"

A Quick Mental Model for Your Project

  • Use GET for fetching: document list/details, status checks.
  • Use POST for actions: your Q&A /query is naturally POST (complex body, not a simple “resource fetch”).
  • Use PUT/PATCH for updating document metadata or settings.
  • Use DELETE to remove a document or ingestion entry.
