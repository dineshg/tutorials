Tenant Identity vs IAM vs Workload Identity (GCP)

This tutorial fixes the #1 enterprise confusion: tenant isolation (your SaaS data boundaries) is not the same thing as Cloud IAM (who can see/use GCP resources) and not the same thing as org group gating (who is allowed to use the app at all). We’ll explain all three using your multi-tenant Doc Q&A app as the running example.

Acronyms (quick)

  • Tenant: a customer org (Company A vs Company B)
  • User: a human inside a tenant (Alice, Bob)
  • OIDC: login tokens from your IdP (Google Workspace)
  • JWT: token format often used for access tokens
  • IAM: GCP permissions to GCP resources (buckets, projects, secrets)
  • Service account: “machine identity” your backend runs as
  • Workload Identity: how GKE/Cloud Run give pods/services service-account credentials without storing keys
  • Org allowlist / group gating: “only members of Group X can use this app”

The 3-lane model (memorize this)

Lane A — End-user identity
Who is calling? Comes in the HTTP request as Authorization: Bearer .... Used to derive user_id and tenant_id.
Lane B — App authorization
What can they see? Your code enforces tenant isolation (DB filters, search filters, RLS backstop).
Lane C — Workload identity (IAM)
What can the service access? The backend’s service account permissions to GCS, Secret Manager, Cloud SQL, logs.
Org allowlist (gating)
Who is allowed in at all? Often based on Workspace/AD groups. Checked during bootstrap/login.
Shortcut: Tenant isolation protects customer data inside your app. IAM protects cloud resources. Group gating protects who can even use the app.

Scenario 1 — A normal end user uses your app (tenant isolation)

Goal

Thousands of users can hit the same API, but each user can only access their tenant’s documents.

What makes this work

  1. User logs in with Google Workspace → frontend receives an access token.
  2. Frontend calls your API with: Authorization: Bearer <token>
  3. Backend validates token, extracts user_id and maps to tenant_id.
  4. Every DB/search call is scoped: WHERE tenant_id = ...
Key point: The user does not need GCP Console access. They are not an IAM principal for your cloud resources.

Scenario 2 — An internal employee sees buckets in the GCP Console (IAM)

Goal

Some employees can see/modify buckets and projects in the GCP Console; others can’t.

What controls visibility

  1. Employee signs into GCP Console with their corporate identity.
  2. GCP checks IAM bindings (direct user or group membership).
  3. Console shows resources they have permission to list/access.
Common confusion: “I can’t see the bucket in console, so tenant isolation must be broken.” No — console visibility is IAM. Tenant isolation is your app’s logic + DB policies.

Scenario 3 — Only users in an org group can use the app (allowlist gating)

Goal

Only users in a particular org group (e.g., “DocQA-Users”) can use the app at all.

Where it’s enforced

  1. User logs in and obtains a token (OIDC).
  2. Frontend calls a bootstrap endpoint like GET /me or POST /auth/bootstrap.
  3. Backend verifies the token, then checks: “Is this user in allowed group(s)?”
  4. If not allowed → return 403. If allowed → map to tenant and proceed.
Group gating answers: “Can you use the product?” Tenant isolation answers: “Which tenant’s data can you access once inside?”

How all three show up in one request (the full picture)

Think of it as three independent checks that happen in different places.

  1. Lane A (OIDC token): Identify caller → user_id, claims, email.
  2. Org gate: Is the user allowed group(s)? If no → 403.
  3. Lane B (tenant isolation): Determine tenant_id → scope DB/search to that tenant only.
  4. Lane C (workload IAM): Backend service account reads secrets / reads & writes to GCS / logs.
POST /query HTTP/1.1
Authorization: Bearer <user_access_token>
Content-Type: application/json

{ "question": "What did Chandima mention on 1 Dec?" }

How Bearer token maps to user_id and tenant_id

Why the token string itself is not the identity

Access tokens rotate (expiry, refresh, re-login). The backend does not “recognize” a user by comparing token strings. Instead it validates the token and extracts a stable identity claim.

Typical mapping logic

  1. Verify token (signature + issuer + audience + expiry).
  2. Extract stable user key (usually sub, combined with iss).
  3. Derive tenant from:
    • a claim (e.g., tid/tenant_id) or
    • a DB lookup (issuer+sub → tenant_id) or
    • a policy mapping (email domain → tenant) for B2B setups.
Best practice: store users keyed by (issuer, sub) in your DB. That stays stable even when tokens change.

How a service account is used to write to a bucket (workload identity)

What’s happening

  • Your backend runs as a service account (machine identity).
  • That service account is granted IAM permissions on a bucket (e.g., write objects).
  • When your code calls the GCS SDK, it automatically obtains a short-lived access token for that service account.
Key idea: The backend uses its own service identity to reach GCS/Secret Manager — not the end user’s token. The end-user token only decides which tenant path/object the backend is allowed to touch.
Important: If you let end users access buckets directly via IAM, you are mixing “cloud control plane identity” with “app tenancy”. Most SaaS designs avoid this — users access files through your API (or via signed URLs), while the backend holds storage permissions.

FastAPI examples (clean + simple)

1) End-user identity dependency (Bearer → claims → user_id/tenant_id)

This is intentionally small: validate token, then map to tenant. In real code your JWT verification can use JWKS.

from fastapi import Depends, FastAPI, HTTPException
from fastapi.security import HTTPAuthorizationCredentials, HTTPBearer
from pydantic import BaseModel

app = FastAPI()
bearer = HTTPBearer(auto_error=True)

class Identity(BaseModel):
    user_id: str
    tenant_id: str
    email: str | None = None

def verify_jwt_and_get_claims(token: str) -> dict:
    # In production: verify signature (JWKS), iss/aud/exp checks.
    # Here: pretend token is valid and return decoded claims.
    return {"sub": "user-123", "iss": "https://accounts.google.com", "email": "alice@acme.com"}

def map_claims_to_tenant(claims: dict) -> str:
    # Example: domain-based tenant mapping (toy).
    email = claims.get("email") or ""
    if "@" not in email:
        raise HTTPException(status_code=403, detail="No tenant mapping")
    return email.split("@", 1)[1]  # "acme.com"

def require_allowed_group(claims: dict) -> None:
    # Example gate. In real life you would check group membership via claims or directory lookup.
    # If user is not in allowed group(s), block the app.
    allowed = True
    if not allowed:
        raise HTTPException(status_code=403, detail="Not allowed to use the app")

def get_identity(creds: HTTPAuthorizationCredentials = Depends(bearer)) -> Identity:
    claims = verify_jwt_and_get_claims(creds.credentials)
    require_allowed_group(claims)

    user_id = claims["sub"]
    tenant_id = map_claims_to_tenant(claims)
    return Identity(user_id=user_id, tenant_id=tenant_id, email=claims.get("email"))

@app.get("/me")
def me(ident: Identity = Depends(get_identity)):
    return ident

2) Workload identity in code (service account → Secret Manager / GCS)

No keys in code. The SDK picks up the pod/service’s identity automatically.

# pip install google-cloud-secret-manager google-cloud-storage

from google.cloud import secretmanager, storage

def read_secret_value(secret_resource_name: str) -> str:
    # Uses workload identity (service account) behind the scenes.
    client = secretmanager.SecretManagerServiceClient()
    resp = client.access_secret_version(request={"name": secret_resource_name})
    return resp.payload.data.decode("utf-8")

def write_tenant_object(bucket: str, tenant_id: str, object_name: str, data: bytes) -> None:
    # Uses workload identity; IAM decides if this service account can write.
    gcs = storage.Client()
    b = gcs.bucket(bucket)

    # App tenancy decides *where* we write (tenant prefix).
    blob = b.blob(f"{tenant_id}/{object_name}")
    blob.upload_from_string(data)
Separation of duties: IAM answers “can the service write to this bucket at all?” Tenant isolation answers “which tenant prefix/object is allowed for this request?”

Common mistakes (and the fix)

❌ Mistake 1: “Tenant = IAM”

Wrong: “If IAM is correct, tenants are isolated.”

Right: IAM is cloud infra permissions. Tenant isolation is your app filtering + DB policy (RLS as backstop).

❌ Mistake 2: Passing user tokens to cloud APIs

Wrong: backend calls Secret Manager with the user’s Bearer token.

Right: backend calls cloud APIs using its service account. User token only controls tenant scoping and app permissions.

❌ Mistake 3: Losing tenant context in async/background work

Wrong: enqueue a job without storing tenant_id.

Right: persist tenant_id with the job and enforce tenant checks when the job runs.

Bank-grade habit: add an audit record for “who requested what” (end-user identity) and “which workload executed it” (service identity).
