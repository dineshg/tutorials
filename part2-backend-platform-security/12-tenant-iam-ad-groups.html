Tutorial: Stop confusing Tenant Isolation vs IAM vs AD Groups

You’re dealing with three different “who are you / what can you access” systems that often get mixed up:

  1. Application tenancy (tenant_id, user_id) → your SaaS data isolation
  2. Cloud IAM → who can see/use GCP resources (Console, buckets, secrets)
  3. Org group gating (AD/Workspace groups) → who is allowed to use your app at all

They can interact, but they’re not the same thing.

The one-sentence mental model

  • IAM controls access to GCP resources (projects, buckets, secrets).
  • Tenancy controls access to your app’s data (documents, rows, search results).
  • AD/Group gating controls who can even use the app (coarse allowlist), before tenancy even matters.

Scenario 1 — Normal end user uses your app (tenant isolation)

Goal

User can only access their tenant’s documents, even though all tenants share the same backend + databases.

Step-by-step

  1. User signs in with Google Workspace
    • React uses OIDC (Auth Code + PKCE).
  2. React receives an access token
    • This is the user’s “proof” for your API.
  3. React calls your backend
    • Every call includes:
    Authorization: Bearer <access_token>
  4. Backend validates token
    • signature + issuer + audience + expiry.
  5. Backend maps token → identity
    • user_id = claims["sub"]
    • tenant_id = mapping(...) (domain mapping or DB lookup)
  6. Backend scopes all data access by tenant
    • DB queries: WHERE tenant_id = ...
    • Search queries: tenant filter
    • Jobs: store tenant_id and enforce later

What this means

  • ✅ Tenant isolation is enforced inside your app + DB/search rules.
  • 🚫 End users do not need GCP Console access or IAM permissions.

Scenario 2 — Internal employee sees buckets in GCP Console (IAM)

Goal

Some employees can see/modify certain buckets/projects, others cannot.

Step-by-step

  1. Employee logs into GCP Console
    • They are a human principal.
  2. GCP checks IAM bindings
    • Example: group:platform-admins@company.com has roles/storage.admin on a project.
  3. Console shows only permitted resources
    • If they lack storage.buckets.list or equivalent permissions, the bucket won’t show / access denied.

What this means

  • ✅ Console visibility = Cloud IAM (users/groups/roles on org/folder/project/bucket).
  • 🚫 This is unrelated to your app’s tenant_id.

The GCP Console is the cloud control plane, not your SaaS tenancy system.

Scenario 3 — App is accessible only to an AD group (org allowlist gate)

Goal

Only users in an org group (from Active Directory) can use the app at all.

Step-by-step

  1. AD group exists
    • e.g. DocQA-Users
  2. Group is synced/federated into Google Workspace
    • e.g. Google Group: docqa-users@company.com
  3. User signs in with Google (OIDC)
  4. React calls /auth/bootstrap (or /me) with: 
    Authorization: Bearer <access_token>
  5. Backend checks two things:
    • token is valid
    • user is in docqa-users@company.com
  6. If allowed: backend does tenant mapping + provisioning
    • determine tenant_id
    • create user row if missing
    • return identity

What this means

  • ✅ Group gating answers: “Can you use the app at all?”
  • ✅ Tenancy answers: “Which tenant’s data can you access?”
  • 🚫 Group gating is not tenancy.
  • 🚫 Group gating is not IAM.

Scenario 3 has two enforcement methods

Method A — Backend enforces group membership (app-level gate)

How it works

  • The backend validates the token and calls an org directory/groups API (or uses cached membership) to answer:
  • “Is alice@company.com in docqa-users@company.com?”
  • Not a member → 403
  • Member → proceed to tenant mapping + provisioning

Why you’d choose it

  • Your backend remains the “source of truth” for access.
  • You can combine it naturally with onboarding/provisioning logic.
  • Works even if you change gateways or hosting.

Trade-offs

  • Backend needs permission to check group membership.
  • You usually cache membership in DB to avoid checking on every request.

Method B — Edge/gateway enforces group membership (Apigee/IAP-style)

How it works

  • An edge layer sits in front of your backend.
  • It verifies the caller identity and enforces a policy: “must be in allowed group”
  • If user is not allowed, the request never reaches FastAPI.

Why you’d choose it

  • Strong perimeter: reduces backend exposure.
  • Central policy for multiple services: auth, rate limits, quotas, logging.

Trade-offs

  • More moving parts.
  • You still typically keep backend checks as a backstop (especially for provisioning + tenant mapping).

Best-practice combo

  • Edge gate = front door protection
  • Backend gate = final authority + tenant mapping + provisioning
  • (Defense in depth)

How all three fit together (the “3-lane model”)

Lane A — End-user identity (OIDC token)

  • Comes from Google login
  • Arrives as Authorization: Bearer ...
  • Used to derive user_id/tenant_id

Lane B — Application authorization (tenant isolation)

  • DB/search filtering rules using tenant_id
  • RLS is a backstop, not the primary identity mechanism

Lane C — Workload identity (service account IAM)

  • Backend runs as a service account
  • IAM grants that service account access to infra:
  • Cloud SQL, Cloud Storage, Secret Manager, logging, queues
  • Not tied to the human user

Common mistakes (and fixes)

Mistake 1: “Tenant = IAM”

No. Tenant is inside your app; IAM is for GCP resources.
Fix: keep end users out of IAM; enforce tenancy in app/DB/search.

Mistake 2: “If user is in AD group, they’re tenant X”

Not necessarily. Group is a coarse gate; tenant mapping is separate.
Fix: group check first, tenant mapping second.

Mistake 3: “Users should access buckets directly via IAM”

Usually no for SaaS customers.
Fix: users access files via your API (or signed URLs); backend uses service account to talk to storage.

Bootstrap decision tree (the exact order)

/auth/bootstrap (or /me):

  1. Verify token (signature, iss, aud, exp)
  2. Check org allowlist (AD/Workspace group) → if fail: 403
  3. Map to tenant_id (domain mapping or DB lookup) → if fail: 403
  4. Provision user if missing (issuer+sub key, store tenant_id)
  5. Return {user_id, tenant_id, is_new_user}
