Modern Auth Additions: Passkeys, DPoP, mTLS, Token Exchange

Why this chapter exists Chapters 6–9 in this Part cover the bedrock — bearer tokens, OAuth2, OIDC, PKCE, JWT vs opaque, JWKS. Since 2023–2025 four additions have moved from “nice to have” to “expected on enterprise reviews.” This chapter is a focused supplement, not a replacement.

1) Passkeys / WebAuthn — replacing passwords for human login

A passkey is a WebAuthn credential bound to the user's device (or a synced keychain like iCloud Keychain / Google Password Manager / 1Password). Instead of typing a password, the browser uses public‑key cryptography over a challenge issued by your server. The private key never leaves the authenticator; only a signed assertion comes back.

Why prefer passkeys over passwords (and even over TOTP)

  • Phishing‑resistant: the credential is bound to your origin (rp.id), so a fake site cannot trigger a usable signature.
  • No shared secret on the server: you store only a public key. A database breach exposes nothing reusable.
  • No "forgotten password" flow: account recovery is delegated to the platform's synced credential store.

Where it sits relative to OIDC

You typically still use OIDC for SSO across apps; passkeys replace the password step at the Identity Provider. For example, Google, GitHub, Microsoft Entra, Okta, and Auth0 all let you enable passkeys as the primary or sole authentication factor on the IdP, and the downstream OAuth2/OIDC flow is unchanged from Chapter 7.

Minimal server skeleton (FastAPI + py_webauthn)

# pip install webauthn fastapi
from fastapi import FastAPI, HTTPException
from webauthn import (
    generate_registration_options, verify_registration_response,
    generate_authentication_options, verify_authentication_response,
)
from webauthn.helpers.structs import (
    PublicKeyCredentialDescriptor, AuthenticatorSelectionCriteria,
    UserVerificationRequirement,
)

app = FastAPI()
RP_ID   = "example.com"
RP_NAME = "Example Corp"
ORIGIN  = "https://example.com"

USERS = {}        # user_id -> {"name": ..., "credentials": [...]}
CHALLENGES = {}   # user_id -> current challenge bytes

@app.post("/webauthn/register/options")
def reg_options(user_id: str, username: str):
    opts = generate_registration_options(
        rp_id=RP_ID, rp_name=RP_NAME,
        user_id=user_id.encode(), user_name=username,
        authenticator_selection=AuthenticatorSelectionCriteria(
            user_verification=UserVerificationRequirement.PREFERRED,
            resident_key="required",   # creates a discoverable passkey
        ),
    )
    CHALLENGES[user_id] = opts.challenge
    USERS.setdefault(user_id, {"name": username, "credentials": []})
    return opts            # browser calls navigator.credentials.create() with this

@app.post("/webauthn/register/verify")
def reg_verify(user_id: str, attestation: dict):
    verification = verify_registration_response(
        credential=attestation,
        expected_challenge=CHALLENGES[user_id],
        expected_origin=ORIGIN,
        expected_rp_id=RP_ID,
    )
    USERS[user_id]["credentials"].append({
        "id":         verification.credential_id,
        "public_key": verification.credential_public_key,
        "sign_count": verification.sign_count,
    })
    return {"ok": True}

The login side is symmetric: you call generate_authentication_options(...), send the resulting challenge to the browser, and verify the assertion with the public key you stored at registration.

Production reality Almost no one writes the server primitives by hand anymore. In an enterprise stack you turn passkeys on at the IdP (Entra / Okta / Auth0 / Keycloak) and your FastAPI app keeps using OIDC unchanged. The snippet above is for understanding what the IdP does.

2) DPoP — sender‑constrained access tokens (RFC 9449)

A plain bearer token (Chapter 9) is like cash: whoever holds it can spend it. If a token leaks via logs, a misconfigured proxy, or an XSS bug, the attacker gets full access until expiry. DPoP (Demonstrating Proof‑of‑Possession) binds the token to a key pair the client holds, so a stolen token is useless without the private key.

How a DPoP request looks

POST /api/v1/orders HTTP/1.1
Host: api.example.com
Authorization: DPoP eyJhbGciOiJSUzI1NiIs...<access_token>
DPoP: eyJ0eXAiOiJkcG9wK2p3dCIsImFsZyI6IkVTMjU2In0.<proof_jwt>

Two headers are sent:

  • Authorization: the access token, prefixed with DPoP instead of Bearer.
  • DPoP: a short‑lived JWT, signed with the client's private key, whose payload includes the HTTP method (htm), the request URL (htu), a nonce, the time (iat), and a hash of the access token (ath).

At token issuance, the AS records the JWK thumbprint of the client's public key inside the access token's cnf.jkt claim. The resource server verifies that the DPoP proof was signed by that exact key, that the htm/htu match this request, and that the proof is fresh.

Validating DPoP in FastAPI (sketch)

import hashlib, base64, time, jwt   # PyJWT
from fastapi import Header, HTTPException, Request

def b64url(b: bytes) -> str:
    return base64.urlsafe_b64encode(b).rstrip(b"=").decode()

async def require_dpop(
    request: Request,
    authorization: str = Header(...),
    dpop: str = Header(...),
):
    if not authorization.startswith("DPoP "):
        raise HTTPException(401, "Expected DPoP scheme")
    access_token = authorization.removeprefix("DPoP ")

    # 1) Decode the DPoP proof header to get the public key (jwk)
    proof_header = jwt.get_unverified_header(dpop)
    public_jwk   = proof_header["jwk"]

    # 2) Verify the proof signature with the embedded public key
    proof = jwt.decode(dpop, public_jwk, algorithms=[proof_header["alg"]])

    # 3) Bind to this exact request
    if proof["htm"].upper() != request.method:        raise HTTPException(401, "htm mismatch")
    if proof["htu"] != str(request.url):              raise HTTPException(401, "htu mismatch")
    if abs(time.time() - proof["iat"]) > 60:          raise HTTPException(401, "stale proof")

    # 4) Bind proof to this access token (ath = base64url(SHA-256(access_token)))
    ath_expected = b64url(hashlib.sha256(access_token.encode()).digest())
    if proof.get("ath") != ath_expected:              raise HTTPException(401, "ath mismatch")

    # 5) Verify access_token (JWKS as in Chapter 9) and check cnf.jkt
    claims = verify_access_token(access_token)        # your JWKS validator
    jkt    = jwk_thumbprint(public_jwk)               # RFC 7638
    if claims.get("cnf", {}).get("jkt") != jkt:       raise HTTPException(401, "token not bound to this key")

    return claims

When to adopt DPoP

  • Public clients (SPAs, mobile apps) that cannot be issued mTLS certs.
  • High‑sensitivity APIs (payments, admin) where bearer leakage is unacceptable.
  • MCP servers that accept tokens forwarded from a host — see Part III.

3) mTLS — sender‑constrained tokens for service‑to‑service (RFC 8705)

For workload‑to‑workload calls inside your platform (Chapter 10 / 11), the equivalent of DPoP is mutual TLS. The calling service presents an X.509 client certificate during the TLS handshake, and the AS binds the issued access token to the certificate's hash via the same cnf mechanism (cnf.x5t#S256).

The flow

  1. Workload requests a token from the AS over mTLS.
  2. AS records the SHA‑256 thumbprint of the client cert in cnf.x5t#S256.
  3. Workload calls the resource server also over mTLS with the same cert.
  4. Resource server verifies that the cert hash from the TLS layer matches the cnf.x5t#S256 claim in the token.

Where it fits in your reference architecture

This is the strongest match for the “workload identity” boundary already covered in Chapter 10 (end‑user vs workload identity) and Chapter 11 (GCP workload identity). On GKE, SPIFFE/SPIRE or Istio's PeerAuthentication issue the SVIDs (X.509) automatically.

4) OAuth 2.0 Token Exchange (RFC 8693) — propagating the user downstream

The classic question from Chapter 10: "My user calls FastAPI with their OIDC token. FastAPI then calls a downstream API that needs the user's identity, not FastAPI's. How do I forward identity safely?"

The 2025 answer is token exchange. FastAPI exchanges the inbound user token at the AS for a new, downstream‑audience token that still carries the user's identity but is scoped narrowly:

POST /oauth/token HTTP/1.1
Host: idp.example.com
Authorization: Basic <FastAPI client credentials>
Content-Type: application/x-www-form-urlencoded

grant_type=urn:ietf:params:oauth:grant-type:token-exchange
&subject_token=<the inbound user access token>
&subject_token_type=urn:ietf:params:oauth:token-type:access_token
&audience=https://crm.example.com
&scope=crm.read

The IdP returns a new access token whose:

  • sub is still the original user,
  • aud is the downstream API,
  • act (the actor claim, RFC 8693 §4.1) records that FastAPI was the acting service — so the downstream API can audit “user X via service Y.”
Anti‑pattern alert Forwarding the inbound user token verbatim to a downstream API is the most common mistake. It works, but the downstream API now sees a token whose aud claim is your service, which a good resource server will reject. Token exchange (RFC 8693) is the correct fix. The on‑behalf‑of (OBO) flow in Microsoft Entra is the same pattern with vendor‑specific naming.

5) Quick decision matrix

BoundaryUseWhy
Human → Browser → Your app OIDC + Passkeys at the IdP, PKCE for the app Phishing‑resistant, no passwords stored.
Browser/SPA/mobile → Your API OIDC + DPoP access tokens Stolen tokens are unusable without the client's private key.
Workload → Workload (inside platform) mTLS + cert‑bound tokens (cnf.x5t#S256) Strongest service‑identity binding; SPIFFE/Istio handle issuance.
Workload → Workload (across orgs) Workload Identity Federation (Chapter 11) + signed JWT assertion No long‑lived secrets to rotate.
Your API → Downstream API on user's behalf Token exchange (RFC 8693) / OBO Preserves user identity, narrows audience, records the actor.
