Background Tasks & Retries in FastAPI

Next step tutorial: Background tasks + retries in FastAPI (with your multi-tenant "ingest" example)

What You're Trying to Achieve

In your app, some operations (like document ingestion) take time. You don't want the user to wait for extraction/embedding/indexing inside the HTTP request. So you:

  1. Accept a request quickly (POST /ingest)
  2. Return immediately with a job_id
  3. Run the heavy work in a background task
  4. Make it safe if the client retries (using Idempotency-Key)
  5. Add server-side retries for flaky dependencies inside the job
Part A — Background tasks: do work after the response

1) The endpoint returns immediately

from fastapi import FastAPI, BackgroundTasks, Depends, Header, HTTPException
from pydantic import BaseModel
import uuid
import time

app = FastAPI()

# --- pretend identity (normally from JWT/Bearer dependency) ---
class Identity(BaseModel):
    tenant_id: str
    user_id: str

def get_identity() -> Identity:
    # placeholder: in real life, derive from Bearer token
    return Identity(tenant_id="tenant-acme", user_id="user-123")

# --- pretend DB tables ---
JOBS: dict[str, dict] = {}               # job_id -> {tenant_id,user_id,status}
IDEMPOTENCY: dict[str, dict] = {}        # (tenant_id, idem_key) -> response

class IngestResponse(BaseModel):
    job_id: str
    status: str

def ingest_job(job_id: str):
    # runs after response
    JOBS[job_id]["status"] = "running"
    time.sleep(2)  # pretend long work
    JOBS[job_id]["status"] = "done"

@app.post("/ingest", response_model=IngestResponse)
def ingest(
    background: BackgroundTasks,
    ident: Identity = Depends(get_identity),
    idempotency_key: str | None = Header(None, alias="Idempotency-Key"),
):
    if not idempotency_key:
        raise HTTPException(status_code=400, detail="Missing Idempotency-Key")

    idem_store_key = f"{ident.tenant_id}:{idempotency_key}"

    # if client retries, return the original response
    if idem_store_key in IDEMPOTENCY:
        return IDEMPOTENCY[idem_store_key]

    job_id = str(uuid.uuid4())
    JOBS[job_id] = {"tenant_id": ident.tenant_id, "user_id": ident.user_id, "status": "queued"}

    response = {"job_id": job_id, "status": "queued"}
    IDEMPOTENCY[idem_store_key] = response

    # schedule work after response
    background.add_task(ingest_job, job_id)

    return response

How the client calls it (retry-safe)

POST /ingest HTTP/1.1
Authorization: Bearer <token>
Idempotency-Key: 9c2f9f2f-9bd8-4d1e-8b45-4a338f89f9a0
Content-Type: application/json

{}
Key point: If the request times out and the client retries with the same Idempotency-Key, it gets the same job_id, not a duplicate job.
Part B — Retries: handle flaky downstream calls

In ingestion you'll call services that may transiently fail (network hiccups, 429, 503). You add retries inside the background job, not by rerunning the whole HTTP request.

2) Add a small retry helper (with backoff)

import time
import random

class TransientError(Exception):
    pass

def retry(fn, attempts: int = 3, base_delay: float = 0.25):
    last = None
    for i in range(attempts):
        try:
            return fn()
        except TransientError as e:
            last = e
            time.sleep(base_delay * (2 ** i))  # 0.25, 0.5, 1.0...
    raise last

3) Use retries in the background ingestion job

def call_elasticsearch_index():
    # pretend random transient failure
    if random.random() < 0.4:
        raise TransientError("ES 503")
    return True

def call_embedding_service():
    if random.random() < 0.4:
        raise TransientError("Embedding timeout")
    return True

def ingest_job(job_id: str):
    JOBS[job_id]["status"] = "running"

    # retry flaky steps
    retry(call_elasticsearch_index, attempts=3)
    retry(call_embedding_service, attempts=3)

    JOBS[job_id]["status"] = "done"
Part C — Multi-tenant isolation (what to remember)

Notice we stored tenant/user on the job:

JOBS[job_id] = {"tenant_id": ident.tenant_id, "user_id": ident.user_id, ...}
The rule: Every job and every downstream action must carry the tenant scope. Even in background work, you never run a job "without tenant context".

📚 Summary (what you learned)

  • BackgroundTasks let you respond fast and do best-effort work after the response.
  • Retries belong inside your outbound calls (ES/LLM/etc.), not by re-running unsafe POSTs.
  • Idempotency-Key makes POST safe to retry from the client.
  • Tenant/user identity should be attached to jobs and used everywhere.
