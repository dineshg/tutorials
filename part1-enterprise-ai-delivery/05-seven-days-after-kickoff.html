AI Lead Playbook: 7 Days After Kickoff

This tutorial is a step-by-step guide for what an AI Lead Engineer does immediately after the pilot kickoff meeting. It is aligned to our example project: Executive Document Q&A (meeting notes/packs → cited answers) with strict access control. The focus is not “build a demo” — it’s “ship a safe, measurable pilot.”

0) North Star: what we’re actually building

A pilot tool that answers questions like: “What did Chandima mention about no meeting on 1 Dec?” and returns:

  • Short answer (2–6 sentences)
  • Citations (doc title + link + excerpt; timestamp if transcript exists)
  • Safe failure (“not found” when evidence is insufficient)
  • Audit trail (who asked, what evidence was used)
Lead-level framing: “If you can’t cite it, you can’t say it.”
Pilot constraint: we only answer over an approved, bounded corpus. “All company docs” is a later phase.

1) The 7 days after kickoff (what I do, day by day)

Day 0–1: Freeze decisions + unblock access

Goal Make the pilot buildable by eliminating ambiguity and starting approval workflows.

  1. Publish the Decision Record (pilot cohort, corpus, exclusions, success metrics, named owners).
  2. Create the Approvals Tracker (data owner approvals, IAM/service identity requests, privacy/risk checks).
  3. Confirm the pilot cohort mechanism: names vs org group (recommended: group).
Why this is first: approvals take longer than engineering. Start them immediately or delivery stalls in week 2.

Day 2–3: Produce reviewable design + sprint-ready execution plan

Goal Get “approval to build” and start engineering with a clear vertical slice milestone.

  1. Write the Architecture Review Pack (HLD) with trust boundaries, access model, data flows, risks.
  2. Write 3–5 ADRs documenting key decisions (citations-only, tenant isolation, corpus gating, etc.).
  3. Convert the plan into epics + stories and define the vertical slice milestone.
Vertical slice definition: ingest a small approved sample → retrieve with filters → answer with citations → audit log → eval run.

Day 3–4: Co-design with EA + Ops and build the “Golden Set”

Goal Lock real workflow and create evaluation gates.

  1. Run a co-design session with EA + Ops to define what “usable output” looks like.
  2. Collect 30–50 real questions and record expected source documents.
  3. Define the scoring rubric (citation correctness, groundedness, refusal correctness).
AI lead rule: you don’t “finish prompting” — you ship prompts behind eval gates and iterate with evidence.

Day 5–7: Set the operating model + drive the first slice to demo readiness

Goal Make the pilot safe to run, not just build.

  1. Define runbooks (wrong answer investigation, “not found” investigation, dependency outage behavior).
  2. Define go-live checklist (auth, gating, audit logs, eval threshold, dashboards).
  3. Demo the vertical slice internally to stakeholders and capture feedback.
Important: A pilot without monitoring, audit logs, and a support owner is a “prototype,” not enterprise delivery.

2.2 Architecture Review Pack (HLD) — what it must contain

  • System diagram + trust boundaries (where identity is verified, where data is stored, where LLM is called)
  • Identity + access model: SSO, group gating, corpus gating, tenant/user mapping
  • Data flow: what is ingested, stored, indexed, retrieved, and sent to the LLM
  • Threats & mitigations: leakage, prompt injection, sensitive logs, exfil
  • Operational view: metrics/logs/traces, failure modes, runbook ownership
Architecture review purpose: approval to build (early) and approval to go-live (later).

2.3 ADRs (Architecture Decision Records) for this project

  • ADR-001 — “Cited answers only; refuse without evidence.”
  • ADR-002 — “Tenant isolation strategy (tenant_id/user_id; DB backstop).”
  • ADR-003 — “Pilot uses corpus allowlist; per-doc ACLs in phase 2.”
  • ADR-004 — “Model boundary: send only retrieved snippets; redact sensitive patterns.”
  • ADR-005 — “Evaluation gates required for prompt/retrieval changes.”

2.4 EA/Ops Co-Design Notes + Output Contract

  • What counts as “proof”: snippet + link, optional timestamp
  • Answer length and tone suitable for executive briefings
  • Refusal rules (“not found” behavior)
  • Examples of good questions vs bad questions

2.5 Golden Set + Rubric (evaluation pack)

You build this from real EA/Ops questions. It becomes your go-live gate.

Golden Set entry example:
- Question: "What did Chandima mention about no meeting on 1 Dec?"
- Expected evidence: "Exec Forum Minutes 2024-12-01" section "Agenda changes"
- Score:
  - Citation correctness (0/1/2)
  - Groundedness (0/1/2)
  - Refusal correctness (0/1)
  - Helpfulness (0/1)
Lead-level move: every prompt/retrieval change runs the golden set and compares metrics.

2.6 Runbooks + Go-live checklist

  • Runbook: “Wrong answer” → check citations → check retrieved chunks → check ingestion freshness
  • Runbook: “Not found” → verify corpus boundaries → verify metadata → verify index update
  • Checklist: access control enforced, audit logs written, eval thresholds met, dashboards live

4) Release gates (what must be true before pilot users get access)

  • Auth gate: only allowed org group can access
  • Corpus gate: retrieval restricted to approved doc sets
  • Citation gate: answers contain citations or refuse
  • Leakage gate: cross-user/cross-corpus tests pass
  • Audit gate: every answer logs user + cited sources
  • Eval gate: golden set meets threshold
  • Ops gate: dashboards + runbook + support owner exist
Enterprise truth: “Working” is not enough. It must be measurable, auditable, and operable.

Result: by the end of week 1, you have a governable plan + measurable gates + build momentum.
