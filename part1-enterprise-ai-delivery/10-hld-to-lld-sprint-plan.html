1) Planning principles (enterprise reality)

What makes a backlog “buildable”
  • Every item has an owner, inputs, outputs, and acceptance criteria
  • Dependencies are explicit (IAM, networking, data access, secrets)
  • Work is sliced end-to-end (vertical), not by layers only
  • Safety gates are included as first-class requirements
Common failure modes
  • Only building “the model part” without operability/audit
  • Stories too large (“build retrieval system”) → unclear progress
  • Security conditions treated as “later” → late rework
  • Dependencies discovered mid-sprint → blocked delivery
Lead-level mantra: “We deliver a safe end-to-end slice early, then improve quality and scale.”

2) Define epics (map directly to your HLD boxes)

Start by creating epics that mirror the architecture. For this project, a clean set is:

  • Epic A — Auth & gating: OIDC validation, org allowlist, tenant mapping, bootstrap endpoint
  • Epic B — Ingestion pipeline: intake event, extraction, canonical store, indexing
  • Epic C — Retrieval: strict tenant/corpus filters, ranking, chunk fetch
  • Epic D — Prompting & citations: context packing, citation format, refusal rules
  • Epic E — Audit & observability: structured logs, traces, dashboards, runbook hooks
  • Epic F — Evaluation harness: golden set, citation correctness checks, regression gate in CI/CD
Tip: If your architecture review produced “approval conditions”, create one epic or track lane for each condition so nothing gets lost.

3) Define the vertical slice milestone (the most important planning move)

A vertical slice is a thin, end-to-end path that proves the system works safely with real constraints. It is not “production ready”, but it is real and demoable.

Vertical slice (Exec Doc Q&A) — definition:
  • One user logs in (Workspace OIDC) → backend validates → returns identity
  • One small approved corpus (e.g., 10 meeting notes) is ingested
  • Query retrieves tenant/corpus-filtered chunks
  • LLM returns an answer with citations (or refuses)
  • Audit log is written (no sensitive snippet dumps)
  • Golden set eval runs and produces a simple report
Anti-pattern: building “ingestion” for weeks before any end-to-end query works. Your team should be able to run a demo early.

4) Turn epics into buildable stories (how to slice work)

Each story should be deliverable in 1–4 days by one engineer and have clear testable outputs. Use a consistent story template:

Story field What to write
Title Verb + object (e.g., “Implement /auth/bootstrap identity endpoint”)
Goal What capability is enabled and for whom
Inputs Request format, upstream data, required secrets
Outputs Response format, side effects (DB rows, logs, events)
Acceptance criteria Functional + safety + observability tests
Dependencies IAM, networking, platform tickets, approvals

Example story breakdown (one epic)

  • Auth/gating: “Verify JWT”, “Check org allowlist”, “Map tenant”, “Provision user”, “Return /me”
  • Retrieval: “Index schema includes tenant/corpus tags”, “Search with filters”, “Fetch chunks”
  • Prompting: “Format citations”, “Refuse when insufficient context”, “Redact patterns”

5) Acceptance criteria (include safety gates, not just “it works”)

In GenAI, acceptance criteria must include correctness, leakage prevention, and governance. Use the “3-layer” criteria pattern:

Functional
  • Endpoint works for happy path
  • Correct request/response schema
  • Handles expected error cases
Safety & security
  • Tenant/corpus filters enforced
  • Prompt injection mitigations applied
  • Refusal behavior when evidence is missing
Operability
  • Structured logs with request_id/user/tenant
  • Metrics for latency and failure codes
  • Tracing across retrieval + LLM call
Evidence
  • Unit tests + integration test runs
  • Golden set eval report updated
  • Demo script for stakeholders
Key acceptance gate: “Cross-tenant retrieval attempt returns zero results (and logs an audit event).”

6) Assign work to 5 direct reports (a practical team shape)

With 5 engineers, assign ownership by stream but ensure the vertical slice comes together. A common structure:

Engineer (example) Primary stream Deliverables
Eng 1 Auth & identity /auth/bootstrap, tenant mapping, org allowlist gate, user provisioning
Eng 2 Ingestion Pub/Sub/Tasks wiring, extractor stub, canonical store writes, index writes
Eng 3 Retrieval Strict filters, top-k search, chunk fetch, ranking tuning hooks
Eng 4 Prompting & citations Context packing, citation format, refusal rules, redaction hooks
Eng 5 Evals + observability Golden set harness, citation correctness checks, dashboards/alerts, runbook skeleton
Your role: manage integration points (contracts, schemas), sequencing, and keep “approval conditions” tracked. You should also reserve time for cross-cutting reviews and stakeholder updates.

7) Example sprint plan (2 sprints to a vertical slice)

Example assumes 2-week sprints. The goal is a demoable vertical slice by the end of Sprint 1 or early Sprint 2.

Sprint 1 — Vertical slice foundations (end-to-end demo)

Epic Stories (examples) Acceptance focus
Auth/gating /auth/bootstrap validates JWT; org allowlist check; tenant mapping; user row create-if-missing 403 if not allowed; stable user mapping; audit log entry
Ingestion Ingest 10 sample docs into canonical store; write minimal index entries with tenant/corpus tags Repeatable ingestion; no cross-tenant tags; basic error handling
Retrieval Search endpoint with strict tenant/corpus filters; returns chunk IDs and text Cross-tenant search returns 0; latency baseline measured
Prompting/citations RAG prompt template; citations required; refusal when insufficient evidence No uncited answers; refusal cases tested
Obs/audit Structured log schema; request_id; traces across retrieval + LLM No raw snippets in logs; dashboard stub exists
Evals Golden set (10–20 Qs); basic evaluator: citation presence + answer groundedness checklist Eval report produced in CI; baseline recorded

Sprint 2 — Quality + safety hardening (pilot-ready)

  • Improve retrieval quality (hybrid ranking, chunking refinements)
  • Add stronger leakage tests and prompt injection tests
  • Add redaction rules and validate policy compliance
  • Expand eval set + add regression gates (must not degrade)
  • Operational readiness: alerts, runbook, on-call handoff

8) Definition of Done (what “complete” means in this org)

Define DoD once and apply to every story. This prevents “works on my machine” delivery.

  • ✅ Code merged with tests (unit + at least one integration test)
  • ✅ Observability: logs/metrics/traces updated for the feature
  • ✅ Security: tenant/corpus enforcement verified (test case included)
  • ✅ Documentation: README snippet or runbook note updated
  • ✅ Demoable: a curl/script or Postman request exists
Lead-level improvement: add a “safety test” checkbox to every story by default.
