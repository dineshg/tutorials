Step 4–5 (GCP): Provisioning Tickets + Concrete Service Choices + Latency + 1000s of Users

This tutorial expands Step 4 and Step 5 for your multi-tenant Document Q&A system on GCP. It answers: which exact services you provision, why GKE / Pub/Sub / Cloud Tasks, what latency targets are acceptable, and how to handle 1000s of simultaneous users in a large enterprise.

1) Concrete reference architecture (GCP services)

Capability Concrete GCP implementation Notes (bank/enterprise friendly)
Frontend Cloud Run (React static via CDN or server-render), behind HTTPS LB Public entry, simple scaling; no direct access to data stores
API backend GKE (private cluster) running FastAPI, multiple replicas Strong network controls, sidecars, private egress, platform patterns
Object storage Cloud Storage (GCS) buckets (raw docs + processed artifacts) Retention policies, CMEK (KMS), uniform bucket-level access
Canonical store Cloud SQL for PostgreSQL (tenants, users, documents, chunks metadata) Private IP + IAM DB auth where possible; RLS optional backstop

Document extraction Option A: Document AI (OCR/forms) called by an ingestion worker Best for scans/PDFs; keep outputs minimal and stored internally
Search / vectors (Your design) Elasticsearch/OpenSearch hosted/managed or in-cluster Must enforce tenant/corpus filters; isolate indices or filter fields
Secrets Secret Manager + CMEK KMS keys Rotation + least privilege; avoid secrets baked into images

Key boundary: Human users authenticate to your app (OIDC token). Your workloads authenticate to GCP using workload identity + service accounts. End users do not get GCP IAM access to buckets/DB.

2) Document extraction tooling: where implemented and why

Extraction is almost never done in the interactive API path. It is an async worker responsibility. The normal pattern:

  1. User uploads document → stored in gs://raw-docs
  2. Upload emits an event (Pub/Sub) → ingestion worker picks it up
  3. Worker performs extraction (Document AI / libraries) → stores canonical text in Cloud SQL + processed text in GCS
  4. Worker generates chunks + embeddings → writes to vector store (with strict tenant/corpus tags)
Option A (common): Document AI
  • Best when you have PDFs, scans, mixed layouts, forms
  • Managed OCR quality tends to beat DIY OCR
  • Worker calls Document AI API; you store only needed output
Option B: Self-managed extractors
  • Apache Tika / unstructured / custom parsers in a worker container
  • Good for Office docs / HTML / text-heavy content
  • More operational burden (libs, CVEs, memory spikes)
Enterprise-friendly practice: “extract → store canonical text internally → index” and keep extraction strictly async with retries and DLQs.

3) Why GKE for the backend (and what Cloud Run would change)

Why choose GKE (common in banks)
  • Private cluster + tighter network isolation patterns
  • Sidecars (proxies, policy agents), service mesh (if required)
  • More predictable performance under heavy concurrency
  • Consistent “platform standard” if the org runs many services on GKE
When Cloud Run is acceptable
  • Faster initial delivery, simpler operations
  • Great for stateless HTTP services
  • Still supports service identities, private networking (with setup)
  • Less flexibility for complex in-cluster patterns
Decision framing: In a large org, you usually align with the platform team’s “golden path”. If the enterprise runs private GKE landing zones, GKE reduces approval friction.

4) Why Pub/Sub + Cloud Tasks (and what they solve)

Pub/Sub: event distribution (fan-out)
  • “Something happened” messaging (document uploaded, extraction complete)
  • Multiple consumers can subscribe (indexer, audit, notifications)
  • High throughput and decoupling
Cloud Tasks: controlled execution (rate + retries)
  • Push a specific HTTP job to a worker endpoint
  • Built-in retry + scheduling + DLQ patterns
  • Enforce per-queue throughput (useful for external API limits)
Typical pattern: Pub/Sub announces the event; Cloud Tasks is used for the “do work” step when you need strict retry, backoff, and rate control.
What about Kafka? Acceptable in some orgs, but usually heavier to operate. If your enterprise already has it, it’s a viable alternative. Otherwise Pub/Sub is the GCP-native path.

5) Latency targets: what’s acceptable and how to meet it

You should define SLOs separately for interactive query and async ingestion. Concrete targets that work for exec Q&A:

Path Target (p50 / p95) How you hit it
/query (interactive) p50: 2–4s, p95: 6–10s Async I/O, fast retrieval (sub-second), bounded context size, caching, LLM timeouts
Retrieval alone p50: 100–300ms, p95: < 800ms Tenant-filtered indices, warm caches, limit top_k, avoid expensive rescoring
LLM call alone p50: 1–3s, p95: 5–8s Streaming responses, concurrency limits, fallback/timeout, smaller prompts
Ingestion (async) Minutes are fine (SLO: < 15 min typical) Worker autoscale, retries, DLQ, document size limits, batch processing
Practical trick: Stream the LLM answer to the UI so the user “sees progress” even if p95 is high. It improves perceived latency.

6) Handling 1000s of simultaneous users (GKE + FastAPI + external LLM constraints)

At scale, your true bottleneck is often the LLM endpoint, not FastAPI itself. Your strategy is: keep the API stateless and fast, and protect downstream dependencies.

6.1 API layer (FastAPI on GKE) — what actually scales

  • Stateless replicas: multiple pods behind a load balancer; each request carries its identity.
  • Async I/O: use async clients for DB/search/LLM so threads aren’t blocked waiting on network.
  • Connection pooling: DB and HTTP clients pooled per pod; avoid per-request reconnects.
  • Timeouts + circuit breakers: fail fast when LLM/retrieval is degraded.

6.2 Downstream protection (what keeps you stable)

Rate limits & fairness
  • Per-tenant concurrency caps (avoid one tenant consuming all capacity)
  • Global cap for LLM calls (protect the quota)
  • Queueing for overflow (return 429 / retry-after when needed)
Degradation strategies
  • If retrieval slow → lower top_k / skip re-rank
  • If LLM slow → shorter prompt / cheaper model / refusal with guidance
  • If dependency down → return “try again” with tracking ID
Key point: “1000s of users” is fine for stateless HTTP + retrieval. The hard part is controlling LLM concurrency and ensuring tenant fairness.

Lead technique: Every ticket should specify: service name, data classification, identity (GSA), network path, and evidence needed for approval.

8) Step 5: Engineering kickoff (internal) — make “good” unambiguous

Goal Align the team on scope, contracts, evaluation gates, and Definition of Done.

Kickoff checklist

  • Confirm vertical slice: the exact demo you will deliver at end of Sprint 1
  • API contracts: /auth/bootstrap, /query, /ingest (request/response + error codes)
  • Evaluation gates: golden set baseline, citation correctness checks, regression threshold
  • Definition of Done: includes tests + logs/metrics/traces + safety checks
  • Integration points: schema for docs/chunks, tenant/corpus tag format, log schema
Best practice: write the vertical slice as a single end-to-end “demo script” (login → ingest sample → ask question → see citations → check audit log entry).
Common rework source: engineers implement in parallel without agreed contracts. Freeze the first version of schemas and endpoints early.
