1) HLD decision principles (what we decide now vs later)

Decide at HLD
  • Major components and trust boundaries
  • Security posture (identity, access enforcement, secrets)
  • Data boundary: what can go to the LLM
  • Core services that create long lead-time dependencies
  • Audit/observability requirements
Defer to LLD
  • Exact schemas, chunk sizing, prompt templates
  • Fine-grained retry/backoff parameters
  • Exact IAM role bindings per resource (unless forced)
  • Exact SLOs (propose targets at HLD, finalize later)
Rule of thumb: if it affects security/privacy, cost materially, operability, or requires another team’s provisioning, it must be decided (or at least bounded) at HLD.

2) Reference architecture (GCP-aligned)

Request path (query time)

  1. User signs in via Google Workspace (OIDC) → SPA gets access_token.
  2. SPA calls backend API with Authorization: Bearer ....
  3. Backend validates JWT, maps to user_id/tenant_id, enforces group/corpus gates.
  4. Backend retrieves chunks (strict filters) → packs context → calls LLM → returns answer with citations.
  5. Backend writes audit logs (user, tenant, docs cited, timings).

Ingestion path (offline)

  1. Document discovered/uploaded → event published.
  2. Worker extracts text + metadata → stores canonical record.
  3. Embeddings/index written to retrieval system with tenant/corpus metadata.
Core HLD principle: access control is enforced in the backend and data layers, not in the browser.

3) GCP service choices (what we pick, why, alternatives, and constraints)

3.1 End-user Identity (Google Workspace SSO)

Quick concept: What is an “Org allowlist”?

An Org allowlist is a “front door” rule that answers: “Is this person allowed to use our application at all?” This is separate from tenant isolation (“what data can you access once inside?”).

Authentication (Google login) proves who you are.
Org allowlist decides whether you’re allowed to enter.
Tenant isolation decides what you can see after you enter.

Where it happens (two common places)

In enterprises, you usually implement the allowlist in one of these places:

Option A — At the IdP (preferred when available)

You configure access rules at the Identity Provider (Google Workspace / your IdP) so only approved users can complete login for this application.

  • How: use Google/IdP app access controls or group-based assignment (only members can use the app).
  • Effect: users outside the group often cannot complete sign-in for this app (blocked earlier).
  • Benefit: “blocked at the door” before tokens are issued for your app.
Option B — In your application server (common and reliable)

Your backend (FastAPI) enforces the allowlist on every request by checking group membership (or a local allowlist) after validating the JWT.

  • How: verify token → read identity → check “is user in approved org group?”
  • Effect: user can log in, but your API returns 403 Forbidden if not allowed.
  • Benefit: uniform enforcement even if IdP-side restrictions are incomplete.
Enterprise best practice: use both when possible.
IdP-side gating reduces exposure and noise; app-side gating guarantees enforcement and supports richer rules (e.g., per-tenant mapping).

How it looks in our project (Google Workspace group)

We define a Google Group like docqa-users@company.com. Only members are allowed to use the product. 

Policy (conceptual):
1) Verify JWT (signature, issuer, audience, expiry)
2) Extract stable identity (issuer + sub, and email if present)
3) Check org allowlist:
   - if NOT in docqa-users@company.com → 403 "not allowed"
   - if IN group → continue
4) Map user → tenant_id (domain or DB lookup)
5) Enforce tenant/corpus filters on retrieval + data access

Why it exists (common reasons)

  • Security & governance: limit exposure to approved users only.
  • Privacy: reduce risk by restricting who can query sensitive corpora.
  • Pilot control: roll out to a small cohort first (execs + EAs), then expand.
  • Cost control: LLM usage is expensive—control who can generate traffic.

Don’t confuse this with GCP IAM

End users typically do not need GCP IAM permissions. They access documents through the application. The backend uses its own service identity (service account / workload identity) to access GCS, DB, secrets, etc.

Org allowlist = “Can this user use the app?”
Cloud IAM = “Can this principal access GCP resources (buckets, secrets, projects)?”
Capability Chosen (HLD) Why Alternatives considered Enterprise constraints / delays
Login + tokens Google Workspace OIDC (Auth Code + PKCE in SPA) Standard enterprise SSO; stateless API validation Identity Platform; custom IdP; IAP in front of app Token audience/issuer alignment; group claim policy; approval for group gating approach
Org allowlist Google Groups gating (e.g., docqa-users@company.com) Simple “allowed to use product” control App-local allowlist; IAP access policy Group ownership, membership governance, and audit rules
HLD note: group gating answers “who can use the app”; tenant isolation answers “what data can they access.” They are related but not the same.

3.2 Frontend hosting (React)

Chosen (HLD) Why Alternatives Delays / internal patterns
Cloud Run (frontend service) Easy rollout; supports rewrites/headers and enterprise networking patterns if needed Cloud Storage + Cloud CDN (static); Firebase Hosting DNS/cert/WAF approvals can be longer than development

3.3 Backend compute (FastAPI)

Chosen (HLD) Why Alternatives Delays / internal patterns
GKE (Kubernetes) Enterprise standard for regulated workloads; consistent policy enforcement; supports internal services Cloud Run (faster to ship); MIG on GCE Landing zone onboarding, hardened node pools, image scanning gates, network policies, mesh requirements
HLD articulation: “GKE is chosen not because it’s trendy, but because it matches enterprise platform patterns and controls.”

3.4 Canonical store (metadata + system of record)

Chosen (HLD) Why Alternatives Delays / internal patterns
Cloud SQL (PostgreSQL) Clear system-of-record; relational metadata; supports backstop controls (e.g., RLS) if needed AlloyDB (scale/perf); Firestore (not ideal for relational workloads) Private IP setup; KMS key approvals; DBA access model; backups/retention policies

3.5 Object storage (raw docs + extracted artifacts)

Chosen (HLD) Why Alternatives Delays / internal patterns
Cloud Storage (GCS) Durable, cheap, standard; fits ingestion pipeline and artifact storage Filestore (rare); third-party storage (vendor risk) Uniform bucket-level access, retention policies, VPC-SC restrictions, data residency constraints

3.6 Retrieval / Semantic Search (vector + strict metadata filters)

At HLD, you choose one path for the pilot and document the fallback/phase-2 alternative. The key requirement is strict filtering (tenant/corpus) to avoid leakage.

Option When we choose it Why Tradeoffs Delays / internal patterns
Option A: Vertex AI Search / Agent Builder Pilot needs speed and managed operations Managed retrieval; fast path; less ops burden Less low-level control; feature fit must be validated Service enablement approvals; data classification rules; integration reviews
Option B: OpenSearch/Elasticsearch on GKE Need maximum control or org already runs search platform Full control: hybrid search, filters, scoring, tuning Higher ops cost; patching/backups/sizing required Platform onboarding queue; cluster hardening; storage class decisions; backup standards
HLD best practice: choose the simplest option that satisfies strict filtering and governance. “Perfect retrieval” is not required for pilot, but “safe retrieval” is.

3.7 Async ingestion orchestration (durable retries)

Chosen (HLD) Why Alternatives Delays / internal patterns
Pub/Sub + Cloud Tasks (or Workflows if needed) Durable eventing + retries; survives restarts; rate limiting GKE-native queues; scheduler jobs; best-effort background tasks IAM approvals for topics/subscriptions; DLQ standards; audit logging for processing events
HLD note: FastAPI background tasks are fine for light best-effort work, but ingestion needs persistence and retries.

3.8 Secrets and keys

Chosen (HLD) Why Alternatives Delays / internal patterns
Secret Manager + Cloud KMS (policy-driven) Rotation, audit, separation of duties; enterprise standard Kubernetes secrets (often restricted); third-party vault Access approval queue; rotation policy alignment; KMS key provisioning in approved region/project

3.9 LLM boundary (external provider calls)

Chosen (HLD) Why Alternatives Delays / internal patterns
Backend on GKE calls external LLM via HTTPS (data minimised) Supports chosen model; keeps enforcement central; can redact/pack context Vertex AI (Gemini) to keep boundary within GCP; multi-provider routing Egress allowlisting/proxy; legal/compliance review of provider; data residency policy checks
HLD statement to include: “Only retrieved snippets are sent to the model. No raw document dumps. Logs never include full snippets.”

3.10 Observability and audit

Chosen (HLD) Why Alternatives Delays / internal patterns
Cloud Logging + Cloud Monitoring + Cloud Trace (optional: BigQuery for long-term audit analytics) Required for enterprise support and governance; standard GCP tooling Splunk/Datadog (if org standard); custom ELK Retention policy approvals; sensitive logging rules; SIEM integration lead time
Audit minimum (per request): who asked, which tenant, what sources were cited, timings, and outcome (answer/refusal).

4) Enterprise delays & critical path (what usually takes longest)

In banks/airlines, “time to pilot” is often gated by approvals and provisioning, not coding. HLD should explicitly call out the critical path so stakeholders understand sequencing.

Workstream What causes delay How the AI lead reduces risk
Landing zone / projects Dev/stage/prod project creation, org policies, shared VPC attachment Request early; align with platform team templates; keep pilot in one approved project if allowed
Networking Private egress, DNS, firewall rules, proxy requirements Document endpoints early; ask for a standard pattern; avoid custom exceptions
IAM / service identities Least-privilege reviews, approvals, periodic reviews Provide role matrix; use standard roles; limit resources touched in pilot
Data owner approval Corpus sign-off, classification review, explicit exclusions Propose a bounded corpus; provide an exclusion list; show audit controls
Security/privacy review Threat model, logging policy, model boundary policy Bring a short threat model + mitigations; demonstrate “cite or refuse”; propose redaction policy
HLD maturity signal: You’re not surprised by these delays — you plan around them and assign owners.

5) How to present this in an architecture review

Recommended presentation structure

  1. Start with trust boundaries (what data leaves the boundary to the model)
  2. Show identity and access enforcement (group gate + tenant/corpus gates)
  3. Declare the chosen services with 1–2 sentences of why
  4. List alternatives considered and why not now
  5. End with risks + mitigations and the approval asks
Reviewer-friendly line: “We are optimising for a safe pilot. Where a managed service reduces operational risk, we choose it. Where strict filtering and control are essential, we use a search platform that enforces metadata filters.”
