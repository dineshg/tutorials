Architecture Reference Design

Here is a clean, realistic reference design featuring:

  • React frontend (hosted)
  • FastAPI backend running on GKE
  • Multi-tenancy (each user only sees their tenant’s data)

This describes the “industry standard” flow using OIDC/OAuth2 + JWT.

0. Components in the setup

Frontend

React app hosted on Cloud Storage + CDN or Cloud Run.

Identity Provider (IdP)

OIDC provider: Google Identity, Auth0, Okta, Entra ID, etc.

Backend

FastAPI pods on GKE exposed via Ingress.

Data

Postgres (shared DB) with tenant_id column.

1. How users “connect” to the React frontend

  1. User types https://app.example.com
  2. DNS → CDN / hosting
  3. Browser downloads static files: index.html, JS bundles, CSS
  4. React app boots in the browser

At this point, the user is not authenticated yet — they’ve only loaded a public website.

2. User login (OIDC Authorization Code + PKCE)

When user clicks “Sign in”:

  1. React redirects user to the IdP login page
  2. User logs in (password/SSO/MFA)
  3. IdP redirects back to your React app with an authorization code
  4. React exchanges the code for tokens:
    • ID token (who the user is)
    • Access token (JWT used to call your API)
    • Optional refresh token (to renew)

Key point: the access token is what your backend uses to identify the user and their tenant.

Where is the tenant?

Two common patterns:

  • tenant_id is a claim inside the JWT (best for multi-tenant SaaS)
  • or backend looks up tenant_id from sub (user id) in your DB

In production, many do both: claim for speed + DB as source of truth.

3. React calls FastAPI (authenticated)

React calls:

GET https://api.example.com/v1/chat

with header:

Authorization: Bearer <access_token>

Ingress routes to FastAPI service in GKE.

4. FastAPI validates the token (stateless and scalable)

Backend does JWT verification:

  • Validate signature using the IdP’s JWKS
  • Validate: iss (issuer), aud (audience), exp (expiry)
  • Extract claims: sub (user_id), tenant_id (or org_id), roles/groups

This is fast and works with thousands of concurrent users because verification is stateless.

Minimal FastAPI pattern

from fastapi import Depends, HTTPException
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

auth_scheme = HTTPBearer()

def verify_jwt(creds: HTTPAuthorizationCredentials = Depends(auth_scheme)):
    token = creds.credentials
    # 1) verify signature using JWKS
    # 2) validate iss/aud/exp
    # 3) return claims dict
    claims = {"sub": "user123", "tenant_id": "tenantA"}  # toy
    return claims

def tenant_context(claims=Depends(verify_jwt)):
    if "tenant_id" not in claims:
        raise HTTPException(status_code=403, detail="No tenant")
    return {"user_id": claims["sub"], "tenant_id": claims["tenant_id"]}

Now every route can safely depend on tenant_context.

5. How multi-tenancy is enforced (the important part)

Industry standard rule: Every data access must be scoped by tenant_id (and often user_id too).

A) Shared database (toy example)

Your tables include tenant_id:

  • documents(tenant_id, doc_id, text, ...)
  • chunks(tenant_id, chunk_id, embedding_ref, ...)

Query example:

SELECT * FROM documents
WHERE tenant_id = :tenant_id AND doc_id = :doc_id;

B) Vector search (Elastic/OpenSearch)

Every retrieval query includes a filter:

  • tenant_id = tenantA

So even if embeddings are shared in one index, retrieval is tenant-scoped.

6. Defense-in-depth: database-level isolation

Even good teams eventually ship a bug. Industry standard is to add a backstop.

Postgres Row-Level Security (RLS)

  • Enable RLS on tables
  • Policy: user can only see rows where tenant_id = current_setting('app.tenant_id')

Then in FastAPI, set it per request (session):

SET app.tenant_id = 'tenantA';

Now, even if a developer forgets a WHERE tenant_id = ..., Postgres blocks it.

7. Concurrency: multiple users at the same time

This just works if you follow the above patterns because:

  • React is client-side; users are isolated by browser sessions
  • Backend is stateless; each request carries its token
  • Tenant context is derived per request
  • Data stores enforce tenant boundaries

8. Where to store tokens (important practical note)

Option 1: Tokens in memory

SPA-friendly. Less risk of XSS token theft if you’re careful. Needs refresh flow.

Option 2: httpOnly secure cookie

Backend sets cookie after login exchange (BFF pattern). Reduces token exposure in JS. Requires CSRF protections.

Summary you can say in interviews

“Users authenticate via OIDC. The React app obtains an access token (JWT) and calls the FastAPI API. The backend verifies the JWT and extracts tenant_id/user_id, then enforces multi-tenancy by scoping every DB and vector-search query to that tenant. For defense-in-depth we enable Postgres Row Level Security so cross-tenant access is blocked even if a query is missing filters. Because each request is stateless and carries identity, it scales naturally to many concurrent users.”
