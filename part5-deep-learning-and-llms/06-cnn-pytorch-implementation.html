1) nn.Conv2d essentials (what you must know before coding CNNs)

In PyTorch, the standard convolution layer for images is torch.nn.Conv2d. It’s “2D” because the convolution slides over the spatial axes (height and width). The color/channel axis is handled as “channels,” not as a spatial direction.

Key idea: A Conv2d layer learns a set of filters (kernels). Each filter produces one output channel (one feature map).

PyTorch tensor layout (important!)

PyTorch uses the NCHW layout for images:

  • N = batch size
  • C = channels (e.g., 3 for RGB)
  • H = height
  • W = width

Many tutorials talk about images as H×W×C; that’s fine conceptually, but in PyTorch code you’ll usually use N×C×H×W.

nn.Conv2d signature (the arguments you’ll use most)

The common form is: \[ \mathrm{Conv2d}(\text{in\_channels},\ \text{out\_channels},\ \text{kernel\_size},\ \text{stride},\ \text{padding}) \]

ArgumentMeaningRule of thumb
in_channels Number of input channels RGB images → 3. Later layers → number of feature maps from previous conv.
out_channels Number of output channels (feature maps) Often grows: 32 → 64 → 128 → …
kernel_size Filter size (usually square) Usually odd: 3, 5, 7 (3 is most common in modern CNNs).
stride Step size for sliding the kernel Stride 1 keeps resolution (with padding), stride 2 downsamples.
padding Zeros added around image border Often chosen to preserve size (common “same-like” behavior).

What does the learned weight tensor look like?

Internally, the conv weights form a 4D tensor: \[ W \in \mathbb{R}^{C_\text{out} \times C_\text{in} \times K_h \times K_w} \]

That is: one kernel per output channel, and each kernel spans all input channels. This matches the transcript idea: “kernel is 4D: input channels, output channels, height, width.” (PyTorch stores it as out×in×kh×kw.)

python • inspect Conv2d parameters 
import torch
import torch.nn as nn

conv = nn.Conv2d(in_channels=3, out_channels=16, kernel_size=3, stride=1, padding=1)

print("weight shape:", conv.weight.shape)  # (out_channels, in_channels, kH, kW)
print("bias shape:", conv.bias.shape)      # (out_channels,)

Why Conv2d and not Conv1d / Conv3d?

  • Conv1d: sequences / time signals (e.g., audio, sensor streams).
  • Conv2d: images (height & width are the spatial axes).
  • Conv3d: volumes (e.g., medical CT “voxels”) or videos (height, width, time).

2) PyTorch “everything is a module”: inheritance and composition

PyTorch neural network building blocks are modules (instances of nn.Module). A Linear layer is a module. A ReLU is a module. A Sequential container is also a module. And your entire model is a module too.

Superpower: modules compose hierarchically. You can build a “module of modules,” then use it like a single module, and repeat.

Inheritance (why your custom model subclasses nn.Module)

To create your own model class, you inherit from nn.Module. Two key pieces:

  1. __init__: define submodules as attributes (layers you want to learn / run).
  2. forward: define the computation (how data flows through layers).
python • minimal custom MLP (Linear → ReLU → Linear) 
import torch
import torch.nn as nn

class TinyMLP(nn.Module):
    def __init__(self, d_in: int, d_hidden: int, d_out: int):
        super().__init__()
        self.fc1 = nn.Linear(d_in, d_hidden)
        self.act = nn.ReLU()
        self.fc2 = nn.Linear(d_hidden, d_out)

    def forward(self, x):
        x = self.fc1(x)
        x = self.act(x)
        x = self.fc2(x)
        return x

model = TinyMLP(d_in=10, d_hidden=32, d_out=3)
x = torch.randn(5, 10)
logits = model(x)
print(logits.shape)  # (5, 3)

Notice the key correspondence: defining layers in __init__, applying them in forward.

3) Sequential vs Custom Models: when to use which

Option A — nn.Sequential

Great when your network is just a straight stack of layers (no branching, no multi-input/output).

Option B — Custom nn.Module

Needed when you must do operations not easily represented as a simple list of layers (reshapes, multiple paths, skip connections, multi-input/output, etc.).

Why custom models matter in CNNs

The classic “boundary” in CNNs is: convolution outputs an image tensor (N×C×H×W), but dense layers expect vectors (N×D). You need a reshape/flatten step in between.

  • Modern PyTorch provides nn.Flatten().
  • But you should still know how to do it manually with x.view(...) or torch.flatten.
python • Flatten boundary: N×C×H×W → N×(C·H·W) 
import torch
x = torch.randn(8, 64, 7, 7)         # (N, C, H, W)
flat1 = x.view(x.size(0), -1)        # (N, C*H*W)
flat2 = torch.flatten(x, start_dim=1)

print(flat1.shape, flat2.shape)      # both: (8, 3136)

4) A CNN model skeleton (two clean ways)

A practical CNN typically has: Conv stack → Flatten → Dense stack.

Way 1: Custom model with two Sequential blocks + explicit view

python • CNN as nn.Module (conv Sequential + view + dense Sequential) 
import torch
import torch.nn as nn

class SimpleCNN(nn.Module):
    def __init__(self, num_classes: int = 10):
        super().__init__()

        self.conv = nn.Sequential(
            nn.Conv2d(3, 32, kernel_size=3, stride=1, padding=1),
            nn.ReLU(),
            nn.Conv2d(32, 64, kernel_size=3, stride=2, padding=1),  # downsample
            nn.ReLU(),
        )

        # We must know the flattened size here (see convolution arithmetic below).
        # For example: if input is (3, 32, 32), output after conv might be (64, 16, 16)
        # so flattened = 64*16*16 = 16384
        self.fc = nn.Sequential(
            nn.Linear(64 * 16 * 16, 256),
            nn.ReLU(),
            nn.Linear(256, num_classes),
        )

    def forward(self, x):
        x = self.conv(x)
        x = x.view(x.size(0), -1)  # flatten
        x = self.fc(x)
        return x

model = SimpleCNN(num_classes=10)
dummy = torch.randn(4, 3, 32, 32)
print(model(dummy).shape)  # (4, 10)

Way 2: Pure Sequential using nn.Flatten()

This is cleaner, but you still need the correct input size for the first Linear layer.

python • CNN as nn.Sequential (using nn.Flatten) 
import torch
import torch.nn as nn

model = nn.Sequential(
    nn.Conv2d(3, 32, kernel_size=3, stride=1, padding=1),
    nn.ReLU(),
    nn.Conv2d(32, 64, kernel_size=3, stride=2, padding=1),
    nn.ReLU(),
    nn.Flatten(),
    nn.Linear(64 * 16 * 16, 256),
    nn.ReLU(),
    nn.Linear(256, 10),
)

dummy = torch.randn(4, 3, 32, 32)
print(model(dummy).shape)  # (4, 10)

A practical trick: compute the flatten size automatically

If you don’t want to do manual arithmetic while prototyping, you can run a dummy tensor through the conv stack inside __init__ to infer the flatten dimension once.

python • infer flatten size with a dummy forward pass 
import torch
import torch.nn as nn

class AutoShapeCNN(nn.Module):
    def __init__(self, input_shape=(3, 32, 32), num_classes=10):
        super().__init__()
        self.conv = nn.Sequential(
            nn.Conv2d(3, 32, 3, padding=1),
            nn.ReLU(),
            nn.Conv2d(32, 64, 3, stride=2, padding=1),
            nn.ReLU(),
        )

        with torch.no_grad():
            dummy = torch.zeros(1, *input_shape)
            out = self.conv(dummy)
            flat_dim = out.view(1, -1).size(1)

        self.head = nn.Sequential(
            nn.Flatten(),
            nn.Linear(flat_dim, 256),
            nn.ReLU(),
            nn.Linear(256, num_classes),
        )

    def forward(self, x):
        x = self.conv(x)
        x = self.head(x)
        return x

model = AutoShapeCNN()
print(model(torch.randn(2, 3, 32, 32)).shape)  # (2, 10)

This “dummy pass” trick is widely used during research/prototyping. You still should learn the arithmetic, but this keeps you productive.

5) Dropout regularization (the real behavior, clearly explained)

Dropout does NOT permanently remove a neuron. It applies a fresh random mask each forward pass in training mode, zeroing some activations temporarily.

Core math (inverted dropout)

Let \(h\) be a layer activation vector/tensor. Sample a mask: \[ m_i \sim \mathrm{Bernoulli}(q),\quad q = 1-p \] Apply: \[ \tilde{h} = \frac{m \odot h}{q} \] This scaling keeps the expected magnitude consistent between train and eval.

What happens to gradients?

The derivative wrt activations is: \[ \frac{\partial \tilde{h}}{\partial h} = \frac{m}{q} \] So if a unit is dropped (\(m=0\)), its gradient is 0 for that sample. Next batch, the mask changes, so it can update again.

Train vs eval mode (must-know)

  • model.train() → dropout active (random masks)
  • model.eval() → dropout disabled (identity)
python • verify dropout randomness in train vs stability in eval 
import torch
import torch.nn as nn

drop = nn.Dropout(p=0.5)
x = torch.ones(1, 10)

drop.train()
y1 = drop(x)
y2 = drop(x)

drop.eval()
y3 = drop(x)
y4 = drop(x)

print("train outputs differ:", not torch.allclose(y1, y2))
print("eval outputs equal:", torch.allclose(y3, y4))

Where to place dropout in CNNs

The common default is to place dropout between dense layers (the classifier head), not inside convolution blocks. For conv feature maps, people sometimes use nn.Dropout2d (drops entire channels), but you should treat it as an experiment.

python • typical dropout placement in a CNN head 
import torch.nn as nn

head = nn.Sequential(
    nn.Flatten(),
    nn.Linear(64*16*16, 256),
    nn.ReLU(),
    nn.Dropout(p=0.5),
    nn.Linear(256, 10),
)

6) Convolution arithmetic (compute output sizes so your Linear layer matches)

CNNs become annoying to wire up when you reach the first dense layer, because you must know the flattened size. That requires knowing the conv output size.

The PyTorch formula (per spatial dimension)

For height (and similarly width), PyTorch effectively uses: \[ H_{\text{out}} = \left\lfloor \frac{H_{\text{in}} + 2P - D\,(K-1) - 1}{S} + 1 \right\rfloor \]

  • \(H_\text{in}\): input height
  • \(K\): kernel size (height)
  • \(S\): stride
  • \(P\): padding
  • \(D\): dilation (usually 1 unless you set it)

Common case: dilation \(D=1\). Then: \[ H_{\text{out}} = \left\lfloor \frac{H_{\text{in}} + 2P - K}{S} + 1 \right\rfloor \]

“Valid” vs “Same-like” behavior

In classic terms (stride 1, dilation 1):

  • Valid: \(P=0\) → \(H_\text{out} = H_\text{in} - K + 1\)
  • Same-like: choose \(P\) to keep size when \(S=1\) (often \(P=\lfloor K/2\rfloor\) for odd kernels)

In Keras you can write padding="same". In PyTorch you usually specify the numeric padding yourself (or in newer versions you may see padding strings in some APIs, but relying on the arithmetic is still essential).

Worked example: 32×32 → (stride 2) → 16×16

Suppose: \(H_\text{in}=32\), \(K=3\), \(S=2\), \(P=1\) (a common “same-like downsample” setting). Then: \[ H_{\text{out}} = \left\lfloor \frac{32 + 2(1) - 3}{2} + 1 \right\rfloor = \left\lfloor \frac{31}{2} + 1 \right\rfloor = \lfloor 15.5 + 1 \rfloor = 16 \]

python • helper to compute conv output size (H/W) 
import math

def conv_out_size(n_in, k, stride=1, pad=0, dilation=1):
    return math.floor((n_in + 2*pad - dilation*(k-1) - 1) / stride + 1)

print(conv_out_size(32, k=3, stride=2, pad=1))  # 16

Why odd kernel sizes are common

With odd kernels (3,5,7…), there is a clear “center” pixel. This makes it easy to do symmetric padding (e.g., \(P=\lfloor K/2\rfloor\)) and preserve alignment.

Shortcut if you hate arithmetic

During prototyping, use the “dummy forward pass” trick shown earlier to infer flatten size. But you still want the formula in your toolbox when reading papers or debugging shape errors.
