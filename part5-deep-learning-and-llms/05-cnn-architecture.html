1) Convolution with color images: add the channel dimension

Real CNNs usually take color images. A color image is a 3D object: \[ X \in \mathbb{R}^{H \times W \times C_{\text{in}}} \] where \(C_{\text{in}}=3\) for RGB.

To convolve a color image, you use a 3D kernel (a “small box”) that spans all input channels: \[ W \in \mathbb{R}^{K \times K \times C_{\text{in}}} \]

Mechanically, it’s the same “multiply + add” you already know—just with an extra sum over channels: you sum over height, width, and channels.

Color pattern-finder intuition: a 3D filter can match “red circles” differently from “blue circles” because it has separate weights for each channel.

If you convolve a 3D image with a single 3D kernel and sum across channels, the output is 2D (the channel dimension “dots out”). That’s fine for one filter—but it breaks the “stack layers” uniformity.

2) Multiple filters solve the uniformity problem

A single filter finds a single feature. But images contain many features—edges, corners, textures, eyes, noses, etc. So CNNs use many filters.

If you apply two different filters \(W^{(1)}\) and \(W^{(2)}\) to the same input \(X\), you get two 2D outputs: \[ B^{(1)} = X \star W^{(1)}, \quad B^{(2)} = X \star W^{(2)} \] Stack them and you get a 3D output again: \[ B = \text{stack}\left(B^{(1)}, B^{(2)}, \ldots\right) \in \mathbb{R}^{H \times W \times C_{\text{out}}} \]

Terminology shift: after the first layer, the last dimension is no longer “color”. It’s the number of channels or feature maps (one map per learned feature).

3) Full convolution in CNN form: shapes + equation

Shapes used in this lecture

  • Input: \(X \in \mathbb{R}^{H \times W \times C_1}\)
  • Weights: \(W \in \mathbb{R}^{C_1 \times K \times K \times C_2}\)
  • Output (same mode): \(Y \in \mathbb{R}^{H \times W \times C_2}\)

Many frameworks store weights as \((C_2, C_1, K, K)\). The transcript’s shape is equivalent—just a different axis order.

Index form (same-mode, stride 1; “deep learning style” correlation)

Let \(Y_{i,j,c}\) be the output at spatial position \((i,j)\) and output channel \(c\) (a feature map). Then:

\[ Y_{i,j,c} = \sum_{c'=1}^{C_1}\sum_{u=0}^{K-1}\sum_{v=0}^{K-1} X_{i+u-\Delta,\;j+v-\Delta,\;c'} \; W_{c',u,v,c} \]

Here \(\Delta=\lfloor K/2 \rfloor\) corresponds to “same” padding for odd \(K\) (conceptually: zeros around the image).

Concept: each output channel \(c\) corresponds to one filter (one learned feature). The output tensor is a stack of feature maps.

4) A convolution layer = convolution + bias + activation

A CNN layer mirrors a dense layer idea:

\[ \text{Dense: } z = Wx + b,\quad a = \sigma(z) \]

For convolution:

\[ Z = X \star W + b,\quad A = \sigma(Z) \]

Bias shape and broadcasting

The convolution output has shape \(H \times W \times C_2\). The bias term is a vector \(b \in \mathbb{R}^{C_2}\): one scalar per feature map. In code, broadcasting adds \(b_c\) to every spatial location \((i,j)\) in that feature map.

Broadcasting intuition: bias is “per-channel”, not “per-pixel”. It’s the same idea as dense layers (one bias per output unit), just expanded across the grid.

python • conv layer with bias + activation (PyTorch) 
import torch
import torch.nn as nn

# Example: Cin=3 (RGB), Cout=16 feature maps, 3x3 kernels
layer = nn.Sequential(
    nn.Conv2d(in_channels=3, out_channels=16, kernel_size=3, padding=1, bias=True),  # "same" for 3x3
    nn.ReLU()
)

x = torch.randn(8, 3, 32, 32)   # NCHW: batch=8, channels=3, H=W=32
y = layer(x)
print(y.shape)  # (8, 16, 32, 32)

Notice PyTorch uses NCHW ordering: \((N, C, H, W)\). The “channels” here are feature maps after the first layer.

5) Why CNNs are efficient: parameter sharing vs dense explosion

Parameter sharing means one small kernel is reused across the entire image. That dramatically reduces parameter count compared to flattening and using a dense layer.

Worked parameter comparison (from the transcript)

Input image: \(32 \times 32 \times 3\). Kernel: \(3 \times 5 \times 5 \times 64\) (64 output feature maps). Using valid convolution, output spatial size is \((32-5+1)\times(32-5+1)=28\times 28\). So output is \(28\times 28\times 64\).

Convolution parameters
\(3 \cdot 5 \cdot 5 \cdot 64 = 4800\)

Dense parameters (flattened)
input: \(32\cdot 32\cdot 3 = 3072\)
output: \(28\cdot 28\cdot 64 = 50176\)
weights: \(3072 \times 50176 \approx 154\) million

Dense would have ~32,000× more parameters in this example. This increases memory and compute and can hurt generalization, because dense must learn separate weights for the “same pattern” at every location.

6) A typical CNN has two stages

CNNs are commonly organized as:

  1. Convolutional stage: repeated blocks that extract image features (often Conv → (Pooling or Stride)).
  2. Dense stage: a standard feed-forward network that performs classification/regression on extracted features.

Mental model: the conv stage is a feature transformer designed for images. Once features are extracted, the dense stage handles non-linear prediction (classification/regression).

7) Pooling: downsampling by summarizing local neighborhoods

Pooling reduces spatial size (e.g., 100×100 → 50×50 with 2×2 pooling, stride 2). Two common types:

  • Max pooling: take the maximum value in each pooling window
  • Average pooling: take the average in each pooling window

python • max pooling and average pooling (PyTorch) 
import torch
import torch.nn as nn

x = torch.tensor([[[[ 1.,  2.,  3.,  4.],
                    [ 5.,  6.,  7.,  8.],
                    [16., 15., 14., 13.],
                    [12., 11., 10.,  9.]]]])  # shape (N=1, C=1, H=4, W=4)

maxpool = nn.MaxPool2d(kernel_size=2, stride=2)
avgpool = nn.AvgPool2d(kernel_size=2, stride=2)

print("max:\n", maxpool(x)[0,0])
print("avg:\n", avgpool(x)[0,0])

8) Why pooling helps: translation invariance

Pooling supports the idea of translation invariance: we often care that a feature exists somewhere, not exactly where it is.

In a small 2×2 region, a feature detector output might look like “high, low, low, low”. Max pooling returns the high value, preserving “feature found” while discarding precise location.

Max pooling intuition: “if the pattern appears anywhere in this window, keep it.” This often matches what you want for recognition tasks.

Pooling layers also have hyperparameters: pool size (e.g., 2×2) and stride (often 2). Overlapping pooling (stride 1) exists but is less common in typical modern setups.

9) Conv + pooling creates hierarchical feature learning

CNNs often repeat the pattern: convolution → pooling → convolution → pooling. Spatial size decreases, while filters remain small (commonly 3×3, 5×5, 7×7).

Why small filters start as “tiny pattern” detectors

A 3×3 filter covers a tiny fraction of a 32×32 image. After downsampling to 16×16, the same 3×3 covers a larger fraction of the image; later it can cover a very large portion (e.g., on 4×4).

This is one reason CNNs learn features hierarchically: early layers tend to pick up small patterns (edges/strokes), later layers pick up larger patterns (parts, then whole objects).

Two linked effects:
1) spatial dimensions shrink
2) fixed-size filters “see” relatively larger areas → bigger patterns

Yes, shrinking loses spatial detail (“where”). But CNNs often compensate by increasing the number of feature maps (“what”) as depth increases.

10) Strided convolution as an alternative to pooling

Pooling isn’t the only downsampling option. Convolution itself can downsample using a stride: stride tells you how far the sliding window jumps each step.

With stride 2, output spatial dimensions are roughly halved—similar to pooling with 2×2, stride 2.

Why skipping pixels can be okay: nearby pixels are often highly correlated in natural images. Skipping every other pixel still preserves much of the structure while reducing computation.

python • strided conv vs conv+pool (shape comparison) 
import torch
import torch.nn as nn

x = torch.randn(1, 3, 32, 32)

# Option A: conv then pool
conv = nn.Conv2d(3, 16, kernel_size=3, padding=1)     # keep 32x32
pool = nn.MaxPool2d(2, stride=2)                      # downsample to 16x16
y_a = pool(torch.relu(conv(x)))

# Option B: strided conv (downsample directly)
conv_s2 = nn.Conv2d(3, 16, kernel_size=3, padding=1, stride=2)
y_b = torch.relu(conv_s2(x))

print("conv+pool:", y_a.shape)  # (1, 16, 16, 16)
print("stride=2:", y_b.shape)   # (1, 16, 16, 16)

Both produce the same spatial size here; the difference is where downsampling happens and how parameters are used.

11) Flatten vs global max pooling (handling variable image sizes)

The dense stage expects a 1D feature vector. A convolutional output is 3D: \[ X \in \mathbb{R}^{H \times W \times C} \] so a common step is to flatten to a vector of size \(H\cdot W\cdot C\).

Why flattening fails for variable image sizes

If the network outputs 2×2×100 for a 32×32 input, flatten gives 400 features. For a larger 64×64 input that becomes 4×4×100, flatten gives 1600 features. A fixed dense layer can’t accept both sizes.

Global max pooling: size-independent output

Global max pooling collapses the entire spatial grid: \[ \text{GMP}(X)_c = \max_{i,j} X_{i,j,c} \] producing a vector of size \(C\), regardless of \(H\) and \(W\).

Interpretation: “I don’t care where the feature appears in the image—if it appears anywhere, keep the strongest evidence.”

python • global max pooling in PyTorch (works for any H×W) 
import torch
import torch.nn as nn

gmp = nn.AdaptiveMaxPool2d((1, 1))  # global max pooling to 1x1

x1 = torch.randn(8, 100, 2, 2)     # e.g., from 32x32 after downsampling
x2 = torch.randn(8, 100, 4, 4)     # e.g., from 64x64 after downsampling

y1 = gmp(x1).squeeze(-1).squeeze(-1)  # shape: (8, 100)
y2 = gmp(x2).squeeze(-1).squeeze(-1)  # shape: (8, 100)

print(y1.shape, y2.shape)

Note: images can’t be downsampled indefinitely. If the spatial size becomes too small (e.g., 2×2), additional stride-2 operations will fail.

12) A modern CNN blueprint (as summarized in the transcript)

A practical CNN often follows:

  1. Convolutional stage: repeated blocks that downsample while increasing channels
    • Option A: Conv → Pool (repeat)
    • Option B: Strided Conv (repeat)
    • Keep filter sizes small (commonly 3×3, 5×5, 7×7)
    • Increase feature maps with depth (e.g., 32 → 64 → 128 → ...)
  2. Bridge: convert 3D feature maps to 1D features
    • Flatten (view) for fixed-size inputs
    • or Global pooling for variable-size inputs
  3. Dense stage: standard feed-forward layers ending with task-appropriate output
    • Regression: 1 output, identity (no activation)
    • Binary classification: 1 output, sigmoid
    • Multi-class (K≥2): K outputs, softmax (often combined with cross-entropy loss)
python • reference CNN skeleton (conv stage → global max pool → dense) 
import torch.nn as nn

class SimpleCNN(nn.Module):
    def __init__(self, num_classes=10):
        super().__init__()
        self.features = nn.Sequential(
            nn.Conv2d(3, 32, 3, padding=1), nn.ReLU(),
            nn.Conv2d(32, 64, 3, padding=1, stride=2), nn.ReLU(),   # stride downsample
            nn.Conv2d(64, 128, 3, padding=1, stride=2), nn.ReLU(),
        )
        self.gmp = nn.AdaptiveMaxPool2d((1, 1))  # global max pool
        self.classifier = nn.Sequential(
            nn.Linear(128, 128), nn.ReLU(),
            nn.Linear(128, num_classes)
        )

    def forward(self, x):
        x = self.features(x)               # (N, 128, H', W')
        x = self.gmp(x).squeeze(-1).squeeze(-1)  # (N, 128)
        return self.classifier(x)          # logits

# Use with nn.CrossEntropyLoss() for multi-class classification

Big picture: convolution learns feature transformers automatically during training (weights + biases), just like dense layers—only the tensor shapes and parameter sharing differ.
