Forecasting (Code): Autoregressive Linear Model in PyTorch

Goal: implement an autoregressive (AR) linear model for time-series prediction, and learn the right vs wrong way to forecast multiple steps ahead.

1) The setup

We’ll build a synthetic dataset: a sine wave (optionally with noise), then create supervised learning examples by sliding a window of length T.

Windowing idea
Input: \(\;X_t = [x_t, x_{t+1}, \dots, x_{t+T-1}]\;\) (shape \(T\))
Target: \(\;y_t = x_{t+T}\;\)

This “AR(T)” model is just linear regression where the “features” are the previous T time steps.

Optional visual: sine wave (online image)

A sine wave is periodic, which makes it great for controlled experiments.

2) Build the dataset (windowing)

We create a length-1000 series, then produce \((X, y)\) pairs using a sliding window. This is exactly where off-by-one mistakes love to hide—so we’ll write it clearly. 

# --- Imports ---
import numpy as np
import torch
import torch.nn as nn
import matplotlib.pyplot as plt

# Reproducibility
np.random.seed(0)
torch.manual_seed(0)

# --- 1) Create a synthetic time series ---
N = 1000
omega = 0.1
series = np.sin(omega * np.arange(N))

# OPTIONAL: add noise (uncomment to test realistic behavior)
# noise_std = 0.2
# series = series + noise_std * np.random.randn(N)

plt.figure()
plt.plot(series)
plt.title("Synthetic time series")
plt.xlabel("t")
plt.ylabel("x[t]")
plt.show()

# --- 2) Turn the series into supervised learning data ---
T = 10  # use previous 10 steps to predict the next step

X_list, y_list = [], []
# t goes from 0 to N - T - 1 so that y[t] = series[t+T] is valid
for t in range(0, N - T):
    X_list.append(series[t : t + T])      # length T
    y_list.append(series[t + T])          # scalar target

X = np.array(X_list)          # shape: (N-T, T)
y = np.array(y_list).reshape(-1, 1)  # shape: (N-T, 1)

print("X shape:", X.shape, "y shape:", y.shape)

Important: For time series, do not shuffle before splitting into train/validation. Your “validation” should be in the future relative to training.

3) Train/validation split (chronological)

# Chronological split: first half train, second half validation
split = len(X) // 2
X_train, y_train = X[:split], y[:split]
X_val,   y_val   = X[split:], y[split:]

# Convert to torch tensors
X_train_t = torch.tensor(X_train, dtype=torch.float32)
y_train_t = torch.tensor(y_train, dtype=torch.float32)
X_val_t   = torch.tensor(X_val,   dtype=torch.float32)
y_val_t   = torch.tensor(y_val,   dtype=torch.float32)

4) The AR model in PyTorch

The AR(T) linear model is literally one layer: it maps \( \mathbb{R}^T \rightarrow \mathbb{R} \).

\( \hat{x}_{t+T} = b + \sum_{i=0}^{T-1} w_i \, x_{t+i} \) 
# Model: linear regression on the window
model = nn.Linear(T, 1)

loss_fn = nn.MSELoss()
optimizer = torch.optim.Adam(model.parameters(), lr=0.05)

5) Training loop (full-batch gradient descent)

Because this dataset is small, we can do “full GD” (use all training data each step). For bigger datasets you’d use mini-batches, but the forecasting logic below stays the same. 

def train_full_gd(model, Xtr, ytr, epochs=300):
    losses = []
    for epoch in range(epochs):
        model.train()  # good habit (matters for dropout/batchnorm later)

        pred = model(Xtr)                 # shape: (n, 1)
        loss = loss_fn(pred, ytr)

        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

        losses.append(loss.item())
    return losses

losses = train_full_gd(model, X_train_t, y_train_t, epochs=300)

plt.figure()
plt.plot(losses)
plt.title("Training loss (MSE)")
plt.xlabel("epoch")
plt.ylabel("loss")
plt.show()

6) Forecasting: the “wrong” way vs the correct way

The wrong way (looks great, but it’s cheating for multi-step forecasting)

This method uses the true validation windows \(X_{\text{val}}\) at every step. That means each prediction is effectively 1-step-ahead using real history.

It answers: “How good is the model at 1-step prediction if I always give it correct inputs?” It does not answer: “How good is the model at forecasting 50 steps into the future?” 

# WRONG for multi-step: uses ground-truth inputs each time ("teacher forcing"-like)
model.eval()
with torch.no_grad():
    wrong_preds = []
    for i in range(len(X_val_t)):
        # X_val_t[i] has shape (T,), but model expects (batch, T)
        p = model(X_val_t[i].view(1, -1))  # (1, 1)
        wrong_preds.append(p.item())

plt.figure()
plt.plot(y_val_t.numpy(), label="target")
plt.plot(wrong_preds, label="wrong forecast (cheating)")
plt.title("Validation: wrong vs target")
plt.legend()
plt.show()

The correct way (recursive / roll-forward forecasting)

Once you move into the future, you don’t have the true \(x_{t}\) anymore — you only have your previous predictions. So after predicting one step, you must feed it back in to predict the next. 

# CORRECT multi-step forecast: use your own predictions as future inputs
model.eval()
with torch.no_grad():
    correct_preds = []

    # Start from the *first* validation input window
    last_x = X_val_t[0].clone()  # shape: (T,)

    for _ in range(len(y_val_t)):
        p = model(last_x.view(1, -1))      # shape: (1, 1)
        correct_preds.append(p.item())

        # Update the window: drop oldest, append newest prediction
        # Keep everything as torch tensors
        last_x = torch.cat([last_x[1:], p.view(-1)])  # (T-1,) + (1,) -> (T,)

plt.figure()
plt.plot(y_val_t.numpy(), label="target")
plt.plot(correct_preds, label="correct forecast (recursive)")
plt.title("Validation: correct vs target")
plt.legend()
plt.show()

What you should observe

  • No-noise sine wave: both methods can look nearly perfect.
  • Noisy sine wave: the “wrong” method often still looks great, while the recursive method degrades, because small errors compound as you roll forward.

7) Why can a linear AR model predict a sine wave so well?

A sine wave has an exact 2-step linear recurrence. You can write:

If \(x_t = \sin(\omega t)\), then
\[ x_{t+1} = 2\cos(\omega)\,x_t - x_{t-1} \]

This means an AR(2) model (two past values) can represent the sine wave perfectly (in the no-noise case). Noise breaks the exact recurrence, which is why recursive forecasting becomes imperfect.

Sketch of the derivation (optional)

Use the identity: \[ \sin(a+b) + \sin(a-b) = 2\sin(a)\cos(b) \] Let \(a = \omega t\) and \(b = \omega\). Then: \[ \sin(\omega(t+1)) + \sin(\omega(t-1)) = 2\sin(\omega t)\cos(\omega) \] Rearranging gives: \[ \sin(\omega(t+1)) = 2\cos(\omega)\sin(\omega t) - \sin(\omega(t-1)) \] which is exactly the AR(2) recurrence.

8) Common gotchas (quick checklist)

  • Shape mistakes: a single window is shape \((T,)\), but the model expects \((1, T)\).
  • Random train/test split: don’t do it for forecasting—split chronologically.
  • Evaluating only 1-step ahead: can be useful, but it’s not the same as multi-step forecasting.
  • Error compounding: recursive forecasting will drift if the model is slightly wrong.
