Recurrent Neural Networks (RNNs): one “cell” reused across time

A practical, code-oriented tutorial on why RNNs exist, how a basic RNN (Elman unit) computes outputs, and how weight sharing saves huge numbers of parameters.

Big idea: An RNN processes a sequence by applying the same small neural “cell” at every time step. The cell carries a hidden state forward, so information can flow through time while using a tiny set of shared weights.

1) Why not just flatten sequence data into a giant vector?

You can flatten a sequence \(X \in \mathbb{R}^{T \times D}\) into a single vector of length \(T\cdot D\) and feed it to a regular MLP. It works mechanically, but it’s usually a bad tradeoff:

  • Parameter explosion: dense layers connect “everything to everything”, so weights scale with \((T\cdot D)\).
  • Ignores sequential structure: treating time steps as just “features” discards the idea of incremental processing.
  • Hard to generalize across time: you’re not explicitly reusing patterns across positions in the sequence.
RNNs exploit the structure: the same computation repeats at every time step, just like CNN filters repeat across image locations.

2) The simplest RNN unit (Elman unit): equations and intuition

At each time step \(t\), the RNN consumes an input vector \(x_t \in \mathbb{R}^{D}\) and updates a hidden state \(h_t \in \mathbb{R}^{M}\). Then it optionally produces an output \(\hat{y}_t\).

Core equations

A standard “vanilla” RNN does:

\[ h_t = \phi(W_{xh}x_t + W_{hh}h_{t-1} + b_h) \] \[ \hat{y}_t = \psi(W_{ho}h_t + b_o) \]

Where:

  • \(W_{xh}\in\mathbb{R}^{M\times D}\): input → hidden weights
  • \(W_{hh}\in\mathbb{R}^{M\times M}\): hidden → hidden (recurrent) weights
  • \(W_{ho}\in\mathbb{R}^{O\times M}\): hidden → output weights
  • \(\phi\): hidden activation (often \(\tanh\) or ReLU); \(\psi\): output activation depends on task
  • \(h_0\): initial hidden state (often zeros; sometimes learned)

“Single unit is used”: what that really means

When people say “the same unit is used at each time step”, they mean:

  • You do not have a different \(W_{xh}\) for each \(t\).
  • You do not have a different \(W_{hh}\) for each \(t\).
  • You reuse the same weights and biases at every step: that’s parameter sharing through time.

The only thing that changes over time is the data (\(x_t\)) and the state (\(h_{t-1}\to h_t\)). This is exactly why RNNs can be far cheaper than flattening.

3) Many-to-one vs many-to-many: why you might “ignore” most outputs

You can compute \(\hat{y}_t\) at every step, but whether you use them depends on the task:

  • Many-to-one: sentiment of a sentence → use only \(\hat{y}_T\)
  • Many-to-many: label each time step (e.g., tagging) → use all \(\hat{y}_t\)
  • Seq2seq (translation): outputs are also a sequence → you need stepwise predictions

4) Pseudocode: forward pass through a sequence

# Given:
#   X: shape (T, D)
#   h0: shape (M,)   (often zeros)
#   weights: W_xh (M, D), W_hh (M, M), b_h (M,)
#            W_ho (O, M), b_o (O,)
# Returns:
#   y_hats: list of length T, each element shape (O,)

h = h0
y_hats = []
for t in range(T):
    h = tanh(W_xh @ X[t] + W_hh @ h + b_h)   # hidden update
    y = softmax(W_ho @ h + b_o)              # output (for classification)
    y_hats.append(y)

# For many-to-one tasks, you would use y_hats[-1] and ignore the rest.

5) Cost savings: RNN vs flattened ANN

Let:

  • \(T\) = sequence length
  • \(D\) = features per time step
  • \(M\) = hidden size
  • \(O\) = output size (e.g., \(O=1\) for binary classification/logit)

Flattened ANN (dense “everything-to-everything”)

If you flatten, input size is \(T\cdot D\). If you also keep a hidden representation for each time step (as in the transcript’s comparison), hidden size becomes \(T\cdot M\).

Approx weights (ignoring biases): \[ (T D)(T M) + (T M)O \]

RNN (shared weights through time)

Weights are independent of \(T\): \[ W_{xh}: M D,\quad W_{hh}: M^2,\quad W_{ho}: O M \] Total: \[ MD + M^2 + OM \]

Key win: RNN parameter count does not grow with sequence length \(T\) the way a flattened dense model does.

Concrete example + Python code (and a plot)

Example: \(T=100\), \(D=10\), \(M=15\), \(O=1\). 

import numpy as np
import matplotlib.pyplot as plt

def ann_params_flat(T, D, M, O):
    # Transcript-style comparison: hidden size = T*M
    return (T*D) * (T*M) + (T*M) * O

def rnn_params(T, D, M, O):
    # Shared through time
    return (M*D) + (M*M) + (O*M)

T, D, M, O = 100, 10, 15, 1
p_ann = ann_params_flat(T, D, M, O)
p_rnn = rnn_params(T, D, M, O)

print("Flattened ANN params:", p_ann)
print("RNN params:", p_rnn)
print("ANN / RNN ratio:", p_ann / p_rnn)

# Quick bar plot
labels = ["Flattened ANN", "RNN (shared weights)"]
values = [p_ann, p_rnn]

plt.figure()
plt.bar(labels, values)
plt.yscale("log")  # so both bars are visible
plt.title("Parameter count comparison (log scale)")
plt.ylabel("Number of parameters")
plt.show()

In this setup, you’ll see a huge ratio (thousands of times more parameters for the flattened ANN), matching the intuition from the transcript.

6) Coding it in PyTorch: manual loop (to really see weight sharing)

The fastest way to “get” RNNs is to implement the loop yourself once. Below is a minimal RNN cell you can train like any other module. 

import torch
import torch.nn as nn
import torch.nn.functional as F

class TinyElmanRNN(nn.Module):
    """
    Vanilla RNN (Elman unit) implemented explicitly:
      h_t = tanh(W_xh x_t + W_hh h_{t-1} + b_h)
      y_t = W_ho h_t + b_o
    """
    def __init__(self, input_size: int, hidden_size: int, output_size: int):
        super().__init__()
        self.input_size = input_size
        self.hidden_size = hidden_size
        self.output_size = output_size

        # These are the ONLY learned weights, reused for every t
        self.W_xh = nn.Linear(input_size, hidden_size, bias=True)
        self.W_hh = nn.Linear(hidden_size, hidden_size, bias=False)
        self.W_ho = nn.Linear(hidden_size, output_size, bias=True)

    def forward(self, x: torch.Tensor, h0: torch.Tensor | None = None):
        """
        x: (batch, T, D)  using batch_first layout here
        h0: (batch, M) or None
        Returns:
          y_all: (batch, T, O)  (pred at every step)
          h_last: (batch, M)
        """
        B, T, D = x.shape
        if h0 is None:
            h = torch.zeros(B, self.hidden_size, device=x.device, dtype=x.dtype)
        else:
            h = h0

        y_steps = []
        for t in range(T):
            xt = x[:, t, :]  # (batch, D)
            h = torch.tanh(self.W_xh(xt) + self.W_hh(h))
            yt = self.W_ho(h)  # (batch, O)
            y_steps.append(yt)

        y_all = torch.stack(y_steps, dim=1)  # (batch, T, O)
        return y_all, h

# Example usage:
B, T, D = 32, 100, 10
M, O = 15, 1
model = TinyElmanRNN(D, M, O)

x = torch.randn(B, T, D)
y_all, h_last = model(x)

# Many-to-one binary classification: use only last step
logits = y_all[:, -1, :]         # (B, 1)
probs  = torch.sigmoid(logits)   # if binary classification
Notice how self.W_xh and self.W_hh are created once in __init__, then used repeatedly inside the time loop. That’s the “single unit reused” idea in code.

7) Using PyTorch’s built-in nn.RNN (recommended for real projects)

PyTorch provides torch.nn.RNN which handles the looping internally and exposes standardized weights (e.g., weight_ih_l0, weight_hh_l0) and batch/sequence layouts. 

import torch
import torch.nn as nn

D, M = 10, 15   # input_size, hidden_size
rnn = nn.RNN(
    input_size=D,
    hidden_size=M,
    num_layers=1,
    nonlinearity="tanh",
    batch_first=True,  # input is (batch, T, D)
)

B, T = 32, 100
x = torch.randn(B, T, D)

# output: (B, T, M)   h_n: (num_layers, B, M)
out, h_n = rnn(x)

# Many-to-one classifier head
head = nn.Linear(M, 1)
logits = head(out[:, -1, :])

If you’re ever unsure about shapes, inputs/outputs, or the exact meaning of those weight tensors, the official docs are the source of truth.

8) Quick mental model recap

  • RNN = MLP cell + memory. The “memory” is the hidden state \(h_t\).
  • One cell reused at every time step. Same weights, new data + updated state.
  • Outputs at every step are optional. Many tasks use only the last output.
  • Cost savings come from sharing. Parameter count doesn’t blow up with \(T\).
